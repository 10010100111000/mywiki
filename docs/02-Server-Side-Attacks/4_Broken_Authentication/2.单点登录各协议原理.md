# 一、简介

**SSO**（Single Sign-On）即**单点登录**，是一种身份验证方案。它允许用户在多个相互独立的软件系统或应用程序中，只需登录一次，就可以访问所有相互信任的系统，而无需重复输入账号和密码。

SSO 的实现通常依赖于以下成熟的开放标准和协议：

- **SAML (Security Assertion Markup Language)**：基于 XML 的标准，常用于企业级应用之间的身份信息交换。
- **OIDC (OpenID Connect)**：基于 OAuth 2.0 的身份认证层，广泛应用于移动端和互联网应用（如“使用 Google/微信账号登录”）。
- **CAS (Central Authentication Service)**：一种常见的开源单点登录协议。

# 二、 中央认证服务CAS

中央认证服务（Central Authentication Service, CAS）是一套轻量化、基于票据（Ticket）实现的单点登录协议，最初为校园与企业内网环境设计，凭借极简的交互结构与清晰的流量特征，成为理解 SSO 协议模型的理想入门对象。

与后续更为复杂的安全断言标记语言（Security Assertion Markup Language, SAML）、开放 ID 连接（OpenID Connect, OIDC）不同，CAS 不依赖 XML、JWT 等结构化封装格式，也不强制要求数字签名与加密扩展，全部交互基于标准 HTTP 重定向与后端接口校验完成。这种简洁性使其在内网办公系统、统一门户、校园业务平台中被广泛部署。

### 1.1 名词解释

| **专业名词**      | **扩展描述**                          | **在架构中的绝对作用**                                       |
| ----------------- | ------------------------------------- | ------------------------------------------------------------ |
| **CAS Server**    | Central Authentication Service        | 信任的绝对核心。负责统一校验 User 的身份，管理全局的 SSO Session，并向各个业务系统签发和验证 ST。 |
| **Protected App** | Service Provider (SP) / Relying Party | 接入 CAS 体系的独立业务系统。自身不负责校验密码，完全依赖 CAS Server 传递过来的 ST 来建立本地信任。 |
| **TGT**           | Ticket Granting Ticket                | 存放于 CAS Server 端的全局会话标识。它代表 User 已经在 CAS 完成了 authentication，是生成后续所有 ST 的根凭证。 |
| **TGC**           | CAS Ticket Granting Cookie            | TGT 在 Browser 端的映射载体。是一个存在于 CAS Server 域名下的 Cookie。只要它没有失效，User 访问新应用时就能实现无感跨域免登。 |
| **ST**            | Service Ticket                        | 由 CAS Server 签发给 Browser，用于携带至 Protected App 的一次性、短效授权凭证。必须与具体的 `service` 参数严格绑定。 |
| **Session**       | User Session                          | 架构中存在两种 Session：一是 CAS Server 维护的**全局 SSO Session** (生命周期由 TGT 决定)；二是各个 Protected App 自己维护的**局部 Application Session**。 |
| **Cookie**        | HTTP Cookie                           | Browser 端的状态存储机制。由于同源策略，`CASTGC` 只发给 CAS Server，而 `JSESSIONID` 只发给 App1。 |
| **service**       | Callback URL Parameter                | 告诉 CAS Server，当 User 认证成功后，应该把携带 ST 的 302 重定向发往哪个具体的 App 业务接口。 |

### 1.2 步骤解析

#### 阶段一：First Access (首次访问)

此阶段是整个 SSO 信任链条的建立过程。

1. **触发拦截**：User 首次尝试访问 `App1`。由于 `App1` 本地没有识别到有效的业务 Session Cookie，拒绝放行。
2. **引导认证**：`App1` 构造一个包含自身回调地址 (`service` 参数) 的 CAS Login URL，通过 HTTP 302 迫使 Browser 跳转至 `CAS Server`。
3. **提交凭据**：`CAS Server` 检查发现 Browser 没有携带 `CASTGC`，判定为没有全局 SSO Session，于是下发登录页面。User POST 提交 Username 和 Password。
4. **签发根凭证与子凭证**：`CAS Server` 校验账密成功后执行关键的**双发操作**：
   - 签发 `CASTGC`：通过 `Set-Cookie` 种在 Browser 中（这是后续跨站免登的核心）。
   - 签发 `ST`：拼接到 `service` 参数提供的 URL 后方，通过 302 重定向把 Browser 踢回 `App1`。
5. **后端校验 (Back-channel)**：`App1` 拦截到 URL 中的 `ticket=ST-xxx`。由于 `App1` 无法判断 ST 的真伪，它会在后端直接发起一次到 `CAS Server` 的 HTTPS 请求（`/serviceValidate`）。
6. **局部会话建立**：`CAS Server` 响应 XML 确认 ST 合法及 User 身份。`App1` 此时完全信任该 User，在自己端建立局部 Session，通过 `Set-Cookie` 给 Browser 下发 `JSESSIONID`，完成首次登录闭环。

#### 阶段二：Second Access To Same Application (同应用二次访问)

此阶段完全脱离 CAS，由业务应用自行管理状态。

1. **携带局部凭据**：User 在 `App1` 内进行点击跳转时，Browser 会自动携带第一阶段获取的 `JSESSIONID` Cookie。
2. **本地验证放行**：`App1` 根据收到的 `JSESSIONID` 直接在本地内存或 Redis 中匹配到对应的业务 Session，校验合法即返回资源数据，全程**无需**与 `CAS Server` 产生任何网络通信。

#### 阶段三：First Access To Second Application (首次跨域访问第二个应用)

此阶段是 SSO "单点登录" 核心价值的体现（无感免登）。

1. **触发拦截与引导**：User 试图访问全新的 `App2`。`App2` 本地无 Session，同样使用 302 将 Browser 重定向至 `CAS Server`。
2. **携带全局凭据**：**（最关键的一步）** Browser 在向 `CAS Server` 发起 GET 请求时，由于命中了 CAS 的域名，**会自动携带第一阶段获取的 `CASTGC` Cookie**。
3. **无感签发**：`CAS Server` 接收到请求后，提取 `CASTGC` 并验证对应的 TGT 依然存活。判定 User 的全局 SSO Session 依然有效，**直接跳过 Login Form 阶段**。
4. **下发新子凭证**：`CAS Server` 直接针对 `App2` 的 `service` 地址，签发一个全新的专属 `ST`，并通过 302 将 Browser 重定向至 `App2`。
5. **后端校验与会话建立**：`App2` 拿到 ST，进入与 `App1` 相同的后台校验流程。校验成功后，`App2` 下发属于自己域名的局部 Session Cookie (如 `MOD_AUTH_CAS_S`)，User 顺利进入 `App2`，全程未感知到需要再次输入密码。

### 2.2 名词解析表

| **专业名词**        | **英文全称 / 扩展描述** | **在注销架构中的绝对作用**                                   |
| ------------------- | ----------------------- | ------------------------------------------------------------ |
| **Single Sign-Out** | 全局单点注销            | SSO 架构的基本安全要求。确保 User 在任意一个接入的应用中触发注销操作后，不仅销毁当前应用的 Session，还要级联销毁全局 Session 以及所有已登录子应用的 Session。 |
| **Back-channel**    | 后端通道通信            | CAS Server 与 Protected App 之间的服务器直连通信（不经过 Browser）。在注销场景中，用于 CAS 主动“通知”业务应用，防止因 Browser 直接关闭而导致前端注销链条断裂。 |
| **LogoutRequest**   | 注销请求载荷            | 通常采用 SAML 标准格式封装的一段 XML 数据。其中包含了需要被销毁的 Session 标识，由 CAS Server 通过 HTTP POST 发送给各个应用。 |
| **Invalidate**      | 失效/销毁操作           | 在 CAS 语境中，指代将 TGT 标记为作废。一旦 TGT 被 Invalidate，所有依赖该 TGT 派生的 ST 和后续免登能力将瞬间失效。 |

------

### 2.3 步骤解析

1. **触发本地注销**：User 在 `App1` 点击注销按钮。`App1` 首先在自己的后端内存或缓存中销毁与该 User 对应的 Application Session（此时 `JSESSIONID` 已在后端失效）。
2. **重定向至全局中枢**：`App1` 通过 HTTP 302 响应，将 Browser 引导至 `CAS Server` 的标准注销端点（通常是 `/logout`）。
3. **发起全局注销请求**：Browser 跟随重定向，向 `CAS Server` 发起 GET 请求。关键点在于，Browser 会自动携带代表全局会话的 `CASTGC` Cookie。
4. **终结全局会话**：`CAS Server` 接收到请求后，提取出 TGT 并将其在后端数据库/内存中执行 Invalidate（彻底销毁）。此时，整个 SSO 体系的核心信任根已被拔除。
5. **执行 Back-channel 通知机制**（防守方与攻击方重点关注区域）：
   - `CAS Server` 会查询该 TGT 在存活期间都向哪些具体的业务应用签发过 ST（在本例中，记录显示访问过 `App1` 和 `App2`）。
   - `CAS Server` 发起并发的**后端服务器直连**请求，向 `App1` 和 `App2` 分别发送包含 `LogoutRequest` 的 HTTP POST 请求。
   - `App2` 的后端接收到该请求后，解析出对应的会话标识，强制在本地将其 Application Session（如 `MOD_AUTH_CAS_S`）销毁。至此，即使用户直接访问 `App2` 且 Browser 仍携带旧的 Cookie，也会因后端状态已销毁而被要求重新登录。
6. **清除前端凭证并反馈**：`CAS Server` 通过 HTTP 响应头向 Browser 发送 `Set-Cookie` 指令，将 `CASTGC` 的值置空，并设置过期时间（使其立刻被 Browser 删除）。随后返回一个标准的注销成功 HTML 页面。

# 三、SAML 2.0

安全断言标记语言（Security Assertion Markup Language, SAML 2.0）是面向企业级场景的标准化联邦身份单点登录协议，基于可扩展标记语言（Extensible Markup Language, XML）封装身份信息，依托非对称加密与数字签名构建信任体系，是政企内网、跨组织身份联合、大型 SaaS 平台企业版的主流身份认证方案。

与中央认证服务（Central Authentication Service, CAS）相比，SAML 2.0 具备标准化身份属性传递、跨域信任互通、强密码学安全校验的能力，解决了轻量化协议在高安全场景下的能力缺陷；协议交互依赖浏览器重定向与表单自动提交完成报文传输，无客户端状态依赖，适配多架构、多域名的复杂部署环境，同时也是内网渗透测试中身份伪造、签名绕过、断言篡改的核心测试目标。

### 1.2 名词解析

| **专业名词**     | **英文全称 / 扩展描述**        | **在架构中的绝对作用**                                       |
| ---------------- | ------------------------------ | ------------------------------------------------------------ |
| **IdP**          | Identity Provider (身份提供者) | 拥有用户身份数据库的核心服务，负责验证用户密码，并向信任的 SP 签发包含用户信息的 SAML Assertion。 |
| **SP**           | Service Provider (服务提供者)  | 提供具体业务功能的应用系统。它信任 IdP，依赖 IdP 发来的 Assertion 来决定是否允许用户访问。 |
| **AuthnRequest** | Authentication Request         | 由 SP 生成的 XML 格式请求，通过 Browser 传递给 IdP，告知 IdP：“我需要你帮我验证这个用户的身份”。 |
| **Assertion**    | SAML Assertion (SAML 断言)     | IdP 生成的核心安全凭证（XML 格式），里面包含了被验证用户的身份标识（Subject）、属性（Attributes）以及有效时间等。这是 SSO 的核心载体。 |
| **Signature**    | XML Digital Signature          | IdP 使用其私钥对 Assertion 附加的数字签名。这是防篡改的根本保证，SP 必须使用预先配置的 IdP 公钥来验证此签名。 |
| **SAMLResponse** | SAML Response                  | 一个包裹着 `Assertion` 的信封（XML 格式）。IdP 将验证结果装入 Response 中发给 SP。 |
| **ACS**          | Assertion Consumer Service     | SP 专门暴露出来用于接收和处理 `SAMLResponse` 的独立网络接口（端点）。这是 SAML 登录流程在 SP 侧的收口。 |
| **Bindings**     | Protocol Bindings (协议绑定)   | 规定了 SAML 消息如何通过底层 HTTP 协议传输。最常见的是 **HTTP Redirect Binding**（用于发送轻量级的 `AuthnRequest`，放在 URL 中）和 **HTTP POST Binding**（用于发送体积庞大的 `SAMLResponse`，放在 POST Body 中）。 |

### 1.3 步骤解析

#### 阶段一：首次访问与身份验证 (Phase 1)

这是标准 SP-Initiated（服务提供者发起）的核心流程。

1. **请求资源**：User 尝试访问 `SP1` 的受保护页面。
2. **生成验证请求**：`SP1` 检查到没有本地会话，生成一个 Base64 编码的 `SAML AuthnRequest` XML 数据。
3. **HTTP 重定向**：`SP1` 通过 302 状态码，指示 Browser 跳转至 IdP，并将 `AuthnRequest` 作为 URL 参数携带（HTTP Redirect Binding）。
4. **送达请求**：Browser 向 IdP 发起 GET 请求。
5. **身份质询**：IdP 发现 Browser 没有携带自身的全局会话 Cookie，返回一个 HTML 登录表单。
6. **提交凭据**：User 输入账号密码，Browser POST 至 IdP。
7. **验证并生成凭证**：IdP 验证账密。成功后，IdP 创建一个带有数字签名的 `SAML Assertion`。
8. **返回断言表单**：IdP 返回一个包含一段 JavaScript 代码和隐藏 HTML 表单的页面，隐藏表单中包含了 `SAMLResponse`。同时，IdP 种下代表**全局 SSO 会话的 Cookie**（如 `idp-session-id`）。
9. **提交断言**：Browser 加载页面后，触发 JS 自动执行 POST 请求，将庞大的 `SAMLResponse` 数据投递给 `SP1` 的 ACS 端点（HTTP POST Binding）。
10. **签名与会话验证**：`SP1` 的 ACS 接收到数据，使用 IdP 的公钥严格校验签名防伪。解析 XML 获取用户身份，在本地内存/Redis 建立应用级 Session。
11. **局部会话建立**：`SP1` 种下自己的本地 Session Cookie（如 `JSESSIONID`），重定向用户至最初请求的页面，完成整个登录闭环。

#### 阶段二：同应用二次访问 (Phase 2)

1. **后续访问**：User 在 `SP1` 内继续操作。由于 Browser 已持有 `SP1` 的 `JSESSIONID`，后续的每次 HTTP 请求都会自动携带它。`SP1` 直接校验本地 Session，不再与 IdP 发生任何联系。

#### 阶段三：跨域免密登录 (Phase 3)

单点登录（SSO）魔力发生的地方。

1. **访问新应用**：User 访问全新的业务系统 `SP2`。`SP2` 同样发现无本地会话，引导 Browser 携带针对 `SP2` 的 `AuthnRequest` 重定向至 IdP。
2. **全局 Cookie 生效**：Browser 在向 IdP 发起请求时，**自动携带了阶段一第 8 步中获取的 `idp-session-id`**。
3. **跳过质询**：IdP 校验该全局 Cookie 依然有效，**直接跳过显示登录界面的步骤**，提取用户信息。
4. **签发新凭证**：IdP 迅速生成一份专属给 `SP2` 的新 `SAMLResponse`，通过自动表单提交的方式发往 `SP2` 的 ACS。
5. **建立新会话**：`SP2` 校验通过，下发属于 `SP2` 的本地 Session Cookie。User 瞬间进入 `SP2` 内部，体验到了无缝的跨系统免密穿越。

# 四、开放授权框架 OAuth 2.0

OAuth （Open Authorization）是一种常用的**授权框架**，它允许网站和 Web 应用在有限范围内访问用户在另一应用上的账号信息。至关重要的是，OAuth 支持用户授予此类访问权限，**同时无需向提出请求的应用暴露自身的登录凭据**。这意味着用户可以精细控制自己希望共享的数据范围，而不必将账号的完整控制权交给第三方。 OAuth 的基本流程被广泛用于集成第三方功能，这类功能通常需要访问用户账号中的特定数据。例如，某应用可能通过 OAuth 请求访问你的邮箱联系人列表，以便为你推荐可关联的对象。

不过，同一机制也被用于提供**第三方认证服务**，允许用户使用自己在其他网站上已有的账号完成登录。



## 1.  角色模型

OAuth 2.0 最初是为了在应用之间共享对特定数据的访问权限而设计的。它通过定义**三个独立参与方**之间的一系列交互来实现工作，这三方分别是：客户端应用（Client ）、资源所有者（Resource Owner ），以及 OAuth 服务提供方（OAuth service provider.）。

> [!IMPORTANT]
>
> 标准的四角色模型是：
>
> 1. **Resource Owner（资源所有者）**
> 2. **Client（客户端）**
> 3. **Authorization Server（授权服务器）**
> 4. **Resource Server（资源服务器）**
>
> 上面描述的OAuth service provider. 是包含授权服务器 和资源服务器的组合

##### client 

客户端可以是**移动应用**或**服务器端 Web 应用。** 它充当中间人，请求访问资源并执行资源所有者允许的操作。例如，我们常见的使用google账户登录其他的网站，此时这个网站就是client：

##### Resource Owner 

资源所有者（Resource Owner）是指对特定数据拥有控制权、并可授权某个应用以其名义访问该数据的个人或系统。该概念的核心在于围绕**用户授权与控制权**展开，是整个体系的基础。

##### Authorization Server 

授权服务器负责在成功验证资源所有者身份并获得其授权后， **向客户端颁发访问令牌** 。授权服务器在 OAuth 流程中扮演着至关重要的角色，它确保只有在合法用户身份验证和同意后，客户端才能获得权限。

##### Resource Server

托管受保护资源的服务器，可接收并处理携带访问令牌（Access Token）的受保护资源请求。该服务器会确保，只有完成身份认证并获得授权的客户端，才能访问或操作资源所有者（Resource Owner）的数据。

## 2.其他术语

1. **访问令牌 (Access Token)**
   一种代表“资源所有者（用户）”授权的凭证，客户端（client）通过它来访问受保护的资源。它具备**有限的有效期（Lifespan）\**和\**特定的权限范围（Scope）**。

   ------

   Access Token 是客户端与资源服务器之间安全通信的基石。它避免了客户端在每次请求数据时都向用户索要账号密码。

   ------

   当你在应用中通过 OAuth 成功登录后，应用会获取一个 Access Token。在令牌有效期内（通常为几分钟到几小时），应用可以代表你执行下单、查看个人资料等操作，而无需你重复输入密码。

2. **刷新令牌 (Refresh Token)**
   一种专门用于获取新 Access Token 的凭证，其目的是在不要求用户重新进行身份验证（即无需重新登录）的情况下，延长会话生命周期。

   ****
   Refresh Token 通常具有**较长的有效期**（数天或数月）。它解决了 Access Token 过期频繁导致的用户体验中断问题。

   ****
   当 Access Token 过期失效（接口返回 401 错误）时，客户端应用会在后台自动使用 Refresh Token 请求授权服务器。服务器验证无误后，会颁发一个新的 Access Token。整个过程对用户透明，用户无需再次手动登录。

3. **重定向 URI (Redirect URI)**

   当资源所有者（用户）批准或拒绝授权请求后，授权服务器会将用户的浏览器（User-agent）重定向至该 URI。它的作用是确保接收授权响应的客户端是合法且正确的。

   该 URI 扮演着身份校验的角色。授权服务器会比对“请求中的 URI”与“应用注册时预留的 URI”是否一致。只有匹配成功，服务器才会发送授权码（Code）。

4. **权限范围（Scope）**
   权限范围是用于限制应用对用户账户访问权限的一种机制。它允许客户端声明所需的访问级别，同时让授权服务器向用户告知应用正在申请的访问权限。权限范围有助于践行最小权限原则。

   无论使用哪种 OAuth 授权类型，客户端应用都必须指定其想要访问的数据以及想要执行的操作类型。这是通过向 OAuth 服务发送授权请求时携带 `scope` 参数来实现的。

   对于基础的 OAuth，客户端应用可以申请访问的 **作用域（Scopes）** 取决于具体的 OAuth 服务。由于作用域名称只是一个任意的文本字符串，不同提供商之间的格式差异可能非常大。有些甚至使用完整的 URI 作为作用域名称（类似于 REST API 端点）。例如，在请求对用户联系人列表的读取权限时，根据所使用的 OAuth 服务，作用域名称可能会呈现以下任何形式：

   - `scope=contacts`
   - `scope=contacts.read`
   - `scope=contact-list-r`
   - `scope=https://oauth-authorization-server.com`

   然而，当 OAuth 用于身份验证时，通常会使用标准化的 **OpenID Connect** 作用域。例如，`scope=openid profile` 将授予客户端应用对预定义的一组用户基本信息（如电子邮件地址、用户名等）的读取权限。我们稍后会详细讨论 OpenID Connect。

5. **State 参数（State Parameter）**
   一个可选参数，用于维持客户端与授权服务器之间的状态关联。通过确保响应与客户端的原始请求相匹配，该参数可防范 CSRF 攻击，是保障 OAuth 流程安全的关键部分。例如，当你发起登录流程时，咖啡店应用会向授权服务器发送一个 State 参数。该参数能够确保你收到的响应与初始请求相关联，抵御特定类型的攻击。
6. **令牌端点与授权端点（Token Endpoint & Authorization Endpoint）**
   授权服务器的**令牌端点**，是客户端使用授权许可（或刷新令牌）交换访问令牌的接口。
   与之相对，**授权端点**是资源所有者完成身份认证、并授权客户端访问受保护资源的接口。



## 3. 授权类型

OAuth **授权类型（Grant Type）** 决定了 OAuth 流程中涉及的具体步骤序列。授权类型还会影响客户端应用在每个阶段与 OAuth 服务通信的方式，包括**访问令牌（Access Token）**本身的传输方式。因此，授权类型通常也被称为“**OAuth 流程（OAuth Flows）**”。

在客户端应用启动相应流程之前，必须先在 OAuth 服务中配置好对特定授权类型的支持。客户端应用在发送给 OAuth 服务的初始**授权请求（Authorization Request）**中，会指定其想要使用的授权类型。

OAuth 2.0 提供了多种授权类型，以适应不同的场景和客户端类型。这些授权类型定义了应用程序如何获取访问令牌，从而代表资源所有者访问受保护的资源。本文将讨论四种主要的 OAuth 2.0 授权类型，重点关注“**授权码（Authorization Code）**”和“**隐式（Implicit）**”授权类型，因为这两种是目前应用最为广泛的授权模式。总体而言，这两种授权类型均包含以下阶段：

1. 客户端应用申请访问用户的部分数据，并在请求中指明要使用的授权类型以及所需的访问权限。
2. 用户被引导至 OAuth 服务方进行登录，并明确对申请的访问权限进行授权确认。
3. 客户端应用获取一个唯一的**访问令牌（Access Token）**，该令牌用于证明其已获得用户许可，可访问申请的数据。具体获取方式会因授权类型的不同而存在显著差异。
4. 客户端应用使用该访问令牌发起 API 调用，从资源服务器中获取对应的数据。

在学习 OAuth 如何用于身份认证之前，先理解 OAuth 这一基础流程的核心原理非常重要。

### 3.1 Authorization Code  类型

授权码模式（Authorization Code Grant）是 OAuth 2.0 中最常用的流程，非常适合 PHP、Java、.NET 等服务端应用。在该流程中，
客户端首先将用户重定向到认证服务器，由用户完成身份验证并授予权限。
随后，认证服务器会将用户重定向回客户端，并附带一个“授权码”。
最后，客户端通过请求认证服务器的令牌端点（Token Endpoint），将该授权码兑换为访问令牌（Access Token）。
这种授权方式以其增强的安全性而著称，因为授权码是在服务器之间交换访问令牌的，这意味着访问令牌不会暴露给用户代理（例如浏览器），从而降低了令牌泄露的风险。它还支持使用刷新令牌来维持长期访问权限，而无需用户重复进行身份验证。

![mermaid-diagram-2026-04-13-105326](./image/2.%E5%8D%95%E7%82%B9%E7%99%BB%E5%BD%95%E5%90%84%E5%8D%8F%E8%AE%AE%E5%8E%9F%E7%90%86/mermaid-diagram-2026-04-13-105326.png)

简而言之，客户端应用与 OAuth 服务首先通过重定向交互一系列基于浏览器的 HTTP 请求，从而启动整个流程。
系统会提示用户是否同意应用所申请的访问权限。
若用户同意，客户端应用将获得一个**授权码（authorization code）**。
随后，客户端应用使用该授权码与 OAuth 服务进行交换，以获取**访问令牌（access token）**，并可凭借该令牌调用 API 接口获取相应用户数据。 
从授权码与令牌的交换环节开始，后续所有通信均通过预先配置好的安全后端通道在服务器之间传输，因此对终端用户不可见。这条安全通道在客户端应用首次向 OAuth 服务注册时便已建立。在此过程中还会生成一个**客户端密钥（client_secret）**，客户端应用在发送此类服务器间请求时，必须使用该密钥完成自身身份验证。 由于最敏感的数据（访问令牌与用户数据）不会通过浏览器传输，这种授权类型可以说是安全性最高的。服务端应用在条件允许的情况下，应尽可能始终采用该授权类型。

![Flow for the OAuth authorization code grant type](./image/2.%E5%8D%95%E7%82%B9%E7%99%BB%E5%BD%95%E5%90%84%E5%8D%8F%E8%AE%AE%E5%8E%9F%E7%90%86/oauth-authorization-code-flow.jpg)

#### 1. 授权请求（Authorization request）

客户端应用向 OAuth 服务的 `/authorization` 端点发送请求，申请获取访问特定用户数据的权限。注意，不同服务提供商的端点路径可能有所不同 —— 我们的实验环境使用 `/auth` 端点实现该功能。但你始终可以根据请求中携带的参数识别出该端点。

```http
GET /authorization?client_id=12345&redirect_uri=https://client-app.com/callback&response_type=code&scope=openid%20profile&state=ae13d489bd00e3c24 HTTP/1.1
Host: oauth-authorization-server.com
```

该请求包含以下值得注意的参数，这些参数通常出现在查询字符串中：

**client_id** 
必选参数，存放客户端应用的唯一标识符。该值在客户端应用向 OAuth 服务注册时生成。

**redirect_uri** 
它是指在将授权码（Authorization Code）发送给客户端应用时，用户浏览器应当被重定向的目标 URI。该地址通常也被称为**“回调 URI (Callback URI)”**或**“回调端点 (Callback Endpoint)”**。许多 OAuth 攻击正是基于对该参数**验证机制的缺陷**而展开的。

**response_type**
决定客户端应用期望接收的响应类型，进而决定其希望发起的授权流程。对于授权码授权类型，该参数值应为 `code`。

**scope**
用于指定客户端应用希望访问的用户数据子集。注意，这些权限范围既可以是 OAuth 提供商自定义的，也可以是 OpenID Connect 规范定义的标准化权限范围。我们将在后续内容中更详细地介绍 OpenID Connect。

**state**
存储一个与客户端应用当前会话绑定的、唯一且不可预测的值。OAuth 服务在返回授权码的同时，应原样返回该值。该参数相当于一种 CSRF 令牌，确保发往客户端 `/callback` 端点的请求来自发起 OAuth 流程的同一用户。

#### 2. 用户登录与授权确认

当授权服务器接收到初始请求后，会将用户重定向至登录页面，提示用户登录其在该 OAuth 提供商处的账号。例如，这通常是用户的社交媒体账号。随后，用户会看到客户端应用申请访问的数据清单，该清单依据授权请求中定义的权限范围生成。用户可以选择是否同意此次访问。

需要重点注意的是，一旦用户为某客户端应用批准了指定权限范围，只要用户在 OAuth 服务上仍存在有效会话，该步骤就会自动完成。换句话说，用户首次选择 “使用社交媒体账号登录” 时，需要手动登录并授予权限；但如果之后再次访问该客户端应用，通常只需单击即可完成登录。

#### 3.授权码下发（Authorization code grant）

若用户同意所申请的访问权限，其浏览器将被重定向至授权请求中 `redirect_uri` 参数指定的 `/callback` 端点。最终产生的 GET 请求会将授权码作为查询参数携带。根据配置不同，响应中还可能携带与授权请求中值相同的 `state` 参数。

```http
GET /callback?code=a1b2c3d4e5f6g7h8&state=ae13d489bd00e3c24 HTTP/1.1
Host: client-app.com
```

#### 4. 访问令牌请求（Access token request）

客户端应用程序收到授权码后，需要将其交换为访问令牌。为此，它会向 OAuth 服务的 `/token` 端点发送服务器到服务器的 `POST` 请求。从此之后的所有通信都在安全的后台通道中进行，因此攻击者通常无法观察或控制这些通信。

```http
POST /token HTTP/1.1
Host: oauth-authorization-server.com
…
client_id=12345&client_secret=SECRET&redirect_uri=https://client-app.com/callback&grant_type=authorization_code&code=a1b2c3d4e5f6g7h8
```

除了 `client_id` 和授权 `code` 之外，您还会注意到以下新参数：

- **client_secret**
  客户端应用程序必须通过包含其在向 OAuth 服务注册时分配的密钥来验证自身身份。
- **grant_type**
  用于让令牌端点明确客户端应用希望使用的授权类型。在授权码流程中，该参数应设为 `authorization_code`。

#### 5.  访问令牌下发（Access token grant）

OAuth 服务会对访问令牌请求进行校验。若一切符合预期，服务器将向客户端应用下发包含申请权限范围的访问令牌作为响应。

```json
{
    "access_token": "z0y9x8w7v6u5",
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "openid profile",
    …
}
```

#### 6. API 调用（API call）

此时客户端应用已获取访问令牌，终于可以从资源服务器拉取用户数据。为此，客户端向 OAuth 服务的 `/userinfo` 端点发起 API 调用，并在 `Authorization: Bearer` 请求头中提交访问令牌，以证明其拥有访问该数据的权限。

```http
GET /userinfo HTTP/1.1
Host: oauth-resource-server.com
Authorization: Bearer z0y9x8w7v6u5
```

#### 7.资源返回（Resource grant）

资源服务器应校验令牌是否有效、是否归属当前客户端应用。若校验通过，服务器将返回所请求的资源，即依据访问令牌权限范围对应的用户数据。

```json
{
    "username":"carlos",
    "email":"carlos@carlos-montoya.net",
    …
}
```

客户端应用最终可将该数据用于既定用途。在 OAuth 身份认证场景下，该数据通常会作为身份标识，为用户创建已认证会话，从而完成登录。



### 3.2 隐式授权类型

隐式（Implicit）授权类型要简单得多。它无需先获取授权码再将其交换为访问令牌，而是在用户授予授权后立即接收访问令牌。您可能想知道为什么客户端应用程序并非总是使用隐式授权类型。答案很简单——它的安全性要低得多。使用隐式授权类型时，所有通信都通过浏览器重定向进行，不像授权码流程那样存在安全的后端通道。这意味着敏感的访问令牌和用户数据更容易受到潜在攻击。此外，它**也不支持刷新令牌** 。

隐式授权类型更适合单页应用程序和原生桌面应用程序，这些应用程序无法轻易地在后端存储 `client_secret` ，因此无法从使用授权码授权类型中获益。

![Flow for the OAuth implicit grant type](./image/2.%E5%8D%95%E7%82%B9%E7%99%BB%E5%BD%95%E5%90%84%E5%8D%8F%E8%AE%AE%E5%8E%9F%E7%90%86/oauth-implicit-flow.jpg)

#### 1. 授权请求

隐式授权流程的启动方式与授权码授权流程非常相似。唯一的显著区别在于 `response_type` 参数必须设置为 `token` 。

```http
GET /authorization?client_id=12345&redirect_uri=https://client-app.com/callback&response_type=token&scope=openid%20profile&state=ae13d489bd00e3c24 HTTP/1.1 Host: oauth-authorization-server.com
```

#### 2. 用户登录和同意

用户登录后，决定是否同意所请求的权限。此过程与授权码流程完全相同。

#### 3. 访问令牌授权

如果用户同意所请求的访问权限，情况就会有所不同。OAuth 服务会将用户的浏览器重定向到授权请求中指定的 `redirect_uri` 。但是，它不会发送包含授权码的查询参数，而是将访问令牌和其他令牌特定数据作为 URL 片段标识符发送。

```http
GET /callback#access_token=z0y9x8w7v6u5&token_type=Bearer&expires_in=5000&scope=openid%20profile&state=ae13d489bd00e3c24 HTTP/1.1
Host: client-app.com
```

由于访问令牌是以 URL 片段的形式发送的，因此它不会直接发送给客户端应用程序。客户端应用程序必须使用合适的脚本来提取该片段并将其存储。

#### 4. API 调用

客户端应用成功从 URL 片段中提取访问令牌后，便可使用该令牌向 OAuth 服务的 `/userinfo` 端点发起 API 调用。**与授权码流程不同的是，这一步是通过浏览器完成的。**

```http
GET /userinfo HTTP/1.1
Host: oauth-resource-server.com
Authorization: Bearer z0y9x8w7v6u5
```

#### 5.资源返回

资源服务器应验证令牌是否有效，以及它是否属于当前客户端应用程序。如果是，它将根据与访问令牌关联的权限范围，发送请求的资源（即用户数据）作为响应。

```bash
{
    "username":"carlos",
    "email":"carlos@carlos-montoya.net"
}
```

客户端应用程序最终可以将这些数据用于其预期用途。在 OAuth 身份验证中，它通常会被用作 ID，以授予用户已验证的会话，从而实现用户登录。

### 3.3 资源所有者密码凭证授权类型

Resource Owner Password Credentials Grant 这种 Grant Type 要求 User（Resource Owner）将其明文凭据（如用户名和密码）直接提供给 **Client Application**。由于这种模式完全打破了 OAuth 2.0 “避免向第三方暴露凭据”的核心安全设计初衷，它仅被允许在 **Client Application** 受到 **Resource Owner** 绝对信任（例如操作系统级别的官方原生应用）或作为旧系统迁移的遗留方案时使用。所有通信均在安全的 Back-channel 中进行，不涉及浏览器 Redirect 流程。

![Resource Owner Password Credentials Grant sequence diagram](./image/2.%E5%8D%95%E7%82%B9%E7%99%BB%E5%BD%95%E5%90%84%E5%8D%8F%E8%AE%AE%E5%8E%9F%E7%90%86/62a7685ca6e7ce005d3f3afe-1724169940244.png)

#### 1，Access token request

由于不需要前端的 Consent 流程，Client Application 在获取到 User 的明文账号密码后，直接向 Authorization Server 的 `/token` 端点发送服务器到服务器的 POST 请求。

```http
POST /token HTTP/1.1
Host: oauth-authorization-server.com
Content-Type: application/x-www-form-urlencoded

grant_type=password&username=carlos&password=qwerty&client_id=12345&client_secret=SECRET
```

此请求包含以下核心参数：

`grant_type`：必须设置为 `password`，以明确指示 Authorization Server 当前使用的流程。

`username` 与 `password`：Resource Owner 的明文凭据。

`client_id` 与 `client_secret`：用于验证 Client Application 本身的合法性。

#### 2. Access token grant

Authorization Server 验证 User 的账号密码以及 Client Application 的身份。如果一切符合预期，服务器将在响应中直接颁发 Access Token。

```json
{
  "access_token": "z0y9x8w7v6u5",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

#### 3.  API call

Client Application 获得 Access Token 后，即可向 Resource Server 发起 API 请求。此时的逻辑与之前介绍的流程完全一致，将 Access Token 放置在 `Authorization: Bearer` header 中。

#### 4. Resource grant

Resource Server 校验 Token 无误后，返回所请求的受保护资源数据。



### 3.4 客户凭证授权类型

客户端凭证授权用于无需用户参与的服务器间交互。客户端使用其凭证向授权服务器进行身份验证并获取访问令牌。在此流程中，客户端使用其客户端凭证（客户端 ID 和密钥）向授权服务器进行身份验证，授权服务器直接向客户端颁发访问令牌，专门用于**机器对机器（Machine-to-Machine, M2M）通信**，如下所示：，

![Client Credentials Grant sequence diagram](./image/2.%E5%8D%95%E7%82%B9%E7%99%BB%E5%BD%95%E5%90%84%E5%8D%8F%E8%AE%AE%E5%8E%9F%E7%90%86/62a7685ca6e7ce005d3f3afe-1724170002373.png)

#### 1. Access token request

Client Application 直接向 Authorization Server 的 `/token` 端点发送 POST 请求。由于没有 User 参与，请求中仅包含 Client Application 自身的身份凭据。

```http
POST /token HTTP/1.1
Host: oauth-authorization-server.com
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id=12345&client_secret=SECRET
```

核心参数说明：

1. **grant_type**：必须设置为 `client_credentials`。
2. **client_id** 与 **client_secret**：Client Application 证明自己身份的机密参数。

#### 2. Access token grant

Authorization Server 对 `client_secret` 进行鉴权。验证通过后，直接向 Client Application 颁发 Access Token。由于此流程不涉及具体的 User 授权，通常不会返回 Refresh Token。

```
{
  "access_token": "a1b2c3d4e5f6",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

#### 3.API call

Client Application 携带此 Access Token 向内部的 Resource Server 发起 API 调用。

#### 4. Resource grant

Resource Server 验证 Token 的合法性及 Scope 后，返回对应的内部服务数据。

# 五、OpenID Connect

**OpenID Connect** 在 OAuth 协议基础上进行了扩展，在基础 OAuth 实现之上提供了专门的身份与认证层。它增加了一些简易功能，让 OAuth 能更好地支持身份认证场景。

OAuth 最初设计时并未考虑身份认证，其初衷是在应用之间实现对特定资源的授权委托。然而，许多网站开始对 OAuth 进行定制，将其用作身份认证机制。为此，它们通常会申请对部分基础用户数据的读取权限，如果获得授权，就认为用户已在 OAuth 服务提供方一侧完成了身份认证。

这类单纯基于 OAuth 的认证机制远非理想方案。首先，客户端应用无法得知用户是在何时、何地、以何种方式完成认证的。由于这些实现方式大多属于定制化的变通方案，因此也不存在用于获取用户身份数据的标准方式。想要正常使用 OAuth，客户端应用必须为不同服务提供方分别配置独立的 OAuth 机制，每个提供方都拥有不同的接口端点、专属权限范围等。

OpenID Connect 通过加入标准化的身份相关功能，解决了大量此类问题，让基于 OAuth 的身份认证更加可靠、统一。

> [!NOTE]
>
> 单纯的 OAuth 2.0 只负责**授权（Authorization）**，即“允许你进入房间”，但不验证“你是谁”。
> OIDC 在 OAuth 之上增加了一个**身份验证（Authentication）**层，专门用来确认用户的身份。

## 1. OpenID Connect 工作原理

OpenID Connect 可以完美融入标准的 OAuth 流程。从客户端应用的角度来看，主要区别在于：它新增了一套**适用于所有服务提供方的标准化作用域（scope）**，以及一个额外的响应类型：**id_token**。

### 1.1 角色模型

OpenID Connect 的角色与标准 OAuth 基本一致，主要区别在于规范中使用的术语略有不同。

- **依赖方（Relying party）**

  请求对用户进行身份认证的应用，等同于 OAuth 中的**客户端应用**。

- **最终用户（End user）**

  正在被认证的用户，等同于 OAuth 中的**资源所有者**。

- **OpenID 提供商（OpenID provider）**

  已配置支持 OpenID Connect 的 OAuth 服务。

### 1.2 OpenID Connect claims和scopes

“**claims**” 一词指的是表示资源服务器上用户信息的 `key:value` 对。例如， `"family_name":"Montoya"` 就是一个声明。

与基础 OAuth 不同（各服务提供方的作用域各不相同），所有 OpenID Connect 服务都使用**完全相同的一套标准作用域**。要使用 OpenID Connect，客户端应用程序必须在授权请求参数中包含**scope=openid**。它们可以包含一个或多个其他标准权限范围：

- profile
- email
- address
- phone

每个作用域都对应对 OpenID 规范中定义的一组用户声明的读取权限。

例如，请求作用域scope= `openid profile` 会授予客户端应用读取一系列与用户身份相关声明的权限，如 `family_name`、`given_name`、`birth_date` 等。

### 1.3 ID token

OpenID Connect 带来的另一项主要增强是 **`id_token` 响应类型**。该类型会返回一个经过 **JSON Web 签名 (JWS)** 加密的 **JSON Web Token (JWT)**。

- **载荷与信息：** JWT 的载荷（Payload）包含了一系列基于初始请求**作用域（Scope）\**的\**声明（Claims）**。此外，它还记录了用户最后一次通过 OAuth 服务进行身份验证的时间和方式。客户端应用可以根据这些信息，自行判断该用户的认证强度是否满足要求。
- **性能优势：** 使用 `id_token` 的主要好处是减少了客户端应用与 OAuth 服务之间往返请求的次数，从而提升整体性能。客户端无需先获取访问令牌（Access Token）再单独请求用户数据，而是在用户完成认证后，直接接收包含这些数据的 ID Token。
- **安全机制：** 与基础 OAuth 仅依赖“受信任通道”不同，ID Token 传输的数据完整性是基于 **JWT 密码学签名**的。因此，使用 ID Token 有助于抵御某些**中间人（MITM）攻击**。然而，由于用于验证签名的加密密钥通常也在同一网络通道中传输（通常公开在 `/.well-known/jwks.json`），因此某些攻击手段依然可行。

**注：** 由于 OAuth 支持多种响应类型，客户端应用在发送授权请求时，可以同时请求基础 OAuth 类型和 OIDC 的 `id_token` 类型，这是完全合规的：

- `response_type=id_token token`（混合模式：同时返回 ID Token 和 访问令牌）
- `response_type=id_token code`（混合模式：同时返回 ID Token 和 授权码）

在这种情况下，客户端应用会同时收到 ID Token 以及授权码（Code）或访问令牌（Access Token）。

### 1.4 识别 OpenID Connect (OIDC)

如果客户端应用正在使用 OpenID Connect，从**授权请求**中通常能一眼看出。最稳妥的检查方法是查看是否存在必须包含的 `openid` 作用域（scope）。

即使登录流程最初看起来并未使用 OIDC，仍值得检查该 OAuth 服务是否支持它。你可以简单地尝试**添加 `openid` 作用域**，或将响应类型（response type）**改为 `id_token`**，观察是否会导致报错。

与基础 OAuth 一样，查阅 OAuth 提供商的文档也是个好主意，看看其中是否有关于其 OIDC 支持的有用信息。你通常还可以通过标准端点 **`/.well-known/openid-configuration`** 访问其配置文件。
