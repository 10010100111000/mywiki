# SQL Injection - MySQL comment: the double dash mystery

**发布时间**：2020年05月24日

**更新时间**：2025年09月27日

**作者**：noraj (Alexandre ZANNI)

**分类**：security , sql, web

## Intro

在采用黑盒测试方法进行渗透测试时，你可以通过常见载荷进行模糊测试，以此发现SQL注入漏洞（SQLi）。除非有详细的错误信息披露原始的SQL查询语句，否则你无法知晓查询的结构。你的输入可能会注入到查询语句的开头，也可能在查询中被多次注入，这些都是未知的。

因此，为了避免出现不受控制的意外副作用，渗透测试人员会希望在载荷末尾添加注释，以此中和并终止后续的查询语句，让查询的执行行为变得更可预测。

## Which comment?

不同的数据库管理系统（DBMS）支持的注释语法不同：部分支持 `#`或 `--`这类单行注释，部分支持相对少见的、类空字节注入的 `%00`注释，还有部分支持 `/* */`这类多行注释，同时也存在一些更特殊、不常见的注释语法。各类数据库管理系统的注释支持情况差异显著。

在黑盒测试场景下，测试人员往往无法知晓目标系统使用的是哪种DBMS，且对DBMS的探测工作也存在难度，因此更倾向于使用兼容性最强的注释语法。

需要注意的是，**不建议使用未闭合的多行注释**——这类注释可能会过度注释代码，若未完成闭合，甚至还会触发报错。

Oracle MySQL、MariaDB、Percona Server、Oracle Database、PostgreSQL、Microsoft SQL Server、SQLite、Apache Ignite、Firebird、IBM DB2等主流数据库，均对结构化查询语言（SQL）做了各自的定制化实现。

**双破折号** **`--`** **是标准SQL中唯一的单行注释语法**，同时也是所有DBMS都支持的注释方式。

## The double dash mystery

既然所有DBMS都支持 `--`注释，那为什么绝大多数注入载荷的末尾使用的是 `-- -`或 `--+`，而非单独的 `--`呢？

答案在于**MySQL的语法特殊性**——这一点是导致该现象的核心原因。

此前所说的“所有DBMS都支持双破折号注释”并非绝对准确，**MySQL是一个例外**。

在标准SQL中，`--`后的所有内容直至行尾都会被视为注释；但在MySQL中，`--`（双破折号）注释语法要求第二个破折号后**必须紧跟至少一个空白字符或控制字符**（如空格、制表符、换行符等）。该语法与标准SQL的注释语法存在细微差异。

现实中，SQL注入漏洞大多通过Web应用的HTTP协议进行利用，而Web浏览器、Web框架、应用后端、代理服务器、底层开发语言等环节，**都有可能截断末尾的空白字符**。这就导致 `--<空格>`的写法可能会被转换为单纯的 `--`，进而被MySQL判定为无效的注释语法。

而MySQL又是目前使用最广泛的DBMS之一，因此我们需要设计能兼容它的通用注入载荷，这也是 `-- -`这类写法出现的原因：在空白字符后再添加任意字符，避免空白字符出现在载荷末尾，从而防止其被截断。实际上，`-- -`与 `-- a`、`-- Z`、`-- !`这类写法的作用完全一致。

那 `--+`的写法又从何而来？

原因是在URL标准中，**空格字符会被编码为加号** **`+`**（例如UTF-8编码的空格U+0020会被编码为U+002B）。当URL被解码时，加号会还原为空格，`--+`也就随之转换为符合MySQL要求的 `--<空格>`。

注：这也是**base64url**（即适用于URL和文件名的Base64编码，遵循RFC 4648标准）使用 `-`替代 `+`、使用 `_`替代 `/`的原因。

但需要注意的是，若 `--+`本身被做了URL编码（会转换为 `--%2B`），那么在解码后，该写法仍无法被MySQL识别为有效的注释语法。

因此，使用 `--<空格><任意字符>`（如 `-- -`）的写法会更安全——即便其被URL编码为 `--%20-`，解码后仍能保持 `-- -`的形式，可被MySQL正常识别。

## Why MySQL requires a whitespace character after the double dash?

查阅MySQL官方文档可知，该设计的初衷是**为了避免自动生成的SQL查询语句出现语法问题**，以下面的场景为例：

假设存在自动生成的更新语句，会将 `payment`字段的值自动插入语句中：

```SQL
UPDATE account SET credit=credit-payment
```

若 `payment`的取值为负数（如 `-1`），语句会变为：

```SQL
UPDATE account SET credit=credit--1
```

在SQL中，`credit--1`本身是一个合法的表达式，但如果 `--`被直接识别为注释的起始符，表达式的一部分会被丢弃，最终语句会变为：

```SQL
UPDATE account SET credit=credit
```

该语句执行后不会对数据产生任何修改，与原本的业务意图完全相悖。这一案例也说明，若允许 `--`直接作为注释起始符，可能会引发严重的业务问题。

因此，MySQL的实现要求 `--`后必须紧跟一个空格，才能被识别为注释的起始序列，这样一来，`credit--1`就能被正常解析为合法表达式，避免了上述问题。

此外，MySQL的命令行客户端还存在一个安全特性：**会忽略以** **`--`** **开头的行**。

## Conclusion

在SQL注入中，**`-- -`** **是兼容性最强、最安全的单行注释写法**：单独的 `--`无法在MySQL中生效，而 `--+`若被编码则会失效，二者均存在明显的使用局限性。
