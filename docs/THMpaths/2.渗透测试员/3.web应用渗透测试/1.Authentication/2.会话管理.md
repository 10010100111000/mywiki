# 一、什么是会话管理？

在讨论所有可能出现的问题之前，我们需要了解会话管理。正如之前的任务中提到的，您无需在每次请求中都发送用户名和密码。然而， HTTP 协议本质上是无状态的。因此，会话用于跟踪用户在使用 Web 应用程序期间的活动。会话管理就是管理这些会话并确保其安全的过程。

### 1.1 会话创建

你可能认为生命周期的第一步只有在你提供凭据（例如用户名和密码）之后才会发生。然而，在许多 Web 应用程序中，初始会话在你访问应用程序时就已经创建了。这是因为有些应用程序甚至在身份验证之前就希望跟踪你的操作。不过，本讨论室的主要重点是已验证的会话。一旦你提供用户名和密码，你就会收到一个会话值，该值会随每个新请求一起发送。这些会话值的生成、使用和存储方式对于确保会话创建的安全性至关重要。

### 1.2 会话跟踪

一旦收到会话值，该值就会随每个新请求一起提交。这使得 Web 应用程序即使 HTTP 协议本质上是无状态的，也能跟踪您的操作。每次发出请求时，Web 应用程序都可以从请求中恢复会话值，并执行服务器端查找，以了解会话的归属者及其拥有的权限。如果会话跟踪过程出现问题，则可能允许攻击者劫持会话或冒充会话。

### 1.3 会话过期

由于 HTTP 协议是无状态的，Web 应用程序的用户可能会突然停止使用。例如，您可能关闭了标签页或整个浏览器。由于协议是无状态的，Web 应用程序无法得知此操作已发生。这时，会话过期机制就发挥作用了。您的会话值本身应该具有一个生命周期。如果生命周期已过，而您向 Web 应用程序提交了一个旧的会话值，则该请求应该被拒绝，因为会话应该已经过期。相反，您应该被重定向到登录页面重新进行身份验证，并重新开始会话管理生命周期！

### 1.4 会话终止

然而，在某些情况下，用户可能会强制执行注销操作。如果发生这种情况，Web 应用程序应终止用户的会话。虽然这与会话过期类似，但其独特之处在于，即使会话的有效期仍然有效，也应终止会话本身。此终止过程中的问题可能使攻击者获得对帐户的持久访问权限。

# 二、会话管理方式

## 1. 基于 Cookie 的会话管理

基于 Cookie 的会话管理通常被称为传统的会话管理方式。当 Web 应用程序想要开始跟踪用户访问时，会在响应中发送 Set-Cookie 标头值。浏览器会解析此标头以存储新的 Cookie 值。让我们来看一下 Set-Cookie 标头的示例：

`Set-Cookie: session=12345;`

您的浏览器将创建一个名为 `session` 的 cookie 条目，其值为 `12345` ，该 cookie 仅在接收到它的域中有效。您还可以向此标头添加多个属性。如果您想了解所有属性的更多信息，请参阅[此处](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Set-Cookie)，但以下是一些值得注意的属性：

- **Secure** - 指示浏览器该 Cookie 只能通过已验证的 HTTPS 通道传输。如果存在证书错误或使用 HTTP 协议，则不会传输该 Cookie 值。
- **HTTPOnly** - 向浏览器指示客户端 JavaScript 不得读取 cookie 值。
- **Expire** - 向浏览器指示 cookie 值何时不再有效，应该被删除。
- **SameSite**  \- 向浏览器指示 cookie 是否可以在跨站点请求中传输，以帮助防止 CSRF 攻击。

使用基于 Cookie 的身份验证时，需要记住的关键一点是，浏览器会自行决定何时将特定的 Cookie 值随请求一起发送。浏览器在检查域名和 Cookie 属性后做出决定，并且 Cookie 会自动附加到请求中，无需任何额外的客户端 JavaScript 代码。

## 2. 基于令牌的会话管理

基于令牌的会话管理是一个相对较新的概念。它不使用浏览器的自动 cookie 管理功能，而是依赖客户端代码来完成此过程。身份验证后，Web 应用程序会在请求正文中提供一个令牌。然后，客户端 JavaScript 代码会将此令牌存储在浏览器的 LocalStorage 中。

当发出新请求时，JavaScript 代码必须从存储中加载令牌并将其作为标头附加。最常见的令牌类型之一是 JSON Web Tokens (JWT)，它通过 `Authorization: Bearer` 标头传递。然而，由于我们没有使用浏览器内置的 Cookie 管理功能，所以情况有点像蛮荒西部，什么都可以做。虽然有一些标准，但实际上并没有强制要求任何行为遵守这些标准。

这些方式的优缺点是相互对应的，下面我们逐一说明：

| 对比维度              | Cookie 会话管理                                        | 基于令牌（Token）的会话管理                           |
| --------------------- | ------------------------------------------------------ | ----------------------------------------------------- |
| 请求携带方式          | 浏览器会在每次请求时**自动携带**Cookie                 | 需通过客户端 JS，在每次请求中以**请求头**形式手动提交 |
| 安全防护机制          | 可通过配置 Cookie 属性，强化浏览器对 Cookie 的安全保护 | 无内置自动安全防护，需自行防范令牌泄露风险            |
| 典型客户端漏洞        | 易受 **CSRF（跨站请求伪造）** 攻击                     | 可抵御 CSRF 等传统客户端攻击                          |
| 去中心化 Web 应用适配 | 与特定域名绑定，去中心化场景中**难以安全使用**         | 由 JS 管理，适配去中心化 Web 应用，自身可携带验证信息 |

# 三、会话管理的安全性

会话创建过程中最容易出现漏洞。让我们深入探讨几个常见的漏洞。

**弱会话值**

如今，由于框架的广泛应用，弱会话值的情况已不常见。然而，随着 LLM 和其他 AI 代码辅助解决方案的兴起，这些老式漏洞的出现频率之高可能会让你感到惊讶。如果系统实现了自定义会话创建机制，那么会话值很可能可以被猜测。一个典型的例子是，某种机制只是简单地将用户名进行 Base64 编码作为会话值。如果攻击者能够逆向工程会话创建过程，他们就可以生成或猜测会话值，从而劫持合法用户的帐户。

**可控会话值**

某些令牌（例如 JWT）会提供创建和验证 JWT 有效性所需的所有相关信息。如果未采取安全措施，例如验证令牌签名或确保签名本身的安全性，攻击者就能够生成自己的令牌。我们将在后续章节中更详细地讨论这些类型的攻击。

**会话固定**

还记得那些在你身份验证之前就已创建会话的 Web 应用程序吗？这类 Web 应用程序可能存在一种称为“会话固定”的漏洞。如果你的会话值在身份验证后没有得到充分轮换，那么位置合适的攻击者就可以在你尚未通过身份验证时记录该会话，并等待你通过身份验证后访问你的会话。

**不安全会话传输**

在现代环境中，身份验证服务器和应用服务器通常是分开的。例如，单点登录 ( SSO ) 解决方案就是一个很好的例子。一个应用程序用于对多个其他 Web 应用程序进行身份验证。为了使此过程正常工作，您的会话数据必须通过浏览器从身份验证服务器传输到应用服务器。然而，在此传输过程中，某些问题可能会悄然出现，从而将您的会话信息暴露给攻击者。最常见的问题是存在不安全的重定向，攻击者可以控制身份验证后您将被重定向到的 URL。这可能导致攻击者劫持您的会话。这种情况并非仅限于自定义实现，[Oracle 的 SSO 解决方案](https://krbtgt.pw/oracle-oam-10g-session-hijacking/)就存在一个重大漏洞，使得此类攻击成为可能。

会话跟踪是第二大安全漏洞来源。

**授权绕过**

当对用户是否有权执行其请求的操作缺乏充分的检查时，就会发生授权绕过。本质上，这导致无法正确跟踪用户的会话及其相关权限。此外，值得一提的是两种类型的授权绕过：

**垂直绕过** - 您可以执行只有更高权限用户才能执行的操作

**水平越权** —— 你可以执行一项自身被允许执行的操作，但操作的对象却是你本不应被允许访问的数据集。

在大多数应用中，**垂直越权**很容易防御，因为通常会使用函数装饰器和基于路径的访问控制配置。 但对于**水平越权**，用户执行的本身是其被允许执行的操作，问题在于他们对**他人的数据**执行了该操作。 要修复这类问题，需要编写实际代码来验证以下内容：用户身份（从会话中提取）、用户请求访问的数据，以及该用户是否被允许访问或修改对应数据集。

**日志记录不足**

事件发生时的一个关键问题是信息不足，无法还原攻击经过。虽然基础设施层面会生成大量日志，但应用程序日志对于了解故障原因至关重要。如果特定会话执行的操作以及追溯该会话到用户的能力缺失，调查中就会出现无法弥补的漏洞。此外，确保日志涵盖已接受和已拒绝的操作也至关重要。在会话劫持攻击中，这些操作看起来是合法的。因此，仅仅记录已拒绝的操作不足以还原攻击全貌。

会话过期机制只有一个漏洞，那就是会话过期时间过长。会话可以看作是电影票。每天晚上放映的是同一部电影，但我们不希望有人能用同一张票重复观看。会话也是如此，我们需要确保会话过期时间能够满足特定应用程序的使用场景。例如，银行应用程序的会话生命周期应该比网页邮件客户端的会话生命周期更短。

此外，对于长时间会话（例如网页邮件客户端的会话），会话本身应记录其使用位置。如果此位置发生变化（这可能是会话劫持的迹象），则应终止该会话。

对于会话终止而言，关键问题在于用户注销时服务器端未能正确终止会话。假设攻击者劫持了用户的会话，即使用户意识到了这个问题，但由于无法在服务器端使会话失效，用户也无法清除攻击者的访问权限。然而，对于令牌而言，如果令牌的有效期嵌入在令牌本身，则问题尤为严重。在这种情况下，可以将令牌添加到黑名单中进行验证。一些应用程序更进一步，允许用户查看并终止其所有会话。此外，在成功重置密码后，也建议终止所有会话，以便用户重新获得对其帐户的完全控制权。-
