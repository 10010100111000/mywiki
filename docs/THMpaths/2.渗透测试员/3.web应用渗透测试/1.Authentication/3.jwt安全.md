# 一、简介

应用程序编程接口（简称 API）如今已变得异常流行。这种繁荣的关键原因之一是能够创建单个 API ，然后该 API 可以同时服务于多个不同的接口，例如 Web 应用程序和移动应用程序。这使得相同的服务器端逻辑可以集中化并重复用于所有接口。从安全角度来看，这通常也很有益，因为这意味着我们可以在单个 API 中实现服务器端安全性，无论使用哪个接口，该 API 都能保护我们的服务器。

然而，随着 API 的兴起，新的会话管理方法也应运而生。由于 Cookie 通常与通过浏览器使用的 Web 应用程序相关联，因此基于 Cookie 的 API 身份验证通常效果不佳，因为该解决方案与其他接口无关。这时，基于令牌的会话管理就派上用场了。

## 1、基于令牌的会话管理

基于令牌的会话管理是一个相对较新的概念。它不使用浏览器的自动 cookie 管理功能，而是依赖客户端代码来完成此过程。身份验证后，Web 应用程序会在请求正文中提供一个令牌。然后，客户端 JavaScript 代码会将此令牌存储在浏览器的 LocalStorage 中。

当发出新请求时，JavaScript 代码必须从存储中加载令牌并将其作为标头附加。最常见的令牌类型之一是 JSON Web Tokens (JWT)，它通过 `Authorization: Bearer` 标头传递。然而，由于我们没有使用浏览器内置的 Cookie 管理功能，因此情况有点像蛮荒西部，什么都可以做。尽管存在一些标准，但没有任何规定强制要求遵守这些标准。像 JWT 这样的令牌提供了一种标准化基于令牌的会话管理的方法。

## 2.什么是 JSON Web Token（JWT）

JSON Web Token 通常被用于处理网络服务中的身份验证会话，例如网页应用、应用程序编程接口（API）以及单页应用（SPA）。除此之外，它还有其他常见应用场景，包括：

- 用于获取新会话令牌的刷新令牌
- 密码重置与邮箱验证令牌
- 分享链接与邀请链接
- 其他可对数据对象（如文档）提供（临时）访问权限的机密令牌

以下是一个用于用户身份验证成功后签发 JWT 的 API 所对应的 HTTP 响应包：

![HTTP request of an API that issues JWT after sign in](./image/3.jwt%E5%AE%89%E5%85%A8/1762427358-example_of_a_jwt_token_issuance.png)

当 JWT 的实现存在缺陷时，目标应用程序可能会容易受到多种 JWT 攻击，我们将在本文后面部分详细讨论这些攻击。在深入探讨这些令牌的利用之前，我们先来解析一个标准的 JSON Web Token，以便更好地理解这些问题是如何产生的。

### 3.1 Header 

头部包含元数据，作为 JWT 库的指令，用于验证、解密和签名令牌。元数据可以包括所使用的算法、令牌类型、任何标识符（例如 `kid` ）以及其他相关元数据。

例如：

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

然后，这个 JSON 经过 **base64url** 编码，构成 JWT 的第一部分。

 JWT 中可能存在的所有相关元数据属性，如下所示：

| **属性 (Property)** | **用途 (Purpose)** | **说明 (Remarks)**                                           |
| ------------------- | ------------------ | ------------------------------------------------------------ |
| `alg`               | 签名算法           | 必需字段。若服务端接受`none`算法或存在**算法混淆**（如 RS256→HS256），可被利用。 |
| `typ`               | 令牌类型标识       | 可选字段。通常为`JWT`或`at+jwt`，可能引发**类型混淆攻击**。  |
| `kid`               | 用于验签的密钥标识 | 可选字段。可能存在注入攻击风险（路径遍历、SQL 注入、命令注入）。 |
| `jku`               | JWK 集的 URL       | 可选字段。可能允许攻击者指定可控密钥，可引发 **SSRF（服务器端请求伪造）** 攻击。 |
| `jwk`               | 嵌入的公钥         | 可选字段。可能允许攻击者指定可控密钥。                       |
| `x5u`               | X.509 证书的 URL   | 可选字段。可能允许攻击者指定可控证书链。                     |
| `x5c`               | X.509 证书链       | 可选字段。可能允许攻击者指定可控证书链，可引发 **SSRF（服务器端请求伪造）** 攻击。 |
| `cty`               | 载荷的内容类型     | 可选字段。可能引发 MIME 混淆攻击，进而可能导致 **XXE（XML 外部实体注入）** 漏洞。 |

### 3.2 Payload

载荷（Payload）包含实际的 JWT 声明（claims）以及应用所需的各类数据。最佳实践是**不要包含任何敏感数据**（例如账单信息、邮箱或其他个人可识别信息 PII），除非这些数据已被加密（即使用 JWE 加密）。在漏洞提交报告中需要注意的是：若仅报告未遵循最佳实践，而该问题并未直接导致可利用漏洞，此类提交通常会被视为仅供参考的发现，一般不予收录。

### 3.3 Signature

要生成签名部分，您需要获取编码后的头部、编码后的载荷、一个密钥，以及头部中指定的算法，并以此进行签名。

例如，如果您要使用 HMAC SHA256 算法，签名将通过以下方式生成：

```
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret)
```

签名用于验证消息在传输过程中未被篡改；对于使用私钥签名的令牌，签名还可验证 JWT 的发送方确为其声称的主体。

这种机制使服务器能够验证令牌自签发以来，其中的任何数据均未被篡改：

1. 由于签名直接由令牌的其余部分推导而来，修改头部或载荷中的任意一个字节都会导致签名不匹配。
2. 在不知道服务器的机密签名密钥的前提下，攻击者无法为指定的头部或载荷生成正确的签名。

## 4. jwt jws jwe

JWT 规范本身的定义其实非常有限。它仅规定了一种将信息（“声明”）表示为 JSON 对象并在双方之间传输的格式。在实际应用中，JWT 很少作为独立实体使用。JWT 规范由 **JSON Web 签名（JWS）** 和 **JSON Web 加密（JWE）** 规范进行扩展，这两项规范定义了具体实现 JWT 的实际方式。

换言之，一个 JWT 通常要么是 JWS 令牌，要么是 JWE 令牌。当人们使用 “JWT” 这一术语时，他们几乎总是指 JWS 令牌。JWE 与之非常相似，区别仅在于令牌的实际内容是经过加密的，而非仅仅做了编码处理。

# 二、利用 JWT 漏洞

从设计上看，服务器通常不会存储任何关于其签发的 JWT 的信息。相反，每个令牌都是一个完全自包含的实体。这一特性有诸多优势，但也带来了一个根本性问题：服务器实际上对令牌的原始内容、甚至原始签名都一无所知。因此，如果服务器没有正确验证签名，攻击者就可以随意修改令牌的其他部分，而没有任何机制可以阻止这种行为。

举个例子，假设有一个包含以下声明的 JWT：

```
{
    "username": "carlos",
    "isAdmin": false
}
```

如果服务器基于该 `username` 字段识别用户会话，攻击者修改该字段的值就可能冒充其他已登录用户。同理，如果 `isAdmin` 字段用于访问控制，这就会成为一个简单的权限提升攻击向量。

在最初的几个实验中，你将看到这些漏洞在真实世界应用中的实际案例。

## 2. alg: none 攻击

它之所以会出现，是因为 **后端服务器盲目信任了客户端发送的 Header 信息**。**正常的后端逻辑应该是：**

1. 接收到 Token。
2. **强制使用特定的算法**（比如 HS256）配合**服务器存的密钥**进行校验。
3. 如果 Token 说自己是 `none` 算法，服务器应该拒绝。

**有漏洞的后端逻辑（攻击发生的原因）：**
很多早期的 JWT 库或开发者的代码逻辑是这样的：

1. 接收到 Token，先解析 Header 里的 `alg` 字段。
2. 看到 `alg: "HS256"`，就去找密钥算一遍哈希。
3. 看到 `alg: "none"`，**它认为这个 Token 是合法的、不需要校验签名的**，于是直接跳过签名验证步骤，直接读取并信任 Payload 里的数据。

攻击者通过以下方式绕过认证：

1. 拿到一个合法的 JWT。

2. 修改 Header：将 `"alg": "RS256"` 改为 **`"alg": "none"`**。

3. 修改 Payload：将 `"user": "wiener"` 改为 **`"user": "admin"`**。

4. **修改 Signature（签名）**：将点（`.`）后面的签名部分彻底删除，但要保留那个点（因为 JWT 必须有三个部分）。

   - 例如：`Header.Payload.`（注意最后那个点）。

   > [!NOTE]
   >
   > 根据 JWT 的标准，如果算法是 `none`，那么第三部分（签名）的内容应该是**空的**。

5. 发送给服务器。服务器一看是 `none`，直接把 `admin` 给放进去了。

## 3. 暴力破解密钥

某些签名算法，例如 HS256（HMAC + SHA-256），使用任意的独立字符串作为密钥。就像密码一样，至关重要的是，这个密钥不能轻易被攻击者猜到或暴力破解。否则，攻击者就可以创建任意头部和有效载荷值的 JWT，然后使用该密钥重新签名，从而获得有效的签名。

在实现 JWT 应用时，开发人员有时会犯一些错误，例如忘记更改默认密钥或占位符密钥。他们甚至可能直接复制粘贴从网上找到的代码片段，然后忘记更改示例中提供的硬编码密钥。在这种情况下，攻击者很容易使用包含[已知密钥的字典](https://github.com/wallarm/jwt-secrets/blob/master/jwt.secrets.list)暴力破解服务器密钥。

> [!WARNING]
>
> 这种攻击**仅在令牌使用对称密钥签名时**才有可能实现。采用 RSA 签名的 JWS 和 JWE 通常需要完整密钥对（包括私钥），这使得猜测类攻击几乎无法实施。

我们建议使用 hashcat 来暴力破解密钥。您可以[手动安装 hashcat](https://hashcat.net/wiki/doku.php?id=frequently_asked_questions#how_do_i_install_hashcat) ，但它也预装在 Kali Linux 系统中，可以直接使用。

> [!NOTE]
>
> 如果您使用的是 Kali 的预构建 VirtualBox 镜像而不是裸机安装程序版本，则可能没有足够的内存来运行 hashcat。

您只需要来自目标服务器的有效且已签名的 JWT 和一个[包含已知密钥的字典 ](https://github.com/wallarm/jwt-secrets/blob/master/jwt.secrets.list)。然后，您可以运行以下命令，并将 JWT 和字典作为参数传递给它：

```bash
hashcat -a 0 -m 16500 <jwt> <wordlist>
```

Hashcat 使用字典中的每个密钥对 JWT 的头部和有效载荷进行签名，然后将生成的签名与服务器返回的原始签名进行比较。如果任何签名匹配，Hashcat 将以以下格式输出已识别的密钥以及其他各种详细信息：

```bash
<jwt>:<identified-secret>
```

> [!NOTE]
>
> 如果你运行该命令超过一次，你需要包含 --show 标志来输出结果。

一旦您确定了密钥，就可以使用它为任何您想要的 JWT 标头和有效负载生成有效的签名。有关如何在 Burp Suite 中重新签名修改后的 JWT 的详细信息，请参阅 [“编辑 JWT”](https://portswigger.net/burp/documentation/desktop/testing-workflow/vulnerabilities/session-management/jwts#editing-jwts) 。



## 4.JWT 头部参数注入

根据 JWS 规范，只有头部参数 `alg` 是必填项。但在实际应用中，JWT 头部（也称为 JOSE 头部）通常还包含其他多个参数。以下几个参数对攻击者而言尤为值得关注。

- **jwk（JSON Web Key）**：提供一个内嵌的 JSON 对象，用于表示密钥。
- **jku（JSON Web Key Set URL）**：提供一个 URL，服务器可通过该地址获取包含正确密钥的密钥集。
- **kid（Key ID）**：提供一个 ID，当存在多个可选密钥时，服务器可通过该 ID 识别出正确的密钥。根据密钥格式，该参数可能会有对应的 `kid` 参数。

如你所见，这些可由用户控制的参数都会告知接收端服务器在验证签名时应使用哪一个密钥。在本节中，你将学习如何利用这些参数注入经过篡改的 JWT，并使用你自己任意生成的密钥进行签名，而非服务器的机密密钥。

---

### 4.1 通过 jwk 参数注入自签名 JWT
JSON Web 签名（JWS）规范定义了一个可选的头部参数 `jwk`，服务器可利用该参数以 JWK 格式将自身公钥直接嵌入令牌本身。

> [!NOTE]
>
>  JWK（JSON Web Key）是一种以 JSON 对象形式表示密钥的标准化格式。

您可以在以下 JWT 标头中看到一个示例：

```json
{
    "kid": "ed2Nf8sb-sD6ng0-scs5390g-fFD8sfxG",
    "typ": "JWT",
    "alg": "RS256",
    "jwk": {
        "kty": "RSA",
        "e": "AQAB",
        "kid": "ed2Nf8sb-sD6ng0-scs5390g-fFD8sfxG",
        "n": "yy1wpYmffgXBxhAUJzHHocCuJolwDqql75ZWuCQ_cb33K2vh9m"
    }
}
```

理想情况下，服务器应仅使用经过严格限制的公钥白名单来验证 JWT 签名。但配置不当的服务器有时会直接信任并使用嵌入在 `jwk` 参数中的任意密钥。

正常流程是：服务器用**私钥 A** 签名，然后用**公钥 A** 验证。

注入攻击的逻辑是：

1. **黑客自制一对密钥**：私钥 B 和 公钥 B。
2. **修改 Token**：黑客把 Token 里的身份改成了 `admin`。
3. **自签名**：黑客用自己的**私钥 B** 对 Token 进行签名。
4. **注入公钥**：黑客把自己的**公钥 B** 塞进 JWT 的 `jwk` 头部字段里。
5. **服务器受骗**：如果服务器的代码写成“从 Token 头部提取公钥来验证签名”，它就会用**公钥 B** 去校验。因为签名确实是用对应的私钥 B 签的，校验会通过。 

**结果：** 服务器觉得自己验证了签名，但其实它只是验证了“这个 Token 确实是由提供公钥的那个人签名的”，而那个人恰恰是黑客。

你可以利用这一行为，使用自己的 RSA 私钥对篡改后的 JWT 进行签名，然后将对应的公钥嵌入到 JWT 头部的 `jwk` 字段中。

你也可以通过手动添加 `jwk` 头部来实施此攻击。不过，你可能还需要更新 JWT 头部中的 `kid` 参数，使其与嵌入密钥的 `kid` 保持一致。该扩展内置的攻击功能会自动为你完成这一步骤。

### 4.2 通过 jku 参数注入自签名 JWT

有些服务器不直接使用 `jwk` 请求头参数嵌入公钥，而是允许您使用 `jku` （JWK 集 URL）请求头参数来引用包含密钥的 JWK 集。验证签名时，服务器会从该 URL 获取相关密钥。

例如在大公司（比如 Google 或公司的微服务架构）里，身份验证服务器（中心站）和各种业务服务器（分站）是分开的：
中心站经常需要更换密钥（为了安全，比如每个月换一次）。如果每次换密钥都要通知成百上千个分站手动更新公钥文件，运维会疯掉。

### 4.3 通过 kid 参数注入自签名 JWT

服务器可能会使用多个加密密钥来对不同类型的数据进行签名，而不仅仅是 JWT。出于这个原因，JWT 的头部中可能会包含一个 `kid`（Key ID，密钥标识）参数，用于帮助服务器在验证签名时确定应该使用哪个密钥。

验证密钥通常以 JWK 集（JWK Set）的形式存储。在这种情况下，服务器只需查找与令牌中 `kid` 相同的 JWK 即可。但是，JWS 规范并没有为这种标识定义具体结构 —— 它只是开发者自行指定的任意字符串。例如，开发者可能会使用 `kid` 参数指向数据库中的某条记录，甚至是某个文件名。

如果该参数还存在**目录遍历漏洞**，攻击者就有可能强制服务器将其文件系统中的任意文件当作验证密钥来使用。

```json
{
    "kid": "../../path/to/file",
    "typ": "JWT",
    "alg": "HS256",
    "k": "asGsADas3421-dfh9DGN-AFDFDbasfd8-anfjkvc"
}
```

如果服务器也支持使用[对称算法](https://portswigger.net/web-security/jwt/algorithm-confusion#symmetric-vs-asymmetric-algorithms)签名的 JWT，这种情况尤其危险。攻击者可以将 `kid` 参数指向一个可预测的静态文件，然后使用与该文件内容匹配的密钥对 JWT 进行签名。

理论上，任何文件都可以这样做，但最简单的方法之一是使用 `/dev/null` ，它存在于大多数 Linux 系统中。由于这是一个空文件，读取它会返回一个空字符串。因此，使用空字符串对令牌进行签名将得到有效的签名。

> [!WARNING]
>
> 这种方法只适合 对称加密的jwt

### 4.4 通过kid参数执行sql注入攻击

与4.3节 的情况类似，如果目标在从数据库加载签名密钥时考虑了 `kid` 属性，我们实际上也可以测试 SQL 注入，甚至是 [NoSQL 注入 ](https://www.intigriti.com/researchers/blog/hacking-tools/exploiting-nosql-injection-nosqli-vulnerabilities)。

请注意以下存在漏洞的代码片段：

```js
// 引入 Express 框架，用于搭建 Web 服务
const express = require('express');
// 引入 jsonwebtoken 库，用于 JWT 的编解码、签名与验签
const jwt = require('jsonwebtoken');
// 引入 sqlite3 库，用于操作 SQLite 数据库，verbose() 开启详细日志模式
const sqlite3 = require('sqlite3').verbose();
// 初始化 Express 应用实例
const app = express();

// 连接数据库（此处省略具体连接参数，如数据库文件路径、权限等）
const db = new sqlite3.Database(...);

/**
 * 从数据库中根据密钥 ID 获取对应验签密钥
 * @param {string} keyId - 密钥 ID（直接来自 JWT 头部的 kid 参数，用户可控）
 * @param {function} callback - 回调函数，用于异步返回查询结果或错误
 */
function getKeyFromDatabase(keyId, callback) {
  // 【⚠️ 高危漏洞：SQL 注入】直接将用户可控的 keyId 拼接到 SQL 语句中
  // 攻击者可通过构造恶意 kid 参数，注入任意 SQL 语句
  const query = "SELECT key_data FROM keys WHERE id = '" + keyId + "'";

  // 执行 SQL 查询
  db.get(query, (err, row) => {
    if (err) {
      // 数据库错误处理：打印日志并返回错误
      console.error('Database error:', err.message);
      return callback(err, null);
    }

    if (!row) {
      // 未查询到对应密钥，返回「密钥不存在」错误
      return callback(new Error('Key not found'), null);
    }

    // 查询成功，将密钥数据返回给调用方
    callback(null, row.key_data);
  });
}

// JWT 身份验证中间件：拦截所有请求，统一验证 JWT 有效性
app.use((req, res, next) => {
  // 从请求头中提取 Authorization 字段
  const authHeader = req.headers.authorization;

  // 校验 Authorization 头格式：必须存在且以 "Bearer " 开头
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Authentication required' });
  }

  // 分割 Authorization 头，提取出真实的 JWT 令牌
  const token = authHeader.split(' ')[1];

  try {
    // 【注意】仅解码 JWT，不做签名验证，目的是获取 JWT 头部的 kid 参数
    // { complete: true } 表示同时返回 header、payload、signature 三部分
    const decodedToken = jwt.decode(token, { complete: true });

    // 校验解码后的 JWT 格式合法性，必须包含 kid 参数
    if (!decodedToken || !decodedToken.header || !decodedToken.header.kid) {
      return res.status(400).json({ error: 'Invalid token format' });
    }

    // 调用 getKeyFromDatabase，根据 JWT 头部的 kid 从数据库获取验签用公钥
    getKeyFromDatabase(decodedToken.header.kid, (err, publicKey) => {
      if (err) {
        // 密钥获取失败，返回 400 错误
        return res.status(400).json({ error: 'Key retrieval error' });
      }

      try {
        // 【核心验签步骤】使用从数据库获取的公钥验证 JWT 签名的有效性
        // 验证通过后，将 JWT 载荷中的用户信息挂载到 req.user，供后续路由使用
        req.user = jwt.verify(token, publicKey);
        // 验签通过，放行请求，进入后续业务逻辑
        next();
      } catch (verifyError) {
        // 签名验证失败（篡改、密钥错误等），返回 401 未授权
        res.status(401).json({ error: 'Invalid token' });
      }
    });
  } catch (error) {
    // 其他身份验证异常（如解码失败），统一返回 401
    res.status(401).json({ error: 'Authentication failed' });
  }
});
```

在**第21行** ，可以看到 `kid` 属性在执行之前被连接到一个未经预处理的 SQL 语句中。在这种情况下，只需跳出上下文并注入一个可预测的有效负载字符串，即可将初始签名密钥替换为我们控制的字符串值。

```json
{
    "alg": "RS256",
    "kid": "example-key' OR UNION SELECT 'intigriti'; --"
}
```

复制我们之前遵循的概念验证步骤，将使我们能够伪造带有恶意声明的 JSON Web Token，从数据库中提取敏感数据，并且在极少数情况下，甚至可以在易受攻击的服务器上执行代码。

### 4.5 其他有趣的 JWT 标头参数

**cty（Content Type，内容类型）**

有时用于声明 JWT 载荷内容的媒体类型。该字段通常不会出现在头部中，但底层解析库往往仍支持它。如果你已找到绕过签名验证的方法，可以尝试注入 `cty` 头部，将内容类型修改为 `text/xml` 或 `application/x-java-serialized-object`，这有可能开辟出新的 **[XXE 攻击](https://portswigger.net/web-security/xxe)**与**[反序列化攻击](https://portswigger.net/web-security/deserialization)**向量。

**x5c（X.509 Certificate Chain，X.509 证书链）**

有时用于传递对 JWT 进行数字签名的密钥所对应的 X.509 公钥证书或证书链。该头部参数可被用于注入自签名证书，与前面讨论的 `jwk` 头部注入攻击原理类似。由于 X.509 格式及其扩展较为复杂，对这类证书的解析过程也可能引入漏洞。此类攻击的详细内容不在本资料范围内，如需更多细节可查阅 [**CVE-2017-2800**](https://talosintelligence.com/vulnerability_reports/TALOS-2017-0293) 和 **[CVE-2018-2633](https://mbechler.github.io/2018/01/20/Java-CVE-2018-2633)**。

## 5. JWT 算法混淆攻击

即使服务器使用了你无法暴力破解的强大密钥，你仍然可以通过使用开发者未预料到的算法对令牌进行签名来伪造有效的 JWT。这被称为算法混淆攻击。

算法混淆攻击（也称为密钥混淆攻击）是指攻击者能够强制服务器使用与网站开发者预期不同的算法来验证 JSON Web Token ( [JWT](https://portswigger.net/web-security/jwt) ) 的签名。如果处理不当，攻击者可能无需知道服务器的秘密签名密钥，即可伪造包含任意值的有效 JWT。

### 5.1 对称算法与非对称算法

JWT 可以使用多种不同的算法进行签名。其中一些算法，例如 HS256（HMAC + SHA-256），使用“对称”密钥。这意味着服务器使用同一个密钥来签名和验证令牌。显然，这个密钥需要像密码一样保密。

![Signing and verifying JWTs using a symmetric algorithm](./image/3.jwt%E5%AE%89%E5%85%A8/jwt-symmetric-signing-algorithm.jpg)

其他算法，例如 RS256（RSA + SHA-256），使用“非对称”密钥对。该密钥对由一个私钥和一个与其数学相关的公钥组成。服务器使用该私钥对令牌进行签名，而公钥则可用于验证签名。

![Signing and verifying JWTs using an asymmetric algorithm](./image/3.jwt%E5%AE%89%E5%85%A8/jwt-asymmetric-signing-algorithm.jpg)

顾名思义，私钥必须保密，但公钥通常会被共享，以便任何人都可以验证服务器颁发的令牌的签名。

### 5.2 算法混淆漏洞是如何产生的？

算法混淆漏洞通常是由于 JWT 库的实现存在缺陷而导致的。尽管实际的验证过程会因所使用的算法而异，但许多库都提供了一种与算法无关的单一签名验证方法。这些方法依赖于令牌头部中的 `alg` 参数来确定应执行的验证类型。

以下伪代码展示了 JWT 库中通用 `verify()` 方法声明的简化示例：

```js
function verify(token, secretOrPublicKey){
    algorithm = token.getAlgHeader();
    if(algorithm == "RS256"){
        // Use the provided key as an RSA public key
    } else if (algorithm == "HS256"){
        // Use the provided key as an HMAC secret key
    }
}
```

网站开发人员假设验证方法只需处理 RS256 等“非加密加密算法”签名的 JWT 时，问题就随之而来了。基于这种错误假设，他们可能会在代码中首先提交一个固定的**场景**，如下所示：

当后续使用该方法的网站开发者**错误地认为**该方法只会处理使用 ** 非对称算法（如 RS256）** 签名的 JWT 时，问题便会出现。基于这个有缺陷的假设，他们可能会始终将一个固定的公钥传入该方法，示例如下：

```javascript
publicKey = <public-key-of-server>;
token = request.getCookie("session");
verify(token, publicKey);
```

在这种情况下，如果服务器收到一个使用**对称算法（如 HS256)**签名的令牌，该库的通用 `verify()` 方法会将**公钥当作 HMAC 密钥**来使用。这意味着攻击者可以使用 HS256 算法和该公钥对令牌进行签名，而服务器会使用同一个公钥来验证签名。

> [!NOTE]
>
> 你用于签名令牌的**公钥必须与服务器上存储的公钥完全一致**。这包括使用相同的格式（如 X.509 PEM 格式），并保留所有不可打印字符（如换行符）。在实际操作中，你可能需要尝试不同的格式才能使攻击成功。
>
> 

### 5.3 执行算法混淆攻击

算法混淆攻击通常包括以下几个主要步骤：

1. 获取服务器的公钥
2. 将公钥转换为合适的格式
3. 创建一个恶意 JWT ，其有效载荷已修改，并且 `alg` 标头设置为 `HS256` 。
4. 使用公钥作为密钥， 通过 HS256 对令牌进行签名 。

#### 步骤 1 - 获取服务器的公钥

服务器有时会通过映射到 `/jwks.json` 或 `/.well-known/jwks.json` 等的标准端点，以 JSON Web Key (JWK) 对象的形式公开其公钥。这些公钥可能存储在一个名为 `keys` 的 JWK 数组中。这被称为 JWK 集。

```
{
    "keys": [
        {
            "kty": "RSA",
```

#### 步骤 2 - 将公钥转换为合适的格式

尽管服务器可能会以 JWK 格式公开其公钥，但在验证令牌签名时，它会使用本地文件系统或数据库中存储的密钥副本。该副本的存储格式可能有所不同。

要使攻击成功，你用于对 JWT 进行签名的密钥必须与服务器本地的密钥副本**完全一致**。除了格式相同外，每一个字节都必须匹配，包括任何不可打印字符。

在本示例中，我们假设需要使用 **X.509 PEM 格式**的密钥。你可以在 Burp Suite 中通过 JWT Editor 扩展将 JWK 转换为 PEM，具体操作如下：

1. 加载扩展后，在 Burp 主标签栏中进入 **JWT Editor Keys** 标签页。
2. 点击 **New RSA Key**。在弹出的对话框中，粘贴你之前获取的 JWK。
3. 选择 **PEM** 单选按钮，并复制生成的 PEM 格式密钥。
4. 切换到 **Decoder** 标签页，对该 PEM 密钥进行 Base64 编码。
5. 返回 **JWT Editor Keys** 标签页，点击 **New Symmetric Key**。
6. 在对话框中，点击 **Generate** 以生成一个 JWK 格式的新密钥。
7. 将生成的 `k` 参数值替换为你刚刚复制的、经过 Base64 编码的 PEM 密钥。
8. 保存该密钥。

#### 步骤 3 - 修改您的 JWT

一旦您获得了格式合适的公钥，就可以随意[修改 JWT](https://portswigger.net/burp/documentation/desktop/testing-workflow/vulnerabilities/session-management/jwts#editing-jwts) 。只需确保 `alg` 标头设置为 `HS256` 即可。

#### 步骤 4 - 使用公钥对 JWT 进行签名

使用 HS256 算法和 RSA 公钥作为密钥对[令牌进行签名 ](https://portswigger.net/burp/documentation/desktop/testing-workflow/vulnerabilities/session-management/jwts#adding-a-jwt-signing-key)。
