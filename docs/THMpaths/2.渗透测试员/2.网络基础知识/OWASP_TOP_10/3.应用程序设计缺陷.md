# AS02：安全配置错误

安全配置错误是指系统、服务器或应用程序部署时使用了不安全的默认设置、不完整的配置或暴露了某些服务。这些并非代码漏洞，而是环境、软件或网络配置方面的错误。它们会为攻击者提供轻易的入口点。

即使是微小的配置错误也可能泄露敏感数据、导致权限提升，或让攻击者有机可乘，入侵系统。现代应用程序依赖于复杂的技术栈、云服务和第三方 API。一个暴露的管理面板、一个开放的存储桶或错误的权限配置都可能导致整个系统安全受到威胁。

案例：

- 默认凭据或弱密码未作更改
- 不必要的服务或暴露在互联网上的终端
- 云存储或权限配置错误（S3, Azure Blob, GCP buckets)）
- 无限制的 API 访问或缺少身份验证/授权
- 详细的错误信息会显示堆栈跟踪或系统详情。
- 存在已知漏洞的过时软件、框架或容器
- 缺乏适当访问控制的暴露AI/ML端点



# AS03：软件供应链故障

软件供应链故障发生在应用程序依赖于已被破坏、过时或未经充分验证的组件、库、服务或模型时。这些弱点并非源于您的代码，而是存在于您所依赖的软件和工具中。攻击者会利用这些薄弱环节注入恶意代码、绕过安全防护或窃取敏感数据。

现代应用程序由众多第三方软件包、API 和 AI 模型构建而成。任何一个被攻破的依赖项都可能导致整个系统瘫痪，攻击者无需修改任何代码即可获取访问权限。供应链攻击可以自动化且分布式进行，因此难以检测且破坏性极大。

案例：

- 使用未经验证或未维护的库和依赖项
- 自动安装更新而无需验证
- 过度依赖第三方AI模型而不进行监控或审计
- 组件的许可证或来源跟踪不良
- 部署后对依赖项中的漏洞缺乏监控

# AS04：加密故障

加密故障是指加密使用不当或根本没有使用加密。这包括算法强度不足、密钥硬编码、密钥管理不善或敏感数据未加密。这些缺陷使得攻击者能够访问本应保密的信息。

攻击者可以通过中间人攻击、对弱密钥进行暴力破解攻击，或者仅仅是发现从未得到妥善保护的秘密来利用这些漏洞。

案例：

- 使用已弃用或存在缺陷的算法，例如MD5, SHA-1, or ECB mode
- 代码或配置中硬编码的密钥
- 密钥轮换或管理措施不当
- 自签名或无效的 TLS 证书
- 在 AI/ML 系统中，未对模型参数或敏感输入进行妥善的机密信息处理

# AS06：不安全的设计

不安全的设计是指系统从一开始就存在逻辑或架构缺陷。这些缺陷源于未进行威胁建模、缺乏设计需求或评审，或是人为错误。

此外，随着人工智能助手的引入，人工智能系统加剧了设计上的不安全因素。开发者常常假定模型是安全、正确或可预测的，或者他们编写的代码完美无瑕。当人工智能系统能够不受限制地生成查询、编写代码或对用户进行分类时，风险就已根植于设计之中，从而导致糟糕的架构模式。

你无法修补一个不安全的设计。它已经根植于工作流程、逻辑和信任边界之中。修复它意味着重新思考系统（现在也包括人工智能）的决策方式。

- 薄弱的业务逻辑控制，例如恢复或审批流程。
- 对用户或模型行为的错误假设
- 具有未经检查的权限或访问权限的组件
- LLM 和自动化代理缺少防护措施
- 测试或调试生产环境中遗留的绕过机制
- 没有持续的滥用案例审查或人工智能威胁建模

人工智能引入了新的设计缺陷。例如，当用户输入与系统提示混合时会发生提示注入，允许攻击者劫持上下文或提取隐藏数据。对模型输出的盲目信任会创建脆弱的系统，这些系统在没有验证或监督的情况下根据人工智能的决策行事，这就是为什么人工审查仍然是必要的。当涉及到来自未经验证来源或在不安全数据上进行微调的污染模型时，它们可能会嵌入隐藏行为或后门，从而从内部破坏系统。
