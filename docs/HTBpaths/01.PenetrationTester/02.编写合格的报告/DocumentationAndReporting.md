# 一、文档与报告简介
出色的**文档与报告撰写能力**在信息技术或信息安全的任何领域都极具价值，掌握这项技能能让我们在职业生涯中快速晋升。拥有高超的技术水平固然重要且必不可少，但如果没有软技能支撑我们的技术攻防能力，无论我们是编写内部策略与流程、技术文档、渗透测试报告，还是其他类型的客户交付物，工作效果都会大打折扣。

笔记记录与报告编写**没有万能模板**，但想要做好，就必须遵循一些基本原则。本模块将探讨多种风格，包括笔记工具、在测试虚拟机中整理证据、为非技术受众撰写执行摘要，以及记录攻击链与技术漏洞。我们尝试提供一套通用方法，任何人都能借此取得良好效果，同时融入我们在面向客户的技术咨询与管理岗位上累计约25年经验总结出的技巧、窍门与最佳实践。

作为测试人员，我们必须牢记一个关键点：**一次渗透测试只是目标网络在某一时刻安全状态的快照**。我们应在报告中加入概述部分，说明执行的工作类型、执行人员、测试所用源IP地址，以及任何特殊情况（例如通过VPN远程测试，或从客户网络内的主机进行测试）。该概述还应注明，测试在特定时间段内执行，报告无法涵盖测试期间范围内系统发生的任何变更及由此产生的新漏洞。我们可以这样表述：
“所有测试活动均在2022年1月7日至2022年1月19日期间执行。”

我们还可以添加免责声明，例如：
“本报告仅代表上述测试期间的安全状态快照，Acme咨询公司无法保证测试时间窗口之外客户信息资产的安全状况。”

## 1、实践中的文档与报告
你可能会想：“这个模块会很无聊”，或者“这主题怎么能开一整门课？”。虽然文档与报告不是最令人兴奋的内容，也远不如攻陷靶机、在实验环境或真实网络中拿到域管权限那样有成就感，但对任何从事咨询工作的人来说，这些都是核心技能。下面几个我们职业生涯中的真实案例，展示了规范、详尽的文档与详细报告，是如何帮我们摆脱可能升级为严重麻烦的困境。

### 场景1——虚拟机崩溃事件
在一次持续很久的项目（近一个月的外网渗透测试）中，我某天登录时，虚拟机无法正常加载。我花了很长时间尝试修复文件系统，但数据已经完全丢失。幸运的是，我一直在主力工作站上用本地版OneNote认真、详细地记录项目笔记。当时我的团队使用共享存储管理所有项目，并有自动脚本在测试结束后同步数据，用于质量审核与归档。在一位资深同事的提醒下，我每天下班前都会把测试证据备份到共享存储。因此，当我从模板重新创建新虚拟机后，只需同步项目数据，就能毫无中断地继续测试。如果我没有遵循规范流程，后果将是灾难性的：丢失数周工作成果，甚至可能导致公司流失客户、我本人丢掉工作。

### 场景2——死亡之ping
从最初的范围沟通会议开始，我就知道这次内网渗透测试会很棘手。客户内部IT团队中有一名成员态度非常敌对，对一切都提出质疑，怀疑我们团队的技术能力，并且对一组关键服务器保护极强（而这些服务器的IP并不在测试范围内）。在测试的枚举阶段，我收到客户的邮件和电话，要求立即停止所有测试，称我们的扫描导致多台对扫描敏感的关键服务器宕机。我出示了所有日志数据、范围文件与带时间戳的原始扫描数据。受影响主机的IP确实出现在我的扫描中，但也在客户确认过的测试范围内。在这件事上，我没有任何过错：我只是在测试已确认的范围，并未执行任何鲁莽的测试行为。不过我从中吸取了教训，并加入到工作流程中：此后，我一定会向客户索要**明确排除在扫描及其他测试之外**的具体IP地址与主机名清单。在这个案例中，我凭借完善的文档证明了自己的操作，并有客户书面确认的测试IP范围。同时，我也找到了优化流程、提升咨询专业度的机会。

### 场景3——慢如糖浆
最后一个案例中，我在客户总部现场执行内网渗透测试，坐在IT员工办公区。我身后是一名态度十分敌对的网络管理员，从启动会议开始就质疑我们的能力与工具，声称其他公司的渗透测试人员因经验不足、操作鲁莽，曾导致网络严重卡顿。测试开始不到20分钟，这名网管就向全体邮件列表发了邮件，然后走到我工位，说我们的扫描让网络陷入瘫痪，并已封禁了我们的源IP。随后我们一起核对了扫描输出，证明我们遵循了最佳实践，我们的操作不可能导致网络卡顿。与此同时，另一名管理员发现，所有网络设备都开启了调试模式，再加上正常的Nmap扫描，足以让设备过载并引发卡顿。关闭调试模式后，测试顺利进行。在这件事里，我们的文档证明了自身操作，迫使客户进一步排查问题。如果我们没有文档（或文档质量很差），责任很可能会被推到我们身上，严重影响客户关系与我们的声誉。

这些案例都体现了完善文档的重要性。我们必须能够为自己的操作辩护，并在需要时向客户出示证据，协助排查问题。渗透测试期间出现任何网络问题，无论是否由我们的操作导致，都经常会被归咎于测试人员。我们需要处于有利位置，保护自己并帮助客户。此外，我们绝不希望因为笔记与整理工作不到位，在丢失证据后匆忙重做测试，或向客户申请更多时间。

本模块中有很多练习所学技能的机会，但完成模块并不强制要求做完这些练习。为了从本模块获得最大价值，我们建议（当然）详细记笔记，并运用这些技巧，找出自己工作流程中可能存在的漏洞或低效环节。一旦发现问题，我们希望你能找到提升效率、减轻报告负担的方法——说实话，没人喜欢写报告，我们只是把它当作必要之恶来接受。学完本模块后，建议在真实项目中练习这些技巧；如果你尚未从事咨询工作，可在训练靶机与实验环境中练习文档与报告撰写。

本模块附带一台预配置的攻击主机与一个小型实验环境，可用于模拟内网渗透测试。充分利用它，反复练习，直到你能熟练识别所有漏洞，并形成适合自己工作流的文档风格。

在本模块中，你可能会看到我们使用“内网测试”“外网测试”等简称，分别对应**内网渗透测试**与**外网渗透测试**。

# 笔记记录与组织
在任何安全评估过程中，**详尽的笔记记录**都至关重要。我们的笔记、工具输出和日志是报告初稿的原始素材，而报告通常是客户唯一能看到的评估成果。即便笔记主要供自己使用，也必须保持条理化，并建立可复用的流程，以节省时间、简化报告编写。详细的笔记在遇到网络问题、客户质询（例如“你是否在某日扫描过某主机？”）时必不可少，因此笔记内容越详尽越好。每个人都有自己习惯的风格，应搭配偏好的工具与组织结构，以达到最佳效果。

本模块会介绍**必备的基础记录要素**（结合专业经验总结），无论在评估、学习大型模块、练习HTB靶机还是备考时记录，都能节省报告撰写时间，也可作为后续参考指南。如果你在大型团队，同事可能需要替你参加客户会议，清晰一致的笔记能确保队友准确、自信地说明执行/未执行的操作。

## 笔记记录示例结构
笔记没有统一通用的结构，每个项目、测试人员都不同。以下结构经过实践验证非常实用，你可根据个人工作流、项目类型、项目场景灵活调整。例如，部分类别不适用于应用专项评估，也可按需增加未列出的类别。

### 核心分类
1. **攻击路径（Attack Path）**
   外网测试获取立足点、内网测试攻陷主机/AD域时，记录完整攻击路径大纲。尽可能用截图、命令输出整理，方便后续直接粘贴到报告，只需调整格式即可。

2. **凭证（Credentials）**
   集中记录所有攻陷获取的凭证与密钥信息。

3. **漏洞发现（Findings）**
   为每个漏洞创建子文件夹，写入描述文本，与证据（截图、命令输出）一同保存；同时在笔记工具中预留板块，汇总漏洞信息，便于报告整理。

4. **漏洞扫描研究（Vulnerability Scan Research）**
   记录漏洞扫描中已研究、已尝试的内容，避免重复工作。

5. **服务枚举研究（Service Enumeration Research）**
   记录已排查的服务、失败的利用尝试、有价值的漏洞/配置错误等。

6. **Web应用研究（Web Application Research）**
   记录通过子域名爆破等方式发现的Web应用；外网做全面子域名枚举、内网扫描常见Web端口，使用Aquatone/EyeWitness截图，记录重点应用、尝试过的默认/通用账号密码。

7. **AD枚举研究（AD Enumeration Research）**
   分步记录已执行的活动目录枚举操作，标注后续需要深入排查的重点区域。

8. **开源情报（OSINT）**
   记录通过OSINT收集的关键信息（如适用）。

9. **管理信息（Administrative Information）**
   集中存储项目干系人联系方式（项目经理、客户对接人POC）、授权规则（RoE）中的特殊目标/标识、高频引用信息；也可作为动态待办清单，记录想到但没时间执行的测试思路。

10. **范围信息（Scoping Information）**
    存储测试范围内IP/CIDR段、Web应用URL、客户提供的Web/VPN/AD凭证；记录所有与评估范围相关的内容，避免反复查阅范围文件、防止越界测试。

11. **操作日志（Activity Log）**
    宏观记录评估期间执行的所有操作，用于后续事件关联。

12. **攻击载荷日志（Payload Log）**
    记录客户环境中使用的Payload、上传文件哈希、上传路径，是关键必备记录。

## 笔记工具
笔记工具选择高度个性化，常用工具如下：
- 本地优先：**Obsidian**、CherryTree、OneNote、Notepad++、Visual Studio Code、Sublime Text
- 云端/协作：Notion、Evernote、GitBook、Outline、Cryptpad、Standard Notes

### 关键选择要点
选择前必须区分**本地存储**和**云端存储**：
- 课程、CTF、实验环境可使用云端工具
- 正式客户项目必须谨慎，严格遵守公司数据存储政策与合同要求，优先使用本地工具

**推荐组合**：
- **Obsidian**：本地存储首选，支持Markdown，高度自定义
- **Outline**：云端/自托管方案，支持Markdown导出导入

两种工具均可导出为Markdown格式，兼容绝大多数笔记软件。

## 日志记录
**必须记录所有扫描与攻击尝试，尽可能保留原始工具输出**，这会大幅降低报告编写难度。即便笔记清晰详尽，也可能遗漏细节，日志可作为补充证据、回应客户质询的兜底依据。

### 漏洞利用尝试记录
**Tmux日志**是终端记录的最佳选择，可保存面板中所有输入输出内容。
必须记录所有利用尝试，用于客户后续事件关联；尤其在漏洞极少时，能证明执行过完整测试。无法提供这类记录会显得非常不专业。

同时建议记录**尝试过但失败**的操作，在无明显漏洞时，可在报告中说明执行的测试类型，证明客户环境已具备相应防护能力。我们可以按如下方式在系统上设置 Tmux 日志记录：
