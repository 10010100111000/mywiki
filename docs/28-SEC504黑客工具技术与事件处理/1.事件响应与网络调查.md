1、示例事件

为了更好地理解复杂且棘手的事件究竟是怎样的，我们来看一个示例事件——即一系列对环境造成或预示着危害的事件集合。该场景本身是虚构的，包括受害者和攻击者在内。不过，它融合了多个真实案例，其中的每个要素在各类事件中都多次出现过。

在这种情况下，受害者是阿古斯公司（Argous Corporation），这是一家为政府、军事和私营企业等特定领域生产计算机芯片的工程公司。

这些攻击者是一个被统称为“绿企鹅”的组织。该组织成员虽未受雇于任何政府，却实际上是国家支持活动的“顾问”。也就是说，他们代表一个国家开展行动，尽管并未直接受雇于该国。这有助于避免其行动被追溯到资助方。

![image-20260501112121689](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501112121689.png)

Argous 公司的企业网络有多台服务器，包括一台域控制器、一台文件服务器和一台数据库服务器。该网络还有 100 台用户工作站以及若干其他网络设备（未在上方幻灯片中展示）。其大部分网络都部署在防火墙之后。

除了工程业务外，Argous 还开发了一款定制化的客户关系管理（CRM）网络应用程序，销售员工无论是出差途中还是在办公室都可使用该应用。

周二上午9点，信息安全团队接到一位系统管理员的电话，对方告知公司客户关系管理（CRM）服务器出现异常流量。具体来说，他们发现TCP 4444端口处于监听状态，而此前并未留意到该端口的存在。

此后不久，信息安全团队的一名成员做出响应并检查了客户关系管理系统。正如系统管理员所描述的那样，发现有一个名为 Office_Remoter.exe 的进程在监听 TCP 4444 端口。还看到有与 4444 端口建立的活跃连接，并立即终止了该进程。

信息安全团队还观察到一个名为 Office_Techneter.exe 的进程，该进程会通过 TCP 443 端口定期向微软的 social.technet.microsoft.com 服务器发起出站连接。此时，他们也会终止 Office_Techneter.exe 进程。在未发现其他异常网络流量后，他们通知系统管理员客户关系管理（CRM）系统已恢复可用。

![image-20260501112345798](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501112345798.png)

现在，我们从绿企鹅攻击者的角度来审视这起事件。你会发现，从攻击者的视角来看，事件的经过与事件响应人员的视角大相径庭。这并非事件响应人员的过错；原因在于，事件通常是在大量恶意活动发生很久之后才被发现的。

绿企鹅开展的第一项行动是情报搜集。该团队成员扫描了阿格斯公司公开可访问的地址空间，发现了防火墙、CRM应用程序以及互联网隔离网络上的其他服务器。此次行动大约在事件被发现前三个月展开。

在这次事件中，攻击者并未采取太多措施来隐藏行踪。尽管他们使用了 VPN 来掩盖自己的 IP 地址，但他们的扫描操作还是在防火墙上触发了数条警报。此外，这些 VPN 服务器的 IP 地址已被认定为恶意地址。

**错误一**：网络管理员忽略了防火墙警报，错误地认为这些警报是随机扫描。他们的理由是，既然数据包未能进入内部网络，防火墙就已经完成了任务。

**错误二**：未使用威胁情报，因此这些扫描看起来是随机的。



![image-20260501112546283](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501112546283.png)

绿企鹅公司采取的下一步是扫描定制的客户关系管理（CRM）网络应用程序。他们发现了两个漏洞：服务器端请求伪造（SSRF）和命令注入。

SSRF 漏洞让绿企鹅攻击者得以读取客户关系管理（CRM）应用的源代码，进而发现了一处命令注入漏洞。该命令注入漏洞赋予绿企鹅执行任意命令的权限。这两处漏洞均无需登录网页应用即可被利用。

**错误3**：Argous 公司聘请了一家渗透测试公司，该公司在今年早些时候发现了这些漏洞。然而，Argous 的管理层并未修复这些漏洞，而是决定将软件开发团队的工作重心放在更有可能带来收入的项目上。他们认为，由于客户关系管理（CRM）服务器不涉及财务或知识产权信息，不太可能成为攻击者的目标。

![image-20260501112649994](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501112649994.png)

此时，攻击者意识到客户关系管理（CRM）网络应用同时拥有面向公网和私网的IP地址。这么做是为了让远程和本地员工都能访问客户关系管理（CRM）的相关功能。对受害者而言，不幸的是，攻击者下载并安装了一个代理程序，该程序在TCP 4444端口进行监听，并将流量转发至内部网络。

实际上，这让攻击者得以进行横向移动，并绕过防火墙的过滤机制传输流量。他们的工具名为 Office_Remoter.exe。

**错误 4**：该主机本无需配置为双宿主模式，但出于业务正常运行的需要，其被如此配置后便一直未做调整。

**错误5**：受害者的内部网络未设置任何内部监控。做出这一决定的原因是IT人员认为配置得当的防火墙能够阻挡来自威胁的流量。

> [!NOTE]
>
> **dual-homed**：双宿主（网络安全专业术语，指主机配备两个网络接口，同时连接内外两个不同网络，是攻击者常用的横向渗透跳板）

![image-20260501112920604](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501112920604.png)

此时，绿企鹅攻击方再次开始进行侦察和扫描，只不过这一次他们是从防火墙后方拥有访问权限的主机发起操作的。

在进一步行动之前，绿企鹅攻击者注意到客户关系管理（CRM）网络应用的数据库中存在经过哈希处理的密码和用户名。他们迅速破解了多个账户的密码。对受害者来说不幸的是，许多被破解的密码也被用户与其域凭据重复使用。这使得绿企鹅成员得以在已验证账户的权限下开始访问资源。他们破解的其中一个账户属于一名数据库管理员，该管理员在数据库服务器上拥有本地管理员权限。

**错误6**：用户重复使用了易被破解的密码，包括数据库管理员的密码。

**错误7**：用户将CRM系统的密码重复用于Windows域

![image-20260501113035417](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501113035417.png)

在继续侦察和扫描的过程中，绿企鹅发现每台系统上都有一个名为 AssetMgtAcct 的本地账户。他们将注意力集中在这个账户上，利用在数据库服务器上的本地管理员权限从内存中提取了 NT 哈希值，并离线破解了密码。这一次耗时比以往更久，但在基于云的密码破解环境中运行几天后，密码还是被成功破解了。

现在，凭借本地网络上所有系统的管理员权限，绿企鹅在更多系统上安装了额外恶意软件，最终提升权限以访问阿格斯公司的域控制器。该恶意软件向已知的恶意IP地址发送信标信号。绿企鹅还在内部文件服务器上部署了备用监听器，以防失去对域控制器的访问权限并需要重新获取访问权。

**错误8**：资产管理账户在整个机构的所有系统上使用了相同的密码。

**错误9**：安装新软件（即持久化方法）时未设置任何警报。****

此时，绿企鹅攻击者 安装了 Office_Techneter.exe 程序，该程序将微软 TechNet 论坛（social.technet.microsoft.com）的帖子用作命令与控制（C2）机制。

在事件响应人员终止了CRM服务器上的两个进程（代理程序Office_Remoter和C2通道Office_Technet er）后，Green Pengtin仍在另外三个系统中留存了恶意软件，包括CRM命令注入漏洞以及来自内部系统的反向TCP连接。

**错误#10**：受害者未为发往TCP 4444端口的入站流量在防火墙中添加过滤规则。

**错误11**：受害者未扫描网络其余部分以查找相关的入侵指标（IoC），从而遗漏了其他已被入侵的系统。

# 2. 六步事件响应流程

![image-20260501113450267](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501113450267.png)

许多组织采用的事件响应模型与上方幻灯片中展示的模型类似。这种经典模型（通常称为 PICERL）包含六个主要阶段。

第一阶段是准备工作，指组织在事件发生前开展的各项活动。这包括制定政策、流程，落实内部监控，推行安全最佳实践等。

在事件响应启动之前，必须先对事件进行识别。这一过程可以通过多种方式实现。可能是用户致电服务台投诉“网络中断”，也可能是来自入侵检测系统的警报、系统管理员的报告等等。这个阶段通常也被称为检测阶段。

一旦发现事件，就需要对受影响的系统进行隔离。根据攻击者采取的具体操作，有多种隔离事件的方法。如果网站被篡改，将其下线是理想的做法（尽管并非总是可行）。有时，隔离会分为多个步骤：短期隔离，随后收集证据，待证据收集完毕后再进行长期（更具侵入性）的隔离。

清除指的是消除攻击者造成的破坏。终止攻击者进程、修改密码、删除恶意数据、启动欺诈调查等，都属于清除的范畴。

恢复指的是为让业务系统重新投入运行而采取的各项措施。想必该事件已对业务运营造成负面影响，而这些系统对于企业维持正常运转至关重要。

经验教训是在撰写最终报告并修复攻击者利用的漏洞时总结得出的。

这种事件响应模型被广泛采用或被改造用作更高级事件响应流程的基础。例如，美国国家标准与技术研究院（NIST）特别出版物800-61《计算机安全事件处理指南》（https://urls.sec504.org/0isr）也与PICERL模型非常相似，不同之处在于其中的遏制、根除和恢复步骤被视为流程中的一个主要步骤。

![image-20260501113725026](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501113725026.png)

经典的 PICERL 方法存在若干问题，问题不在于理论层面，而在于执行层面。例如，在准备阶段，数量惊人的组织连基础的安全措施都未能落实。传统的最佳实践（如最小权限原则、强密码等）执行得极为不到位。同样明显缺失的还有网络监控、日志聚合、活动审计等措施，以及用于审查这些信息的资源。许多组织在运用威胁情报方面也远未达到其应有的有效程度。

界定范围环节最大的问题并非执行本身。尽管受影响的组织可能能识别出事件的部分情况，但将范围限定在已知已遭入侵的系统上是一个致命错误。相反，他们应当（至少）在整个企业范围内扫描相关指标，以找出其他已被入侵的系统。

另一个常见问题是未能控制事件，在许多情况下（如针对 Argous 的事件），甚至完全跳过了这一环节。急于终止攻击者的进程通常意味着关键证据无法被收集，这会大幅增加全面了解事件的难度。范围界定不当也会导致控制措施失效，因为攻击者已经扩散到的系统处于未知状态。所有这些因素都会导致清除措施不当，让攻击者得以在环境中持续存在。而就业务系统的恢复而言，这是受害方可能做得更为周全的环节之一。毕竟，一旦业务运营受到负面影响，管理层往往会有所察觉。

在经验总结过程中，常见的问题是未能识别和/或解决根本原因。如果攻击者能够猜出弱密码，那么问题就在于：最初为何允许使用弱密码，以及需要修复哪些问题？是政策不完善？还是执行力度不足？

还有一个更深层次的问题，那就是采用线性（静态）的方法来处理事件响应。

## 事件响应动态方法（DAIR）

![image-20260501113910444](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501113910444.png)

有效的事件响应并无固定模式。各类事件会在不同时间发生，部分还会相互重叠（即多种情况会同时出现），而静态或线性的应对方式缺乏足够的灵活性，无法契合现代事件响应的实际需求。例如，你必须先检测到事件，再对其进行验证，但不能因某一事件就停下所有工作；检测是一项持续进行的活动，发现事件只是事件响应流程中的一个节点。同样，在清除环节对文件系统中的文件进行分析，可能会发现攻击者在最初的检测阶段未曾察觉的战术手段，这就需要我们重新执行之前的步骤，以更全面地了解受影响的系统。

与其将事件响应视为有明确开端和结尾的多个步骤，不如从途经点、结果和行动的角度来思考。为支持这一方法，我们开发了一种名为“动态事件响应方法（DAIR）”的新模型，如本页所示。

在DAIR模型中，我们能看到多个与PICERL以及NIST特别出版物《NIST800-61r2：计算机安全事件处理指南》（https://urls.sec504.org/0isri）中事件响应步骤相似的关键节点。不过，我们对DAIR模型的应用方式以及强调的核心行动有所不同。在DAIR模型中，我们以“检测”环节启动整个流程，该环节会综合考虑威胁狩猎和事件发现的多项要素，同时也会为高效的事件响应做好准备工作。当我们检测到**感兴趣事件（EOI）**时，会对其进行核实，以确认它确实是一起事件，并且是基于组织优先级需要我们做出响应的重要事件。接下来，我们进入响应行动循环，在此过程中，我们界定事件的影响范围、隔离受影响系统、清除已知威胁并恢复系统。随着我们对事件、攻击者的战术、技术与程序（TTPs）、涉及的系统以及业务关注点的了解不断深入，响应行动循环会持续推进，这些信息也会为后续的范围界定、威胁清除和系统恢复提供更多依据。

在完成响应动作循环以最佳方式应对事件所需的次数后，我们以复盘动作结束整个流程。在该动作中，我们评估受事件影响的系统以及我们自身的事件响应行动，以找出改进的机会。随着时间的推移，我们也会收集指标来衡量事件响应流程的效率。

围绕这些操作的灰色背景元素，代表了作为事件响应分析师，我们需要从决策者那里获取见解的所有对象：关键利益相关者、管理层、高管层以及其他代表组织利益的人员。在本课程中，我们将重点关注事件响应的技术层面：识别漏洞、梳理威胁指标（IOCs）、收集和审查与事件相关的证据，以及更多相关工作。不过，在整个过程中，我们会与决策者沟通并听取其意见，以确保事件响应操作符合组织的需求和优先级。

这是事件响应动态方法（DAIR）模型。它与 PICERL 模型或 NIST SP NIST800-61 指南并无显著偏离。相反，它帮助我们在明确重要额外考量因素的同时，重塑现有模型。DAIR 模型让我们能够运用相同的阶段，更高效地为事件响应职能做准备、识别事件、做出应对，并评估其有效性。在学习本课程材料的过程中，我们将持续应用 DAIR 模型。

接下来，我们来详细探讨将这种方法应用于事件响应的具体步骤。

### 准备阶段

![image-20260501114155234](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114155234.png)

DAIR、PICERL 以及 NIST SP 800-61r2 事件响应流程均以准备阶段为起点。在该阶段，我们会致力于让组织为应对网络安全事件做好准备。这将涉及多项工作，包括了解组织及其系统、提升内部可见度、制定恢复与验证相关的政策和实践规范，以及组建事件响应团队。

总结准备工作的最佳表述之一是：了解你的组织。理解企业看重什么。他们关注哪些方面？就事件通知执法部门与控制并清除威胁而言，哪项更为重要？此外，你还需精通公司政策——这既是因为你需要明确哪些情况属于事件、哪些不属于，以便在调查期间最大限度地利用资源，也是因为这能帮助你确定恰当的应对方式。

了解现代网络的正常状态也同样重要。这意味着内部可见性至关重要，因为事件发生期间的大部分证据通常来自内部日志记录机制（例如，内部入侵检测系统、防火墙日志、终端日志等）。关于如何实现这一可见性，有时存在争议：是采用基于网络的工具还是基于主机的工具。两者各有优劣，哪种方式效果最佳取决于事件类型和所处环境。无论选择哪种方式，都要确保能够清晰掌握网络层面和主机系统上的运行状况。此外，事件日志分析对于识别并区分正常与异常活动至关重要。收集到却从未审查的证据，对检测事件毫无帮助。

企业有时遇到问题的另一个原因是未对恢复流程进行测试。尤其是在处理包含一种或多种持久化方法的入侵事件时，最有效的恢复方式之一是从黄金镜像（例如用作系统基准的可信镜像）中擦除并重装系统，然后从可信备份中恢复数据。如果没有经过验证的流程和可信备份，就无法完成这一操作。

准备阶段的一部分工作还包括制定测试计划，以便我们能够验证预期的系统的功能性。制定测试计划至关重要，这不仅是为了应对突发事件，也是为了验证对平台的更新不会破坏必要的功能。许多机构会将系统测试委托给不同的主题专家（SMEs）；系统管理团队会制定测试计划，以确保基本的系统功能（登录、创建备份、服务验证等）正常运行，而主题专家则可能制定更详细的测试计划，以确保在系统内完成的操作能生成预期的输出。

最后，准备阶段还包括为事件响应团队自身做好准备。这可以采取培训（如本课程）、桌面演练以及定期审核事件响应和安全团队道德义务的形式进行。更多信息可访问 https://ethicsfirst.org/。

### 检测

![image-20260501114252705](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114252705.png)

作为一名事件响应分析师，你大部分时间都将用于为事件做准备以及检测组织内的威胁。在检测环节中，你会利用信息来源来识别感兴趣事件（EOI）和攻击指标（IOA）。EOI 和 IOA 这两个术语相关联，下面我们来分别解析它们。

攻击指标是指任何能揭示针对组织的过去或正在进行的攻击的事件。它们反映了攻击者为实现其目标所采用的战术、技术和程序（TTPs），无论攻击结果是成功还是失败。攻击指标通常被视为漏洞利用之前的活动，但这些事件可能在事件发生期间的任何时候发生（例如，攻击者在系统被入侵后试图绕过终端防护）。攻击指标的示例可能包括异常的登录失败、Web服务器日志中的漏洞利用证据或恶意电子邮件附件。

相比之下，利益相关事件更贴合组织的特定关注点和需求。任何攻击都可被视为攻击指标，而利益相关事件则是达到值得调查和潜在应对程度的受关注（或利益相关）事件。利益相关事件可包含攻击指标，但前提是根据组织政策认定其需要采取行动。指标的示例可能包括某一组织出现异常的登录失败记录，但另一组织可能会忽略此类事件，除非这些登录失败还与同一源IP地址的成功登录相关联。

检测可以是一种被动功能，即我们在威胁上报给我们时对其做出响应（例如，你从组织内部或外部人员处得知一起事件），也可以是一种主动功能，即事件响应团队主动寻找影响性攻击指标（IOA）和事件性攻击指标（EOI）。这种主动式检测方法被称为威胁狩猎，它能帮助组织缩短平均检测时间（MTTD，即事件发生开始到组织识别出该事件的时间间隔）。缩短平均检测时间是事件响应团队的持续目标，以此作为降低事件对组织造成影响的一种方法。

检测到事件后，我们立即开始记录报告时间，这是另一个有用的指标衡量事件响应团队的有效性：平均响应时间（MTTR）。平均检测时间（MTTD）是指从事件发生到被发现的时间间隔，而平均响应时间（MTTR）则是指从发现事件到团队遏制并修复威胁的时间间隔。本模块我们将继续探讨从检测到修复所涉及的各个步骤。

高效的事件响应团队认识到，被动和主动检测都依赖于多个数据源。除了技术数据源（网络使用统计数据、日志事件、终端警报等）之外，我们还可以与组织内部人员的信息来源密切合作。帮助台支持分析师在托管系统上发现异常行为，或系统管理员察觉到异常数量的支持请求，这些都可能揭示出事件指标（EOI），进而帮助发现组织内的安全事件。

### 验证、分类

![image-20260501114401774](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114401774.png)

一旦检测到潜在事件，下一步就是核实并进行初步分类。这将有助于指导你的初始事件响应工作。例如，应对勒索软件事件的方式与应对一名员工通过电子邮件威胁另一名员工的方式不同。在这两种情况下，我们都希望在启动完整的事件响应流程之前先核实事件。随后，我们需要通过让相关方参与（例如，系统管理员与人力资源部门）来对事件进行分类。

验证与分类操作并未在 PICERL 框架或 NIST SP 800-61r2 标准中被单独明确提及。但这一操作是事件响应流程中的重要环节，能确保事件响应团队的目标与业务需求保持一致。在 DAIR 示意图中可以看到，核心操作周围的灰色背景从验证/分类步骤开始，一直延伸至响应操作循环阶段。该区域明确涉及决策者的意见输入，包括管理人员、高管、关键利益相关者以及公司领导层。作为事件响应团队的技术分析师，我们通常负责技术层面的决策，并将尽可能收集到的全面信息提供给组织的决策者。验证与分类操作阶段，我们首先要确认组织认定这是需要开展调查的关注事件（EOI），随后根据组织优先级通过分类步骤对资源进行优先排序。

有多种商业工具可帮助事件响应团队跟踪事件细节（有时甚至能在检测阶段之前就开展跟踪），不过许多组织仍同时依赖手动文档工具。检测阶段之后，跟踪事件的一个优质资源是CrowdStrike Services数字取证与事件响应（DFIR）跟踪电子表格，可在https://urls.sec504.org/6v4jc获取。



# 案例、应用程序日志

![image-20260501114520960](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114520960.png)

我们来聊聊事件响应分析师常遇到的一个难题。看看安全运营中心（SOC）分享的某台WordPress服务器的日志数据，这是一次黑客攻击、无害事件，还是情况不明？

在这段简短的日志片段中，我们可用于分析的数据有限，但能看到多个 WordPress 事件，所有事件均报告事件 ID 1003（登录失败）。日期和时间信息覆盖了约11分钟的时间段，各事件之间的间隔从几秒到不到两分钟不等。源 IP 地址均唯一且分布广泛。每个报告的事件都对应一次登录失败尝试，仅有一个事件记录了两次登录失败尝试，另一个事件则记录了九次登录失败尝试。

这是一次黑客攻击、良性事件还是未知情况？作为事件响应分析师，我们经常面临类似的问题和两难困境：要么没有足够的信息做出明智的决策（比如这段简短的日志摘录），要么数据量过大而无法逐一审查（这里有40太字节的数据包捕获数据）。通常，我们需要提出问题来帮助我们理解系统：这台服务器的正常负载是多少？出现这种登录失败模式是否正常？服务器是否通常会处理来自如此广泛分布的IP地址的登录尝试？

# 范围界定

![image-20260501114657415](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114657415.png)

从事件被核实和分类完成到你完成调查的这段时间界限并不清晰。与线性事件响应流程的理想模式不同，我们实际调查事件的过程杂乱无章，需要在掌握新信息、有了更深入的了解后，回头重新开展之前完成过的分析任务。在 DAIR 模型中，我们将这组响应操作设计为一个循环，在循环中完成每个阶段，并根据早期分析活动的结果，多次重复执行这些阶段。

首先，范围界定环节是我们确定事件所涉及系统的步骤。这可能需要收集和检查证据、扫描网络、进行漏洞扫描、枚举配置细节或软件版本等操作。在事件初期，除了初步检测到的信息外，你可能掌握的范围界定信息十分有限。在完成包含、根除和恢复等其他事件响应步骤后，你很能更清楚攻击者的战术、技术和流程（TTPs），从而加深对事件的理解，并明确事件的有效范围。

例如，假设某起事件涉及向 Windows 系统导入新证书，同时对 HKCU\SOFTWARE\Microsoft\SystemCertificates 注册表项进行了修改。范围界定有助于我们通过检查网络中的其他系统是否存在类似的注册表项修改，来确定该事件还涉及哪些其他系统。缺乏有效的范围界定是我们在本模块前面提到的阿格斯公司数据泄露事件中的关键失误之一。

范围界定有助于我们了解事件的整体范围，但这通常需要获取组织内其他系统的数据。包括微软的系统中心配置管理器（SCCM）产品、终端检测与响应（EDR）工具以及软件物料清单（SBOM）产品在内的安全平台，在获取事件详细信息后，都对高效开展范围界定具有重要价值。即便没有部署这些系统，事件响应团队的分析师也可以借助包括 PowerShell 在内的其他工具，自动化开展有效范围界定所需的大部分分析工作。我们将在本节后续内容中探讨如何使用 PowerShell 进行实时调查。



# 攻击遏制

![image-20260501114757443](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114757443.png)

在遏制环节中，目标是阻止攻击者在已被入侵的网络内继续其操作。在我们采取额外步骤分析违规事件的性质之前，我们希望限制攻击者在组织内部可进行的操作。这对于常规攻击（即组织外部的攻击者利用内部系统实施攻击）和内部攻击都至关重要。通过遏制措施，我们能限制攻击者采取行动的机会，比如内部人员删除与人力资源行为违规相关的已发送电子邮件。

有助于实现遏制目标的活动示例包括：

- 使用专用VLAN隔离主机，以消除外部网络连接
- · 应用网络访问控制列表（NACLs）或安全组（SGs）等逻辑控制措施 修改 DNS 条目，使主机指向新的 IP 地址
- 通过防火墙控制阻止攻击者的IP地址
- 锁定用户账户以防止后续登录

遏制行动中的另一项重要工作是为后续调查收集数据。一旦攻击者无法与系统进行交互，事件响应团队可能希望收集证据以用于额外的调查目的，包括日志数据、实时系统配置详情、系统内存、磁盘镜像等。收集证据的决策在很大程度上由组织的目标决定。收集永远不会被使用的数据会给组织带来不必要的成本，因此公司管理层需要确定收集数据的成本是否值得为实现事件响应目标而付出。

遏制行动中的各项活动将助力实现其他目标，例如根除工作。例如，为实现遏制而枚举用户权限以阻止其访问，能够为后续撤销并重新发放必要的访问密钥提供所需的洞察。这有助于说明一个理念，即任何特定的活动都能同时助力实现不止一个目标。这也让我们明白，从成果而非线性步骤的角度思考为何至关重要：某一行动中的活动同样会为后续行动中的活动提供支撑。

# 清除

![image-20260501114922113](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501114922113.png)

清除操作旨在撤销攻击者所做的行为。它与遏制的区别在于，遏制是阻止攻击者在组织内部的操作，而清除则是移除他们所造成的所有变更。

清除行动需要通过开展调查行动并分析收集到的证据来获取洞察。借助实时调查技术（如执行命令以检查系统配置），我们可以了解攻击者所采取的行动。或者，我们也可以使用内存调查技术来获取类似的洞察，同时降低对实时系统中证据进行修改的风险。我们还可以采用其他技术，例如开展网络调查或恶意软件调查，以更深入地了解攻击者的战术、技术和流程（TTPs）。本节后续部分将对这些调查技术展开更详细的介绍。

以下是一些有助于实现根除目标的活动示例：

- 从受信任的备份中恢复系统（这意味着备份需处于可用状态）
- 移除攻击者的持久化进程、账户等
- 开展漏洞评估并修补漏洞
- ·处理欺诈交易
- 恢复被篡改的源代码

与遏制措施类似，此处的部分操作（如处理欺诈交易和恢复被修改的源代码）也有助于实现其他目标，包括恢复操作。

# 恢复

![image-20260501115020921](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501115020921.png)

在恢复阶段，我们会运用从前期检测、界定范围、遏制和根除阶段中获得的经验，来处置该事件以及导致事件发生的漏洞。根除行动的重点是分析证据并清除攻击者，而恢复步骤则会考量导致数据泄露的所有其他相关细节。基于这些经验，我们会对配套系统和流程进行调整，以恢复系统运行并防止未来再次发生类似事件。

恢复行动中的一项关键工作是开展根本原因分析，以探究导致事件发生的原因和经过。根本原因分析并非只关注单一的漏洞或入侵行为，而是需要我们审视所有未能阻止事件发生的控制措施。

例如，若某个组织因密码设置薄弱而遭受安全破坏，一项简单的根除措施就是修改密码。但在进行根本原因分析时，你需要提出以下问题：为何允许设置薄弱密码？是政策执行不到位吗？是政策虽执行到位但设计过于宽松，从而允许设置弱密码吗？是密码审计未能检测出该弱密码吗？是否存在我们未发现的先前登录失败记录，而这些记录本可以更早提醒我们这场攻击？

有时，找出事件的根本原因并非易事，尤其是当涉及多个因素和人员时。有一次，我以顾问身份与一名事件响应团队合作，处理一起公司财务管理软件所用敏感 Linux 系统的入侵事件。当我们得知最初的入侵是由于系统存在弱根密码，而该系统随后被用作跳板获取内部系统访问权限时，团队员工的情绪顿时激动起来。安全团队中有一人迅速将责任归咎于设置了弱密码的系统管理员，而该系统正是最初被入侵的目标。然而，这并非真正的根本原因分析。真正的根本原因分析不应只盯着设置弱密码的人，而是要探究为何会允许出现这样的弱密码，以及系统管理员是否接受过选择强密码的相关培训。根本原因分析的深度远超表面证据，它以不断追问“为什么”和“如何”为基础，从而制定更全面的纠正措施。

在恢复操作中，我们实施纠正性控制以修复漏洞并执行采取额外控制措施以防止未来发生类似事件。这些纠正措施不仅针对事件中涉及的直接系统，还会考量受数据泄露影响的其他系统。例如，假设某开发人员的工作站发生数据泄露，在恢复环节中，我们不仅会修复导致此次入侵的漏洞，还会考虑开发人员可访问的系统所面临的额外威胁。针对该事件的其他恢复工作可能包括：撤销并重新生成访问云环境的密钥（滚动更新密钥）、审查并验证开发人员可访问的源代码的所有变更、核查为软件开发工作流新建的所有基础设施，例如新增的云服务或源代码管理（SCM）相关变更（如新增的 GitHub Actions）。即便我们未发现攻击者具体使用了与数据泄露相关联的系统（比如开发人员所用服务的身份验证保存令牌），也会采取措施恢复这些系统，以确保攻击者不会在之后卷土重来。

恢复阶段的一部分工作是在实施补救控制措施后完成测试。这些测试计划应在系统恢复上线前，针对变更（补丁安装、新增控制措施、云环境调整等）验证系统功能。在可行的情况下，应让系统所有者和主题专家（SMEs）参与，在系统或环境发生重大变更后验证系统功能。

测试完成后，就该将系统恢复至可用状态了。进行系统恢复时，尽量选择在非工作时段操作。网络活动较少时更便于监控，不过这也意味着系统的停用时间会比立即恢复时更长。为简化监控而延长停机时间所产生的成本，需要决策者进行考量。

系统恢复上线后，需密切监控是否出现攻击者再次入侵的迹象——尤其是若其安装了任何类型的持久化工具，这表明其有再次入侵的意图。需监控网络传感器、操作系统和应用程序日志，查找与该事件相关的特定入侵迹象。

# 重复响应操作

![image-20260501115153546](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501115153546.png)

在DAIR模型中，我们强调需要多次执行响应行动循环。在开展调查并执行遏制、清除和恢复行动的过程中，你将进一步了解攻击者的战术、技术与流程（TTPs），以及组织存在的技术或政策漏洞。这些洞察可能会揭示组织内的其他威胁，包括其他已被入侵的系统、额外漏洞、更多网络威胁情报（CTI）细节等。

结合调查获得的新见解，你将多次回顾这些响应操作。了解攻击者使用的新IP地址或域名，会促使你重新开展范围界定工作，以找出事件中涉及的其他系统。了解一种利用DNS进行命令与控制（C2）的持久化机制，可能会影响你实施遏制策略的方式。了解一种利用经过签名的Windows驱动程序绕过终端检测与响应（EDR）系统的战术，可能会对你的根除策略构成挑战。

在DAIR模型中，我们会根据需要多次应用响应行动循环，以有效理解并应对事件。



# 复盘

![image-20260501115232367](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501115232367.png)

完成响应行动循环通常是在组织决策者的指导下进行，或是当事件中已无更多需重新评估的要素时完成。此时，你应将事件详情的复盘作为事后活动来完成。

在复盘环节，我们会回顾事件期间汲取的经验教训，并撰写最终事件报告（有时也称为事后总结报告），该报告将记录事件中的亮点、不足，以及为改进事件响应流程、预防未来事件而做出的各项调整。

有时，报告会经过润色并附上封面，随演示幻灯片一起分发给关键利益相关方审阅。有时，复盘则是对事件响应活动的评估，以笔记形式记录在票务系统中，供管理层或事件响应团队日后参考。事件发生后投入到复盘和最终成果（正式报告或简易总结）中的时间，很大程度上取决于其对组织的价值。

事件响应团队和支持管理层复盘复盘报告的细节会很有帮助，借此讨论已做出的变更、优化事件响应流程的机会，并肯定所取得的成果与成功。这也是申请升级安全基础设施的好时机，因为事件发生后，安全问题会立刻成为人们关注的焦点。

复盘报告的格式因组织而异。部分组织可能希望记录事件详情、事件时间线、组织影响评估以及事件发生后采取的行动。另一些组织则可能处理的事件数量极多，撰写此类报告过于耗时，因此会将复盘简化为几段文字，引用票务系统或其他行动跟踪平台中的相关链接即可。无论复盘报告采用何种格式，应尽可能以事实呈现细节，若无法确证则注明为推测。在呈现事件细节时需保持客观、避免偏见，以维护与组织内其他人员的可信度和信任关系。

很少有组织会公开分享事件复盘报告。安全分析师塞巴斯蒂安·海格撰写了一篇一份实用指南，可帮助你撰写事件响应简报报告，还附带可适配你所在组织的模板，访问地址为 https:/urls.sec504.org/10tmc。

DAIR 模型与 PICERL 中描述的步骤或 NIST SP 800-61《计算机安全事件处理指南》并无本质差异。这三种模型均认可，组织需在通过复盘或事件后审查活动完成事件处置前，做好事件准备、开展检测工作，并实施遏制、根除与恢复操作。DAIR 模型引入了其他模型未涵盖的两个关键要素，具体而言，即验证事件并进行分类分级的需求，以及确定事件在组织资产中波及范围的界定步骤。DAIR 模型采用响应操作循环模式，随着我们逐步了解攻击者、深入剖析导致组织暴露的漏洞、调查事件响应过程中收集的数据，会多次执行界定范围、遏制、根除和恢复这些步骤。此外，我们将技术分析团队与组织内关键利益相关方的决策区分开来，这是确保事件响应流程契合组织需求与优先级的核心环节。

在本课程中，我们将继续在研究事件响应调查技术和黑客工具的过程中，持续应用 DAIR 模型来开展事件响应工作。

在本模块中，我们探讨了理解应对网络安全事件的流程和技术所必需的相关主题。我们首先回顾了绿企鹅攻击者组织入侵Argous公司的事件。在此次事件中，应急人员发现某一系统存在恶意软件并立即将其清除。但由于未遵循事件响应流程，Argous公司的工程师错失了多次全面掌握入侵影响范围的机会。

我们研究了线性PICERL（准备、识别、遏制、根除、恢复和经验总结）事件响应模型，发现尽管该流程本身结构完善，但这些步骤的实际应用往往会让组织面临风险，表现为识别不彻底、未采取遏制措施、恢复力度不足，且常常无法总结经验教训。相比之下，动态事件响应方法（DAIR）是对事件响应成功步骤的现代化重新解读，也是本模块的核心内容。

在本模块中，我们介绍了事件响应动态方法（DAIR），这是一种适用于事件响应流程的现代化方法。我们探讨了DAIR模型的早期阶段：通过应用检测和威胁狩猎技术为事件做准备，随后进行事件验证与分类。我们还研究了响应步骤——确定范围、遏制、根除和恢复，在此过程中始终并听从决策者的指导。这一过程是迭代的，我们从数据收集（遏制行动阶段）、分析（根除行动阶段）以及根本原因梳理（恢复行动阶段）中获得的见解，都将助力我们更全面地了解事件，从而更精准地界定事件范围。在多次完成这些行动后，我们会以复盘收尾整个DAIR模型，总结成功之处、不足之处，以及可做出的改进措施，以此优化事件响应能力和组织的整体安全性。

在本课程的后续内容中，我们将在研究攻击者的战术、技术与流程（TTPs）时运用该模型。我们将结合实操实验，进一步拓展这些实用领域的知识，同时学习如何运用基于分析的事件响应方法，采取技术步骤来有效应对组织所面临的威胁。

# 实时调查

接下来，我们将回顾在系统被入侵之前、之中以及之后对活动系统进行检查的各个方面。

一个常被问及的问题是，如何找到你尚不知情的妥协？本质上，你该如何发现那些已遭入侵的系统，而攻击者的行为又不会留下已知的特征指标？换个说法，你该如何在你的网络中找到目前尚未察觉的攻击者？

或者在发生事件时，如何通过刚发现的威胁指标（IOCs）来确定范围，以找到其他被入侵的系统？此外，如何核实可能发生事件的相关报告？

一种观点是检查网络流量，因为在大多数情况下，攻击者会通过网络与系统进行交互。然而，并非所有恶意软件都会产生网络流量，而且加密技术会让识别网络流量中的攻击者行为变得困难得多。你还必须考虑那些采取措施融入周边环境的攻击者，他们会使用服务器消息块（SMB）、远程桌面协议（RDP）和虚拟网络计算（VNC）等协议。

在接下来的几张幻灯片中，我们将探讨分析运行中的Windows系统主机环境的方法，以尝试解答这些问题。本模块中我们所讲解的分析技术，也适用于内存分析技术的研究。

在本课程中，我们将以微软 PowerShell 作为主机分析的主要工具。PowerShell 为分析人员提供了强大的工具，用于获取有关主机或 Windows Active Directory 域的信息。它还通过 PowerShell 管道的筛选器和数据迭代器为我们提供分析功能。此外，还可通过编写从简单到复杂的脚本，实现安全分析的自动化。PowerShell 的使用方式可以很基础，它预先配置了命令提示符中你可能已经熟悉的常用命令别名。随着你对 PowerShell 愈发熟悉，它将成为编写脚本和实现自动化的强大工具。

除了强大的功能之外，PowerShell 也存在一些缺陷。它的内省能力有限（即在运行时调查功能的能力），且当攻击者获取 PowerShell 会话的访问权限时，会带来安全风险。PowerShell 能提供对主机相关信息的高效数据访问，但所提供的信息也可能令人遗憾地存在缺失。此外，PowerShell 的学习门槛较高，还需要我们转变访问和处理数据对象的思维方式。

我们认为投入时间学习 PowerShell 是值得的，因为它能为你提供改进和加速分析 Windows 乃至 UNIX 系统的机会。在本模块中，我们会介绍多个 PowerShell 命令示例，并简要讲解构成 PowerShell 工作原理基础的核心要素。在本节末尾的实战训练营中，我们还会花更多时间深入学习 PowerShell。

## 检查进程

检查正在运行的系统的一项基础操作是调查正在运行的进程。借助 PowerShell，我们可以使用 Get-Process 命令行工具（命令行工具是实现特定功能的 PowerShell 命令，通常用于获取数据）。不带任何参数运行 Get-Process，将显示有关正在运行进程的多列信息：

![image-20260501120317318](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501120317318.png)

`Get-Process` 也接受参数，包括特定的进程名。我们还可以使用一个或多个星号通配符来匹配多个进程。

调用 `Get-Process` 时，会返回一个对象。对象是 PowerShell 中使用的一种基本数据类型，包含代码或数据。运行 `Get-Process` 时，我们会看到几列默认信息，但对象中还包含未显示的额外信息。要获取这些额外信息，我们可以使用竖线（管道符）构建 PowerShell 管道，将 `Get-Process` 的输出发送到另一个 cmdlet（例如 `Select-Object`）。借助 `Select-Object`，我们可以从对象中获取额外信息，也可以使用星号通配符获取所有数据字段。

![image-20260501120729842](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501120729842.png)

许多 PowerShell 命令行工具默认在本地系统上运行，但也能从远程系统检索信息（需具备相应权限）。添加 `-ComputerName` 参数，后跟 Windows 主机名或 IP 地址，即可尝试使用你的登录凭据从指定系统获取该命令行工具的相关信息。

我们在使用 PowerShell 时很快会发现一个问题：并非总能从一些 cmdlet 中获取到所需的全部信息。Get-Process 对于查询进程详情很有价值，但它既不会提供父进程信息（比如启动了你正在调查的进程的那个进程），也不会给出启动该进程时使用的命令行参数。对于事件响应人员来说，父进程 ID 和命令行参数是判断进程是否存在可疑迹象的关键依据。

幸运的是，微软提供了 `Get-CimInstance` 这个 cmdlet。CIM 即通用信息模型，是 Windows 管理规范（WMI）框架的一部分，借助它我们可以查询 Windows 主机的详细信息。`Get-CimInstance` 能为我们提供不同元素的大量有价值信息；要获取进程信息，可参考本页所示的 win32_Process 类。借助 PowerShell 管道，我们可以筛选出一个或多个进程的进程 ID、进程名以及命令行详情，具体如本页所示。

```powershell
PS C:\> Get-CimInstance -Class Win32 Process | Select-Object ProcessId, 3652 nc.exe ProcessName, CommandLine "C:\Tools\nc.exe"-1 -p 11111-e cmd.exe
```

我们还可以将 Get-Process 的输出与 Get-CimInstance 结合使用，以获取有关进程的其他信息。在本页的第二个示例中，

```powershell
PS C:\Users\Sec504> Get-Process 'lsass'| Select-Object -Property Id
644
PS C:\Users\Sec504> Get-CimInstance -Class Win32 Process | Where-Object -Property ParentProcessId -EQ 644

ProcessId Name     HandleCount WorkingSetSize VirtualSize   
2608 	  cmd.exe  49           2908160       2203370045440 
```



我们使用 `Get-Process` 选择本地安全机构子系统服务（LSASS）的 Id 属性（即进程标识符）。借助此信息，我们可以使用 `Get-CimInstance`  cmdlet 查找以 LSASS 为父进程的所有子进程（这种情况通常不会出现，但却是常见的入侵迹象/指示符）。此处的管道操作会稍微复杂一些：

1. `Get-CimInstance -class Win32_Process |` —— 返回包含 Windows 进程信息的对象，用 `|` 启动 PowerShell 管道
2. `Where-Object` — 启动一个循环，检查 `Get-CimInstance` 返回的每个进程
3. `-Property ParentProcessId` -- 基于 ParentProcessId 属性值创建筛选表达式
4. `-EQ 644` -- 显示 ParentProcessId 参数等于 644 的进程

此 PowerShell 管道的输出表明，本地安全机构子系统（LSASS）进程启动了一个 cmd.exe 子进程。这并非 LSASS 的正常行为，应将其作为指示性指标（IOC）进行调查。

注意：Get-Process 使用属性名 Id 来指代进程 ID，而 Get-CimInstance 使用属性名 Process Id 来指代相同的信息。

## 识别可疑进程

![image-20260501122344830](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501122344830.png)

那么，哪些进程或属性是可疑的呢？与事件响应和数字取证中的许多情况一样，不存在一份包含所有可疑进程和可疑属性的权威清单。这在一定程度上取决于经验，但以下是我们过去常见的一些情况：

- 该进程对系统而言是全新的还是基本未被识别？这通常需要首先明确系统中本应存在哪些进程。
- 该名称是否看起来是随机（或伪随机）生成的？
- ·该程序是公认的程序，但从非标准路径运行吗？
- . 该进程是否为可疑进程的子进程？
- 父子进程关系通常是否可疑？例如，Isass.exe 生成 cmd.exe 的副本是相当异常的。
- 该进程是否与任何可疑（非进程类）活动相关联，比如与已知的恶意主机进行通信？
- 该进程是否使用了 Base64 编码的命令行选项？

密切关注任何带有 base64 编码命令行选项的正在运行的进程，尤其是 PowerShell 进程。许多攻击者和恶意软件编写者会利用 PowerShell 的 `-EncodedCommand` 选项在命令行中指定要运行的脚本内容。这样一来，文件系统中就不会残留多余的 `.ps1` 脚本文件。请注意，并非所有使用 `-EncodedCommand` 的 PowerShell 脚本都是恶意的，但它们值得进一步调查。

请记住，这些都是可疑迹象，不能保证某个进程就是恶意的。这意味着你可能仍需要分析这个可疑进程。

最后，别忘了转换方向。如果你发现了可用于识别其他可疑进程（或其他类型的伪迹）的信息，务必对这些内容也展开调查。

在对运行中的 Windows 系统进行检查时，另一个需要调查的要素是当前的网络使用情况。这可能包括等待攻击者传入连接的监听进程，或是与攻击者的命令与控制（C2）基础设施建立的传出连接。

PowerShell 的 `Get-NetTCPConnection` cmdlet 会显示网络连接信息，包括监听端口、绑定端口（保留端口号但尚未接受连接）、已建立的连接以及其他多种状态。不带参数运行时，`Get-NetTCPConnection` 会展示用于排查网络问题的实用详情，涵盖处于不同连接状态的监听连接和出站连接。

`Get-NetTCPConnection` 还可配置为专注于特定的网络连接状态（如监听服务），并通过管道传递给 `Select-Object` 以获取所需的特定属性，本页对此有相关示例。`Get-NetTCpConnection` 有一个特别实用的调用方式展示在第三个示例中，该示例针对特定的远程 IP 地址（例如攻击者的可疑 IP 地址）进行排查，以调查所有状态下的所有活跃连接。

Get-NetTCPConnection 存在一个问题，即虽然它能显示与网络连接关联的进程 ID，却不会展示进程名称。不过，获取到进程 ID 后，你可以使用 `Get-Process -Id 进程ID` 来查询进程名称。或者，这条 PowerShell 命令可通过结合带有 `@()` 语法的哈希表，在一条命令中同时获取网络连接详情和进程名称：

```powershell
PS C:\> Get-NetTCPConnection | Select-Object local*,remote*, state, @(Name= 'Process';Expression=( (Get-Process -Id $_.OwningProcess).ProcessName)} | Format-Table
```

### 解析 Get-NetTCPConnection 的地址、端口和状态

![image-20260501123442682](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501123442682.png)

Get-NetTCPConnection 的输出结果可能难以解读，因此我们花点时间来看看一些不直观的字段输出。在本页的示例中，我们可以看到多个列，包括本地地址（本地系统的 IP 地址）、本地端口、远程地址、远程端口和状态。乍一看，这些标题很直观，但列出的部分值可能会让人困惑。

首行的本地地址（LocalAddress）为`::`，该标识表示445端口在设备所有**IPv6接口**上处于监听状态。远程端口（RemotePort）对监听状态的条目无实际意义，因此该字段标注为0。同理，本地地址（LocalAddress）为 0.0.0.0 时，同样代表端口在所有网络接口监听，二者的区别为 0.0.0.0 对应设备所有**IPv4接口**。

本地端口9001上的监听服务被限制在特定本地地址127.0.0.1。127.0.0.1是IPv4的环回地址，意味着它仅能从本地系统访问。从该输出中我们可以判断，TCP 9001端口上有服务在监听，但该服务仅能从本地系统访问。由于127.0.0.1仅本地系统可访问，因此没有外部系统能够访问该系统的TCP 9001端口。

本地端口 139 上的监听服务类似，不同之处在于 TCP 端口 139 仅在 IPv4 地址 192.168.171.190 处于监听状态的网络接口上可访问。实际上，这与监听所有接口（0.0.0.0 或 ::）类似，除非系统拥有多个网络接口。例如，若一个系统同时拥有有线和无线接口，那么此输出中 TCP 端口 139 的监听服务将仅接受配置了 192.168.171.190 这一 IP 地址的接口（无论是有线接口还是无线接口）上的连接请求。

其余两个条目均针对处于不同状态的 TCP 连接的出站连接，其中“已建立（Established）”是本地地址与远程地址之间的开放 TCP 连接，而“等待关闭（Time Wait）”是正在关闭过程中的 TCP 连接。有关 TCP 连接不同状态的更多信息以及 Get-NetTCPConnection 命令的其他选项，可在微软官方文档中查看，网址为 https://urls.sec504.org/mdpti。

## 识别可疑的网络活动

![image-20260501124443583](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501124443583.png)

与识别可疑进程类似，目前并不存在一份完美的可疑网络流量识别指标清单。基于这一点，以下是一些需要考虑的常见事项：

1. 任何与相关进程异常的网络活动——例如，记事本多次连接80端口，或某个服务建立多个出站连接。不过在这种情况下，这也可能是自动更新服务。
2. 任何针对特定环境的异常网络活动。示例包括非工作时段的大量活动、长期运行的HTTP和HTTPS会话，以及信标流量——这类流量具有一定规律性，且表现为频繁的小型出站通信。
3. 特定技术独有的任何指标。例如，通过 SMB 进行横向移动意味着与其他内部系统建立了到 SMB 端口（TCP 和 UDP 135-139 以及 445）的连接。任何往来已知恶意主机的流量。你可以根据现有的威胁情报，或基于其他可疑进程与连接（即基于横向渗透）来确定这一点。

另一项实时排查活动是检查Windows服务。Windows服务是通过Windows服务.exe进程控制的后台任务，可配置为自动启动。攻击者常将Windows服务用作持久化手段，以便在系统重启后仍能持续访问已被入侵的设备。

我们可以使用 PowerShell 的 `Get-Service` 命令来检索所有正在运行的服务的信息，也可以选择性地指定服务名称来获取该服务的相关信息。与其他命令一样，我们还可以将 PowerShell 管道与 Select-Object 结合使用，以检索更多信息。

## 异常计划任务

![image-20260501130505306](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501130505306.png)

Windows 系统的另一项常见现场调查任务是检查计划任务。计划任务是指在指定延迟后（例如每 30 分钟运行一次）、特定时间点（例如凌晨 3 点 23 分运行）或 Windows 事件日志中记录特定事件后（例如事件日志中记录登录失败时运行）执行的作业。攻击者可能利用计划任务在已入侵的系统中维持持久权限，在需要时运行本地程序。这些攻击者创建的计划任务可能使用怪异的名称（如随机字符串），但有时也会刻意融入系统，采用系统中未使用的合法软件名称。

我们可以使用`Get-ScheduledTask` 来获取系统上的计划任务列表。不带参数运行时，`Get-ScheduledTask` 会显示所有计划任务的列表，我们也可以获取特定计划任务名称的相关信息，还可如本页所示选择性地使用通配符。与其他 cmdlet 一样，我们可以使用 `Select-Object` 或 `Format-List` 来检索计划任务的特定属性。

与其他 PowerShell cmdlet 一样，`Get-ScheduledTask` 不会提供你可能想查看的有关计划任务的所有详细信息，例如要执行的命令或提供给该命令的命令行参数。要获取这些额外的详细信息，我们可以使用 `Export-ScheduledTask` cmdlet 将计划任务导出为 XML 格式（如本页所示）。

在调查过程中，你可能还想知道计划任务的上次运行时间。要获取此详细信息，请使用 `Get-ScheduledTaskInfo` cmdlet，通过它你可以查看 LastRunTime 属性、NextRunTime 属性，以及上次执行任务的状态（位于 Last TaskResult 中）。

## 异常日志

![image-20260501130713363](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501130713363.png)

另一个实用的实时调查功能是利用 Windows 事件日志数据。`Get-WinEvent` cmdlet 是用于查询 Windows 事件日志数据的现代 PowerShell 功能（取代了旧版 cmdlet `Get-EventLog`）。不带参数运行时，`Get-WinEvent` 会提示你输入事件日志名称，并显示指定事件日志中所有事件的摘要信息。不过，通过一些额外的筛选，我们可以将分析聚焦于特定事件日志资源在指定日期范围内的事件。

在此页面的第一个示例中，我们创建了两个变量（其标识符为 `$start` 和 `$end`），它们是 `Get-Date`  cmdlet 的返回值，分别代表一次调查的开始日期和结束日期。要指定事件日志资源以及开始和结束日期，我们使用 `-FilterHashtable` 参数，并通过 @{} 语法结合 LogName、StartTime 和 EndTime 参数构建一个字典对象。

`Get-WinEvent` 还可通过 `-LogName` 参数接受事件日志资源名称，我们也能使用 `Where-Object` cmdlet 对结果进行筛选，以显示特定事件。选择使用 `Where-Object` 还是构建 `FilterHashtable` 通常取决于个人偏好，不过 `FilterHashtable` 的执行速度通常比 `Where-Object` 管道更快。

在本页面的第二个示例中，我们展示了一个 `Get-WinEvent` 管道，用于检查系统事件日志中所有事件 ID 为 7045（新服务已创建）的事件。我们选择将输出通过管道传递给 FormatList，以便在单独的行中获取 TimeCreated 和 Message 字段，从而查看完整的消息详细信息（使用 `Select-Object` 会返回消息的截断版本）。在输出结果中，我们看到系统上安装了一个新服务，该服务的名称可疑，并且以可疑的方式调用了 &COMSPEC%（Windows 命令提示符解释器），该调用会在隐藏窗口中启动 PowerShell。使用 FilterHashTable 的等效查询为 Get-WinEvent -FilterHashtable @{ LogName='System'; Id='7045'; }。

在这些示例中，我们研究了系统和安全事件日志，但 Windows 还会在其他事件日志中记录大量详细信息。我们可以运行 `Get-WinEvent -Listlog * | Select LogName, RecordCount` 来获取系统上所有事件日志的列表以及每个日志中的记录数。

使用事件日志数据在调查中可能会有所帮助，但攻击者可能会尝试清除或修改此类数据以掩盖其行为。也可以使用其他事件日志查询工具来收集事件日志数据，用于法医调查，例如可从 https://ericzimmerman.github.io 获取的 EVTXECMD。

注意：要访问 Windows 安全事件日志，必须以管理员身份打开 PowerShell。右键单击 PowerShell 图标，然后选择“以管理员身份运行”。

## 差异分析

![image-20260501131630494](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501131630494.png)

在进行实时调查时，很容易因待调查的数据量而不知所措。通常会涉及数百个服务、进程、计划任务和网络连接，更不用说文件系统上数十万的注册表项和文件需要排查。

聚焦调查的一种方法是使用差异分析。通过差异分析，我们会利用系统部署时的配置基准信息（通常使用作为工作站或服务器部署起点的黄金镜像），并将保存的结果与当前系统进行对比。

在本页的示例中，我们首先使用 `Get-Service` 收集服务列表，然后通过重定向运算符 >（大于号）将输出保存到名为 baseline-services-20220325.txt 的文件中。

在对系统进行调查时，我们会生成相同的输出，这次将 `Get-Service` 的输出重定向到 services-liveinvestigation.txt。接下来，我们使用 `Get-Content` 命令创建两个 PowerShell 对象，分别包含两个文件的内容，其中一个对象的标识符为 `$baseline` 和`$current`。最后，我们使用 `Compare-Object` 比较这两个对象，通过 `-ReferenceObject` 指定基准对象，用 `-DifferenceObject` 指定被调查主机的服务信息。

`Compare-Object` 命令的输出能帮助我们快速发现黄金镜像与被调查系统之间的配置差异。SideIndicator 列可帮助我们查看差异出现在何处。SideIndicator 为 => 表示差异对象发生了变化，为 <= 表示引用对象发生了变化。一个简单的记忆方法是，箭头的方向指明了 InputObject 列中显示的变更所对应的位置。

例如，在本例中，我们可以看到两个在两个系统上都存在的服务：软件保护服务与 Windows 模块安装程序服务。同时可见，这两项服务的状态已由运行变更为停止。输出首行中的 **=>** 标识表明，停止状态对应**对比对象**（即变量`$current`所代表的受检系统数据）。同理，输出第三行显示，受检系统中新增了一项名称为`vjinxSolt`、描述为`HQIQATUbyNM`的服务（**=>** 标识代表该条目为对比对象中的新增项）。该行下方的标识发生变更，表明软件保护服务在**基准对象**（即变量`$baseline`所代表的黄金镜像系统服务查询结果）中处于运行状态。

差异分析在加快实时调查流程方面极具价值，但其发挥作用依赖于具备必要的基准信息。在尚未收集基准信息的情况下，可选择部署新系统（使用标准镜像）来收集基准信息，随后将其与被调查系统一同用于分析。

## 额外的辅助工具

![image-20260501132212537](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501132212537.png)

除了 PowerShell 命令行工具外，Sysinternals 工具套件（https:/docs.microsoft.com/en-us/sysinternals/）是一套出色的工具，你可以考虑将其加入到事件响应与检测工具库中。以下是几款常用的工具：

- Process Explorer 可提供正在运行的进程的极其详细的信息（包括内存检查）。
- . Process Monitor 会实时显示（并记录）文件系统、注册表、网络和进程活动。
- . TCPView 会将正在监听的 TCP 和 UDP 端口追溯至所属进程。

## 总结

![image-20260501132411480](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501132411480.png)

这份可视化关联摘要作为 Summary-LiveExamination.pdf 包含在你的 ISO 媒体文件中

在本模块中，我们重点讲解了实时调查的流程——即运用各类工具和技术，分析运行中的系统活动，以识别入侵指标（IOCs）。我们聚焦于Windows系统，借助PowerShell及其他工具，从疑似参与事件的系统中收集并评估数据伪影。（我们将在第1节末尾的训练营资料中，讲解Linux实时调查技术）。

我们首先介绍了如何使用 Get-Process 以及 Get-CimInstance 类的 Win32_Process 命令来枚举进程。在 Windows 系统上对进程进行枚举时，通过检查命令行、父子进程关系、非标准路径等不同要素，能够发现攻击者的指示符（IOCs）。编码后的命令行参数也可能指向恶意进程，我们可以借助 CyberChef 来辅助解码命令行参数及其他数据。

网络连接排查也可用于识别监听进程、传入连接和传出连接。PowerShell 的 `Get-NetTCPConnection` 命令提供了多种选项来查询网络连接，可将连接或监听服务与本地进程相关联。

我们了解到Windows服务是由services.exe进程控制的后台任务。攻击者通常会在已被入侵的Windows系统中添加一项服务，以便在后续时间重新获得访问权限。借助Get-Service的输出结果，我们可以识别服务名称和其他参数，从而区分授权服务与未授权服务。尽管Get-Service不会披露服务的其他特征，例如可执行文件路径和命令行参数，但我们可以使用GetCimInstance -Class Win32_Service来获取这些信息以及其他相关数据。

我们还研究了使用 PowerShell 进行注册表查询。PowerShell 的 Get-ChildItem cmdlet 用于列出文件系统的内容，我们也可以类似地使用它来列出注册表项，此时需指定 HKLM 或 HKCU 作为顶级项，而在文件系统操作中我们通常会指定驱动器号。要检索注册表项的值我们可以使用 Get-ItemProperty 这个 cmdlet。

要枚举本地用户账户，我们使用 `Get-LocalUser` 命令行工具。除了验证所有本地用户账户均已获批准外，我们还可以调查本地组成员身份：使用 `Get-LocalGroup` 列出 Windows 系统上的组，使用 `Get-LocalGroupMember` 枚举组成员。

Windows 上的计划任务将按照事件日志事件的时间执行。我们可以使用 PowerShell cmdlet Get-ScheduledTask 列出 Windows 计划任务。该 cmdlet 会提供计划任务的摘要信息；若要获取完整的计划任务详细信息，可使用 Export-ScheduledTask cmdlet 将其导出为 XML 格式。若要查看计划任务的上次运行时间、下次计划运行时间以及上次执行状态，可使用 Get-ScheduledTaskInfo cmdlet。

我们还研究了用于查看事件日志信息的 PowerShell 功能，包括枚举本地事件日志列表，并使用 Get-WinEvent 检索汇总或详细信息。通过结合其他 PowerShell 功能，我们可以构建筛选哈希表，或利用常规的 PowerShell 管道功能以及 Select-Object  cmdlet，在事件日志数据中搜索特定的事件 ID 或按日期范围进行筛选。

我们还研究了在实时调查过程中应用差异分析的流程。差异分析能够借助基线信息（包括服务列表、计划任务、本地用户账户、网络服务监听器等配置细节），极大提升你检查实时系统的能力。我们会使用 Compare-Object  cmdlet 将这些基线细节与被调查系统的配置进行对比，从而识别出新增（或缺失）的细节。

如果本模块是你首次接触 PowerShell，面对要熟练运用这些功能所需记住的大量名称和参数，你可能会感到无从下手。随你班级的 ISO 镜像文件下载包，我们还提供了多份 PowerShell 速查表，助力你运用这些命令。其中，由埃德·斯库迪斯（Ed Skoudis）和菲尔·史密斯（Phil Smith）编写的《PowerShell 速查表》（PowerShellCheat Sheetv4.0.pdf）提供了大量运用 PowerShell 以及构建 PowerShell 管道来检索信息的优质示例，而史蒂夫·安森（Steve Anson）编写的《PowerShell 快速参考指南》（PowerShel1-Quick-Reference.pdf）则对许多用于实时调查的 PowerShell 命令行工具进行了出色的总结。

# 网络调查

![image-20260501132930634](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501132930634.png)

网络流量能够提供有关所发生事件的宝贵线索。至少，由于大多数攻击者不会坐在本地键盘前操作，他们必须通过网络与目标系统进行通信。

网络调查的证据可来自几个基本类别。原始网络流量（通常以实时捕获或磁盘上存储的数据包捕获形式存在）是可供分析的底层且详尽的网络信息来源。防火墙、代理服务器等网络设备同样是有价值的证据来源。最后，生成网络交互日志的主机设备（例如存储在Windows事件日志中的信息）也能提供有价值的分析依据。

虽然网络调查可能会颇有成效，但重要的是要认识到有一些问题可能会影响你的调查。首先是可访问性，因为并非所有设备都能轻松获取其生成的数据。有时，数据只能以不利于分析的格式导出；有时，它甚至可能根本无法导出。

网络证据的另一个问题是保真度。由于并非每个来源都会记录每个数据包或每次交互的所有细节，你需要分析的数据可能会受到限制。

最后，加密可能会对人们查看数据并理解其含义的能力产生负面影响。

网络信息在调查中可能是一项宝贵资产，尤其是在梳理时间线信息方面。但不应将其作为唯一的分析手段。相反，网络分析技术应结合其他分析方式，包括实时分析、内存分析和恶意软件分析。在本模块中，我们将探讨如何评估基于网络的数据源，以生成有关攻击者的威胁情报。

## tcpdump

![image-20260501133037564](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501133037564.png)

Tcpdump 是捕获和处理网络数据包最常用的工具之一。它已被广泛使用数十年，但仍在积极维护中。可在所有主流平台上使用，也常见于许多嵌入式设备中。

虽然 tcpdump 通常被视为一种底层工具，但它也能够对 IP、TCP、UDP、ICMP 等协议进行基础的协议分析。正如我们很快就会看到的，tcpdump 的真正优势之一在于其过滤功能。

### 常用命令

```bash
$ tcpdump -i interface  #捕获某个网络接口的流量，也可使用 any 接口。
$ tcpdump -i interface -w file #捕获指定接口的流量并将其写入文件
$ tcpdump -r file -n #从文件中读取数据包，且不解析主机和端口。
$ tcpdump -r file -n -A #从文件读取数据包，不进行解析，以 ASCII 格式显示。
```

上面展示了一些 tcpdump 常用的命令行选项。

要使用 tcpdump 捕获特定网卡接口的数据包，需使用 -i 选项后接接口名称。例如，在 Unix 系统中，-i eth0 会捕获 eth0 接口上的数据包。要获取接口列表，可运行带有 -D 选项的 tcpdump 命令。

如果你想将捕获到的数据包写入文件，可使用 -w 选项，后面跟上要写入的文件名。例如，以下命令将从 vmnet 42 接口捕获数据包，并将其保存到名为 output.pcap 的文件中。

```bash
$ tcpdump -i vmnet42 -w output.pcap
```

要从 pcap（或 pcapng）文件中读取数据包，请使用 -r 选项并后跟文件名。以下命令将从名为 packets.pcap 的文件中读取数据包。若要让 tcpdump 不解析端口号和地址（即仅显示数字形式），可使用 -n 选项（部分旧版本的 tcpdump 可能需要指定两次 -n；可写作 -n -n 或合并为 -nn）。

```bash
$ tcpdump -nnr packets.pcap
```

当 tcpdump 读取数据包时，它通常会以每行一个数据包的汇总形式显示输出。如果想要查看数据包内容中人类可读的 ASCII 字符，请使用 -A 选项。

```bash
$ tcpdump -r file -n -A 
```

### 过滤

![image-20260501133511117](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501133511117.png)

如前所述，tcpdump 的强大特性之一在于其过滤功能。tcpdump 用于指定过滤（捕获和/或排除）数据包方式的语言，基于一种名为伯克利数据包过滤器（简称 BPF）的技术。

使用BPF的过滤表达式由一个或多个原语和运算符组成。一个原语由一个或多个限定符和一个ID构成。限定符主要分为三类：

第一类是类型，用于指定ID所指代对象的类型，有效的类型限定符包括host（主机）、net（网络）、port（端口）和portrange（端口范围）。
另一类是方向限定符，用于描述ID所关联的方向，有效值为src（表示ID应为源地址、端口、网络等）和dst（表示ID应为目的地址、端口、网络等）。
第三类限定符是协议，可指代任意常见协议，如ip（网际协议）、tcp（传输控制协议）、udp（用户数据报协议）、icmp（互联网控制消息协议）等。

可以使用与（&&）或（||）等布尔运算符组合多个表达式。

最后，你还可以添加括号来强制某些操作优先于其他操作执行。括号也能让 BPF 编写者的意图对其他人来说更清晰。

## 网络代理

![image-20260501133729606](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501133729606.png)

在企业环境中使用网络代理的情况并不少见，因为它们能带来多种不同的好处。例如，缓存型网络代理可以减少带宽使用和网络拥塞。企业也可以使用网络代理屏蔽不适合业务场景的网站。

通过网络代理记录网站访问和网络应用使用情况，可为事件响应分析创造宝贵的资源。调查人员可利用代理日志构建更全面的用户活动画像，并识别异常和可疑流量。在某些情况下，若配置得当，代理日志甚至能显示加密会话的网络请求。

目前有多种 Web 代理服务器解决方案可供选择，包括 Microsoft Windows Web 应用程序代理、Blue Coat、Forefront TMG 以及 Squid。在本课程中，我们将分析 Squid 代理软件的部分日志文件。Squid 代理自 20 世纪 90 年代中期问世以来，一直是目前使用最广泛的代理应用程序之一。它能够缓存请求、记录用户代理字符串，还支持多种协议（不仅限于 HTTP 和 HTTPS），功能远不止这些。

### 访问日志

![image-20260501134211931](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501134211931.png)

访问日志用于记录通过 Squid 代理发送的单个请求。它们是用户可定义格式的文本文件，不过默认格式相当冗长。值得注意的是，尽管通常会显示 URL，但根据代理和客户端的配置方式，你可能会看到也可能看不到 HTTPS 请求对应的 URL。用于生成幻灯片上示例的代理和客户端被配置为允许拦截加密流量。

Squid 访问日志的默认格式为每行一条记录，每条记录分为以下字段：

1. .Timestamp:请求被记录的时间，以自1970年1月1日世界协调时起的秒数表示，精确到毫秒。该字段通常在处理完请求本身后进行记录。要确定请求首次被接收的时间，需用该字段减去下一个字段（持续时间）。
2. ..Duration：代理服务器处理请求所花费的毫秒数。
3. ·Client: ·发出请求的系统的IP地址。
4. Result Codes:Squid 结果代码，后跟一个斜杠（/），以及 HTTP 状态代码。
5. Size:发送至客户端的数据的字节数。
6. HTTP Method：客户端请求中的 HTTP 请求方法
7. . URL：客户端请求的 URL（如果可用）。
8. User：发出请求的客户端身份。身份通过查看 HTTP 认证信息、配置的外部程序、TLS 认证信息和 IDENT 查询来确定。如果没有任何方式能提供身份，则显示短横线（-）。
9. ·Content Type:·来自HTTP回复的内容类型字段。

数据包捕获信息也是事件响应人员的宝贵资源。我们研究了将 tcpdump 用作分析工具的方法，重点分析了用于捕获数据包以及从保存的数据包捕获文件中读取数据的命令行参数。tcpdump 的一大重要价值在于可运用伯克利数据包过滤器（BPF），依据多种条件对数据包捕获文件中的特定事件进行排查，从而挖掘出有价值的信息，包括每个已记录数据包的日期和时间等细节。

# 内存调查

现在我们将学习如何使用 Volatility 从内存（RAM）镜像中分析数据结构和工件。

![image-20260501134916339](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501134916339.png)

调查内存镜像的内存取证已成为许多事件响应调查的核心手段。这是因为随机存取存储器（RAM）中存储着大量有价值的信息，而这些信息并不会留存于磁盘中。此外，对取证内存镜像进行分析，意味着在分析过程中证据不会再发生变动。

在分析内存之前，你需要一款能够收集内存的工具。迈克·科恩为Windows系统开发的WinPmem工具是一款能从Windows系统中捕获随机存取存储器（RAM）的简易工具，它会将输出结果保存为原始文件格式（例如，输出文件是未经过编码或格式化的随机存取存储器内容）。本页面展示了WinPmem输出内容的一段节选，该节选捕获了一台Windows 11系统的内存。WinPmem是免费开源软件，可在https:/github.com/Velocidex/WinPmem获取。

注意：WinPmem 必须以管理员权限运行，才能在目标系统上捕获内存。

获取到内存镜像后，你需要一款分析工具。在本课程中，我们将使用 Volatility 工具。使用 Volatility 这类内存分析工具的优势之一是，我们可以沿用与实时调查模块中相同的基本分析策略，来识别可疑进程、网络连接等内容。

因此，尽管工具不同，但分析策略和可疑指标是相同的。

## Volatility 概述

![image-20260501170545002](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501170545002.png)

Volatility 既是一个框架，也是一种分析内存镜像的工具。它于 2007 年首次发布，主要有两个版本：Volatility 2.x 和 Volatility 3.x。Volatility 3 是对 Volatility 框架的重大更新，于 2021 年推出，具备多项性能和易用性方面的特性。由于此次更新的特性，Volatility 2 中部分此前可用的功能在 Volatility 3 中已不再提供（或尚未提供）。针对尚未移植到 Volatility 3 的特定插件，你可能仍会看到 Volatility 2 的持续使用。、

Volatility 基于 Python 开发，可在安装了 Python 解释器的 Windows、macOS 和 Linux 系统上运行。使用 Volatility 时，需通过 `-f` 参数指定捕获文件，然后指定要使用的插件名称。Volatility 插件采用类似类的命名规则，格式为**平台.类.插件名**（例如 `windows.pslist.PsList`）。若想查看 Volatility 支持的插件列表，可访问本页面显示的网址。

### 内存取证

![image-20260501171214215](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501171214215.png)

在本模块中，我们研究了多款可与 Volatility 配合使用的插件，借助这些插件我们能够获取内存捕获文件并对系统进行排查。在很多方面，内存排查与我们之前接触的实时排查步骤相似；我们会评估系统进程、网络连接和注册表痕迹，以识别事件并了解其影响范围。针对内存排查，我们通过 Volatility 和内存捕获文件来完成这项分析，而非使用在实时系统上可运行的 PowerShell 命令行工具或 Windows 传统命令。

与实时调查所执行的分析步骤类似，内存分析同样从关注事件（EOI）开始。若关注事件为可疑进程，Volatility 的 windows.pslist.PsList 和 windows.pstree.PsTree 插件可帮助我们评估进程详情，包括进程名称、进程ID、父进程ID以及执行开始时间。若关注事件为可疑网络监听器，那么 Volatility 的 windows.netscan.NetScan 便成为一款实用的分析工具，它能识别源IP地址和目的IP地址、端口、进程名称、连接类型和状态以及进程ID。一旦获取到进程ID，windows.pslist.PsList 和 windows.pstree.PsTree 插件也能发挥作用，继续对系统展开评估。若关注事件为可疑程序，我们可先通过 Volatility 的 windows.cmdline.CmdLine 插件启动分析，随后再借助 windows.pslist.PsList 和 windows.pstree.PsTree 插件对进程进行详细分析。

作为事件响应人员，我们需要确保记录自己的分析过程。养成将 Volatility 分析结果保存到文本文件的习惯，有助于加快分析速度，也能让数据在后续报告撰写时随时可用。运行 Volatility 时，使用 `-q` 参数屏蔽进度指示器，并在命令行末尾添加 > 输出文件名，将输出重定向到文件中。

Volatility 对事件响应人员来说是一款实用的分析工具；它让我们能够借助 WinPmem 或其他工具生成的内存捕获文件，对系统进行排查，以此替代实时调查与分析，对系统的评估流程保持不变：我们运用原本在运行系统上会执行的分析步骤，借助 Volatility 的插件功能开展分析，同时还能获得离线分析能力以及存储在内存捕获文件中的持久数据这两项额外优势。

# 恶意软件调查

![image-20260501171459599](./image/1.%E4%BA%8B%E4%BB%B6%E5%93%8D%E5%BA%94%E4%B8%8E%E7%BD%91%E7%BB%9C%E8%B0%83%E6%9F%A5/image-20260501171459599.png)

此前，我们讨论过事件响应人员可能会看到可疑运行进程，但无法确定其是否为恶意的情况。例如，名称怪异的进程未被企业防病毒软件标记。由于可疑进程不一定是恶意的，你需要对未知进程进行调查。

同样地，如果你发现了一种新的恶意软件，而其他任何人都尚未生成其指标列表，那么你需要自行对该恶意软件展开调查。本模块中的技术将帮助你确定相关指标，你可借此在企业范围内进行扫描，找出其他已被入侵的系统。

此外，恶意软件调查的结果还可用于为威胁情报提供支持。研究恶意软件通常采用两种基本方法。第一种是借助环境监控工具来监测恶意软件样本与环境的交互行为，这种方法有时被称为行为分析。另一种常见方法是在调试器、反汇编器等工具中直接分析恶意软件的代码。

行为分析这个名称有些令人困惑，因为检查代码的目的也是为了理解（分析）恶意软件的行为。这两种策略的真正区别在于你所做的操作，而非你想要找出的内容。
